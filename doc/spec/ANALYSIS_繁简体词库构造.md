# 繁简体词库构造方案

**文档版本**: v1.0
**创建日期**: 2026-03-31
**最后更新**: 2026-03-31

## 1. 背景与目标

### 1.1 问题描述

《史记》知识库需要提供繁简体阅读切换功能。使用OpenCC进行基础转换时，存在**上下文相关的字符转换**问题，最典型的是"后"字：

- **作为爵位/人名**时应保持"后"：如"吕后"、"太后"、"皇后"
- **作为时间/方位**时应转为"後"：如"后世"、"其后"、"然后"
- **作为神祇/人名**时应转为"後"：如"后稷"、"后羿"

OpenCC的标准s2t转换会统一将"后"转为"後"，导致"吕后"错误地显示为"吕後"。

### 1.2 解决目标

构建一个`custom-variants.json`词库，用于在OpenCC转换后进行**二次修正**，确保：
1. 上下文相关字符转换正确
2. 词库规则准确、可维护
3. 避免引入文本版本差异

## 2. 技术方案

### 2.1 整体架构

```
简体文本
  ↓
[OpenCC s2t 基础转换]
  ↓
繁体文本（可能有上下文错误）
  ↓
[Custom Variants 修正]
  ↓
最终繁体文本（上下文正确）
```

### 2.2 转换层次

1. **OpenCC基础转换**：处理99%的标准繁简转换
2. **Custom Variants修正**：处理上下文相关的特殊情况

转换顺序：**先OpenCC，后Custom Variants**（在前端JavaScript中实现，见`docs/js/simp-trad-converter.js`）

## 3. 词库构造过程

### 3.1 方案探索

#### 3.1.1 尝试1: 自动逐字比较（失败）

**方法**：
- 使用OpenCC将简体底本（`archive/chapter/*.txt`）转为繁体
- 与维基文库繁体版本（`archive/wikisource_shiji/*.html`）逐字比较
- 提取差异作为custom variants

**实施**：
```bash
# 创建虚拟环境
python3 -m venv .venv
.venv/bin/pip install opencc-python-reimplemented beautifulsoup4

# 运行比较脚本
.venv/bin/python scripts/compare_with_wikisource.py
```

**结果**：失败
**原因**：
- 维基文库底本与我们的简体底本**不是同一版本**
- 提取到的"差异"大部分是**文本校勘差异**而非繁简转换差异
- 例如：
  - `"九" → "一"` （年份不同）
  - `"盲，而舜母死" → "更娶妻而生象"` （整句不同）
  - `"君" → "臣"` （人物称谓不同）

**教训**：不同版本的古籍不能直接逐字比对来提取繁简规则。

#### 3.1.2 尝试2: 手动规则构造（成功）

**方法**：
1. 识别OpenCC转换的已知问题（如"后"字问题）
2. 基于《史记》文本特点，手动列举常见词汇
3. 参考维基文库繁体版本验证个别词汇
4. 构建精简但准确的规则集

**实施**：见[3.2 最终词库](#32-最终词库)

**结果**：成功，创建32条精准规则

#### 3.1.3 尝试3: 模糊匹配提取（失败）

**动机**：希望通过改进算法，从维基文库自动提取更多规则

**方法**：
1. 使用`difflib.SequenceMatcher`进行模糊文本对齐
2. 过滤版本差异（长度差异、数字等）
3. 使用已知异体字表验证
4. 频率统计（≥3次出现）确保可靠性

**实施**：
```bash
.venv/bin/python scripts/extract_variants_fuzzy.py
```

**结果**：失败
- 提取405条规则，筛选出48条高频规则
- 但验证发现**所有规则都不应该添加**

**原因**：
1. **23条引号规则**：这是标点风格差异（大陆`""`vs台湾`「」`），不是繁简转换
2. **22条异体字规则**：这是异体字选择差异，不是OpenCC缺陷
   - 例如：`"击吴" → OpenCC: "擊吳" vs 维基: "撃呉"`
   - OpenCC已经正确转换为繁体"擊"，但维基使用了异体字"撃"
   - 类似的：`"录" → OpenCC: "彔" vs 维基: "錄"`
3. **3条重复规则**：与手动词库重复

**关键发现**：
- ✅ 验证了手动规则的高频性（如"原"出现63次，"琅"出现12次）
- ✅ 模糊匹配成功过滤了版本差异和校勘差异
- ❌ 但**无法区分"繁简转换差异"和"异体字选择差异"**

**根本问题**：
- 我们需要的是"OpenCC未正确转换的规则"
- 提取到的是"OpenCC已转换，但异体字选择与维基不同"
- 维基文库使用的繁体字异体是编辑偏好，不应作为我们的转换标准

**教训**：
1. 与维基文库比较不适合构建繁简词库
2. 维基的异体字选择与OpenCC不同，这不是问题而是合理差异
3. 自动提取方法需要先验证OpenCC是否已正确处理

**详细分析**：见[doc/spec/繁简词库对比分析.md](繁简词库对比分析.md)

### 3.4 词库设计原则

1. **精简优先**：只包含确认必要的规则，避免过度设计
2. **词级优先**：优先使用词级规则（如"吕后"），减少歧义
3. **长词优先**：在JavaScript应用时，长词优先匹配（longest-match-first）
4. **只修正已知问题**：不试图替代OpenCC的通用功能
5. **可验证性**：每条规则都可在《史记》文本中找到实例

## 4. 应用实现

### 4.1 前端实现

位置：`docs/js/simp-trad-converter.js`

```javascript
function convertText(text) {
    if (!text || !converter) return text;

    // Step 1: OpenCC基础转换
    let result = converter(text);

    // Step 2: 应用custom variants
    if (customVariants && Object.keys(customVariants).length > 0) {
        // 按长度排序，长词优先
        const entries = Object.entries(customVariants)
            .sort((a, b) => b[0].length - a[0].length);

        for (const [simp, trad] of entries) {
            const regex = new RegExp(escapeRegExp(simp), 'g');
            result = result.replace(regex, trad);
        }
    }

    return result;
}
```

### 4.2 词库加载

```javascript
// 异步加载custom variants
fetch('../data/custom-variants.json')
    .then(response => response.json())
    .then(data => {
        customVariants = data;
    });
```

## 6. 维护与扩展

### 6.1 新增规则流程

1. **发现问题**：在繁体显示中发现转换错误
2. **确认类型**：判断是OpenCC问题还是上下文问题
3. **验证必要性**：确认该词在《史记》中多次出现
4. **添加规则**：在`custom-variants.json`中添加
5. **测试验证**：在实际文本中测试

### 6.2 规则优先级

遵循"longest-match-first"原则：
- "吕后娘娘"会先匹配"吕后"（如果有此规则）
- 避免规则冲突：不添加会互相覆盖的规则

### 6.3 未来扩展

如发现更多上下文相关字符，可按以下类别扩展：
- **一字多义**：如"干"（干戈vs干涉）、"历"（历史vs历法）
- **地名专用**：如特殊地名的繁体写法
- **人名专用**：如某些人名的特殊繁体

## 9. 总结

### 9.1 关键决策

1. **放弃自动比对**：文本版本差异使自动提取不可行
2. **精简优先**：32条精准规则优于数百条含噪声的规则
3. **词级为主**：用词组而非单字减少歧义

### 9.2 成果

- ✅ 解决了"后"字的上下文转换问题
- ✅ 创建了可维护的精简词库
- ✅ 提供了清晰的扩展路径

### 9.3 经验教训

**失败的尝试**：
- ❌ **尝试1**：不同版本古籍的自动逐字比对 → 提取到的是文本校勘差异
- ❌ **尝试3**：模糊匹配提取 → 提取到的是异体字选择差异和标点风格差异
- ❌ 试图构建"完整"的繁简词库

**成功的做法**：
- ✅ **尝试2**：聚焦核心问题（"后"字上下文转换）
- ✅ 手动构建高质量规则（32条）
- ✅ 保持词库精简可控

**三个关键认识**：
1. **文本版本差异** ≠ 繁简转换差异（尝试1的教训）
2. **异体字选择差异** ≠ OpenCC缺陷（尝试3的教训）
3. **维基文库不适合作为繁简词库的构建来源**
