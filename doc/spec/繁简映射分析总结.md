## 一、核心发现

### 1.1 总体统计

| 指标 | 数量 | 说明 |
|-----|------|------|
| **总字符数（汉字）** | 577,186 | 《史记》全书的汉字总数 |
| **独特字符数** | 4,787 | 去重后的不同汉字数量 |

### 1.2 映射关系分布

| 类型 | 数量 | 占比 | 说明 |
|-----|------|------|------|
| **繁简相同** | 3,357个 | 70.1% | 繁简体字形相同，无需转换（如"人"、"大"、"山"） |
| **一对一映射** | 1,343个 | 28.1% | 简体字唯一对应一个繁体字（如"国"→"國"） |
| **一对多映射** | **87个** | **1.8%** | **简体字对应多个繁体字，取决于上下文** |

### 1.3 词频统计

**87个一对多字符的词频分布**：

| 指标 | 数值 | 说明 |
|-----|------|------|
| **总出现次数** | 19,493次 | 87个字符在《史记》中的总频率 |
| **平均每字** | 224.1次 | 平均出现频率 |
| **中位数** | 99次 | 一半字符频率低于此值 |
| **最高频** | 1,810次 | "夫"字 |
| **最低频** | 1次 | "噪""症""仿"各1次 |

**频率分组**：
- **高频（Top 10）**: 10,590次，占54.3%
- **中频（11-50名）**: 8,299次，占42.5%
- **低频（51-87名）**: 604次，占3.2%

### 1.4 关键认识

1. **98.2%的字符可以一一映射**（繁简相同 + 一对一映射）
2. **仅1.8%的字符（87个）需要上下文判断**
3. **这87个字符占《史记》全文的3.38%** - 影响重大！
   - 平均每29.6个字就有1个需要上下文判断
   - 虽然字符种类少，但使用频率高
4. 这87个字符是繁简转换的**核心难点**
5. **前10个字符占总频率的54.3%** - 应优先处理

**上下文规则**:
1. **爵位**（保持"后"）：
   - 皇后、太后、王后、后妃
   - 吕后、赵太后

2. **时间/方位**（转为"後"）：
   - 后世、后代、后人
   - 其后、之后、然后、以后、而后
   - 前后、先后、最后

3. **人名**（转为"後"）：
   - 后稷（周朝始祖）
   - 后羿（夏代射日英雄）
   - 后土（土地之神）

**OpenCC默认行为**: 统一转为"後" ❌

## 四、对自定义词库的启示

### 4.1 哪些字需要自定义词库？

根据分析，以下字符**最需要**自定义词库：

| 优先级 | 简体 | 频率 | 理由 |
|-------|------|------|------|
| ⭐⭐⭐ | **后** | 1,212 | OpenCC全部转"後"，爵位场景错误 |
| ⭐⭐⭐ | **复** | 1,054 | OpenCC全部转"復"，"重复"等场景错误 |
| ⭐⭐ | **发** | 637 | OpenCC全部转"發"，"头发"场景错误 |
| ⭐⭐ | **里** | 559 | OpenCC全部转"裏"，"里程"场景错误 |
| ⭐ | **历** | 129 | OpenCC全部转"歷"，"历法"场景错误 |
| ⭐ | **系** | 133 | 三种形式，OpenCC可能错误 |
| ⭐ | **干** | 106 | 三种形式，OpenCC可能错误 |

### 4.2 词库构建策略

基于统计结果，推荐**三层策略**：

#### 第一层：核心词库（高频+高错误率）
- **目标**: 5-10个字，20-30条规则
- **字符**: 后、复、发、里
- **方法**: 手动列举所有高频词组

#### 第二层：补充词库（中频+明确规则）
- **目标**: 10-20个字，30-50条规则
- **字符**: 历、系、干、台、谷、斗等
- **方法**: 基于领域词表（历史、地理）构建

#### 第三层：观察名单（低频+待验证）
- **目标**: 不立即构建，发现问题时添加
- **字符**: 其余60+个字
- **方法**: 用户反馈驱动

### 4.3 当前词库评估

**现有词库**（`docs/data/custom-variants.json`，32条规则）：

## 五、统计方法说明

### 5.1 分析流程

```python
# 1. 加载《史记》所有章节
text = load_all_chapters()

# 2. 提取所有汉字
chars = extract_chinese_chars(text)

# 3. 对每个字符进行上下文转换
for char in chars:
    contexts = find_all_contexts(char, text)
    for ctx in contexts:
        trad = opencc.convert(ctx)
        record(char, trad)

# 4. 统计每个字符对应的所有繁体形式
for char in chars:
    trad_set = get_all_trad_forms(char)
    if len(trad_set) > 1:
        one_to_many[char] = trad_set
```

### 5.2 上下文采样

- **单字转换**: `opencc.convert(char)`
- **上下文转换**: 在文本中查找该字的所有出现位置，提取前后各2个字作为上下文
- **去重**: 对每个字符的所有转换结果去重

### 5.3 局限性

1. **未区分词性**: 同一个字在不同词性下可能有不同繁体（如"发"作动词vs名词）
2. **上下文窗口固定**: 前后各2个字可能不足以覆盖所有语义
3. **依赖OpenCC**: 统计结果依赖OpenCC的转换逻辑

## 七、结论

1. **98.2%的字符可以自动转换**，OpenCC已经能很好地处理
2. **仅1.8%（87个字符）需要上下文判断**，这是繁简转换的核心难点
3. **前5个高频字符**（后、欲、周、于、复）占据了**6,082次**出现，应优先处理
4. **建议补充词库**：在现有32条基础上，增加"复"、"发"、"里"、"历"的上下文规则
5. **长尾策略**：其余60+个低频字符，采用"发现问题时添加"的策略
