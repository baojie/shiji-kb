## 一、研究方法

### 1.1 总体流程

第三轮反思采用与前两轮相同的自动化Agent管线，在SKILL文档（已含26条错误模式+12条推理逻辑）和经两轮修正后的事件索引基础上运行：

1. **提示词生成**：`run_review_pipeline.py --prompt NNN` 为每章生成审查提示词，嵌入最新SKILL规则和该章事件索引
2. **Agent反思执行**：以5-20章为一批，并行调用Agent工具，逐章对每个事件进行年表交叉验证、reign-year换算校核、格式规范检查
3. **结果吞入**：`run_review_pipeline.py --ingest NNN reflect_NNN.json` 更新pipeline状态
4. **修正应用**：`apply_reflect_fixes.py NNN` 将修正写入事件索引文件

### 1.2 审查策略

第三轮的审查重心进一步聚焦于精细度和收敛性验证：

| 维度 | 第一轮 | 第二轮 | 第三轮 |
|------|--------|--------|--------|
| 主要问题 | 系统性错误（塌缩、混淆、大幅偏差） | 精细调校（确定性升级、格式修正） | 收敛验证（残留清理、最终确认） |
| 修正量级 | 1,010处（7.8/章） | 431处（3.3/章） | 465处（3.6/章） |
| 典型偏差 | 数十至数百年 | 1-5年或仅格式变更 | 格式升级为主，少量年份微调 |
| 新模式发现 | 25条 | 1条 | 0条 |

### 1.3 技术实现

- **批量并行执行**：130章分6批（每批5-20章并行Agent），总耗时约2小时
- **Agent写入兼容**：部分Agent受权限限制无法直接写文件，通过Agent输出提取或单章重跑解决
- **第二轮备份**：开始前将130个第二轮reflect文件备份至 `kg/events/reports/round2_backup/`

### 3.1 修正分类

| 类型 | 修正数 | 占比 | 说明 |
|------|--------|------|------|

| B. 年表交叉验证修正 | 115 | 24.7% | 年份经年表验证后微调 |

| **总计** | **465** | **100%** | |

### 3.2 分析

**确定性升级仍是第一大类（36.8%）**，表明三级标注体系（精确/推算/近似）的标注精度优化是一个渐进收敛过程。每一轮都会发现更多年表有载但未用精确标记的事件。

**年表验证修正（24.7%）** 在第三轮中主要为1-3年的微调，或对通论章节（023礼书、025律书）中残留的兜底值（前300年等）的修正。大幅偏差（>10年）的案例已基本在前两轮清除。

**综合修正（29.5%）** 主要集中在001五帝本纪、002夏本纪等远古章节，以及069苏秦列传等存在史料系统性分歧的章节，这些章节由于缺乏可靠锚点，每一轮都可能产生不同程度的年代调整。

**新模式发现为零**，证明SKILL的26条错误模式+12条推理逻辑已完全饱和。

## 四、典型案例分析

### 案例1：通论章节兜底值清理——礼书（023）

**问题**：023礼书为通论性质章节，横跨殷商至汉武帝约1000年。第一、二轮已修正了最严重的塌缩，但仍有9/14个事件残留章节兜底值（前300年）或生卒中点误标。

**第三轮修正**：13处，按各事件所属帝王的在位期独立定年：
- 023-001 三代礼制起源：前300年 → [约公元前1600年]（殷商建立时期）
- 023-005 孔子论礼：前300年 → [约公元前500年]（孔子活跃期）
- 023-009 叔孙通定礼：前300年 → （公元前200年）（年表有载）

**特征**：通论章节的兜底值清理需要多轮迭代，因为每个事件的正确年代需要独立推理，无法批量处理。

### 案例2：全章精确纪年升级——商君列传（068）

**问题**：068章商鞅变法年代记载详细，年表覆盖极全。前两轮已修正了年份，但17个事件中大量仍使用`[约]`推断标记。

**第三轮修正**：17处，全部为确定性标注升级：
- 068-002 商鞅入秦：`[约公元前361年]` → `（公元前361年）`
- 068-005 徙木立信：`[约公元前356年]` → `（公元前356年）`
- 068-014 商鞅车裂：`[约公元前338年]` → `（公元前338年）`

**特征**：年份数字本身正确，仅标注格式需升级。此类修正的根因是前两轮关注于修正年份错误，对格式升级的覆盖不够彻底。

### 案例3：裸年份格式统一——卫康叔世家（037）

**问题**：037章21个事件中，13个使用裸年份格式"公元前XXX年"（无括号标记），不符合三级标注规范。

**第三轮修正**：13处，全部统一为`（公元前XXX年）`精确标注格式。

**特征**：这是第二轮发现的错误模式#26（裸年份缺少确定性标记）的批量清理。

### 案例4：单锚点塌缩+相对纪年计算——李将军列传（109）

**问题**：109章16个事件中仍有4处单锚点塌缩（事件被归到前166年/前141年/前88年），以及相对纪年计算错误。

**第三轮修正**：10处
- 109-005 马邑之谋：前88年 → （公元前133年）（年表明确记载）
- 109-006 "马邑后四岁"：前123年 → [公元前129年]（133-4=129）
- 109-011 "元朔六年后二岁"：前123年 → [公元前121年]（123-2=121）
- 109-015 "广死明年"：前124年 → [公元前118年]（119+1=118）

**特征**：相对纪年（"后N年"/"明年"）的计算错误在前两轮未被发现，因为需要先修正锚点事件的年份才能正确计算。这是多轮迭代的价值所在。

### 案例5：史记系年偏差残留——苏秦列传（069）

**问题**：069章苏秦列传在第一轮已做了系统性下移（从前333年移至前311年），但第三轮仍发现20处需要进一步调整。

**第三轮修正**：20处
- 9个事件因前两轮使用了折中年份，第三轮按年表+马王堆帛书进一步精确化
- 苏秦说六国的具体年份按各国君主在位期独立定年

**特征**：存在史料系统性分歧的章节（《史记》vs出土文献），每一轮都在逼近更准确的年代标注，但因缺乏绝对锚点，收敛速度较慢。

## 八、方法论总结

### 8.1 三轮迭代的自然分工

| 轮次 | 核心功能 | 类比 |
|------|---------|------|
| 第一轮 | 发现系统性错误+建立规则库 | 粗磨（去除大缺陷） |
| 第二轮 | 精细调校+规则验证 | 细磨（精度优化） |
| 第三轮 | 收敛验证+残留清理 | 抛光（最终确认） |

### 8.2 收敛性结论

三轮反思的关键指标变化：

```
修正量：  1,010 → 431 → 465  （第三轮基本持平，主要为格式优化）
新模式：  25 → 1 → 0          （第二轮即饱和）
无修正章：12 → 25 → 60        （稳步增长，趋向全面收敛）
```

**结论**：
- **年份准确度**已在第一轮后基本收敛，第二三轮的年份实质性修改极少
- **标注精度**（确定性升级）是一个渐进过程，三轮后仍有少量残留但影响不大
- **规则库**在第二轮已完全饱和（26条），第三轮零新增确认了这一点
- **第四轮如果执行**，预计修正量将大幅下降（<100处），且几乎全部为格式优化，建议视下游应用需求决定是否执行

### 8.3 Agent反思管线的成熟度

经三轮130×3=390次Agent执行验证：

| 维度 | 评估 |
|------|------|
| 全自动化程度 | 高（提示词生成→Agent执行→结果吞入→修正应用全流程脚本化） |
| 批量并行能力 | 从每批5章提升到每批20章，总耗时从4小时缩短到2小时 |
| 错误模式饱和 | 已确认（第三轮零新增） |
