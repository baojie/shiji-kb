# 事件识别与年代审查——反思创造过程详解

> 基于Claude Agent的《史记》3,092事件结构化提取与两轮年代反思审查全流程技术文档

---

## 一、项目背景

《史记》130篇，上起五帝（约前2600年），下至汉武帝太初年间（前104年），跨度约2,500年。本项目的目标是：从130篇原文中提取结构化事件，标注公元纪年，并通过Agent反思迭代修正年代错误，最终建成高质量历史事件知识库。

核心挑战：
- 《史记》使用王朝纪年（如"秦昭王十三年"），需换算为公元纪年
- 同一年号可能指不同时期（如"二年"可能是惠帝二年或汉二年）
- 自动推断脚本存在系统性错误，需要AI语义理解来发现和修正
- 130篇章体裁各异（本纪/世家/列传/书/表），处理策略需差异化

## 二、事件识别

### 2.1 粒度控制

事件粒度是首要设计决策。过细则噪声多、冗余大；过粗则信息丢失。最终采用"战略级"粒度：

- **合并原则**：同一战役的多次交锋合并为一个事件；同一政策的颁布与执行合并
- **拆分原则**：不同主体的独立行动拆分；时间跨度超过5年的事件拆分
- **目标密度**：每章15–40个事件（实际均值23.8个）

### 2.2 事件类型体系

定义11种事件类型，覆盖《史记》主要叙事范畴：

| 类型 | 说明 | 典型事件 |
|------|------|---------|
| 战争 | 军事冲突 | 长平之战、巨鹿之战 |
| 继位 | 君主即位/禅让 | 尧禅位于舜 |
| 政治活动 | 外交/会盟/出使 | 张骞出使西域 |
| 政治改革 | 制度变革 | 商鞅变法 |
| 政治整顿 | 肃清/整肃 | 吕后诛诸吕 |
| 家族事件 | 婚丧/继嗣 | 赵氏孤儿 |
| 建设 | 工程/营建 | 修建长城 |
| 文化活动 | 著述/学术 | 司马迁作《史记》 |
| 经济活动 | 商贸/财政 | 盐铁专营 |
| 自然灾害 | 天灾/异象 | 地震、旱灾 |
| 改革 | 一般性变革 | 历法改革 |

### 2.3 章节分类策略

130篇按体裁分为五类，采用差异化提取策略：

| 体裁 | 篇数 | 策略要点 |
|------|------|---------|
| 本纪（12篇） | 帝王编年 | 按年表顺序提取，注意年号切换 |
| 世家（30篇） | 诸侯/功臣传 | 注意世系跨代，世代数≠年数 |
| 列传（70篇） | 人物传记 | 注意追叙（传记开头早年经历）和并列传 |
| 书（8篇） | 制度专题 | 跨时代综述，锚点稀疏 |
| 表（10篇） | 年表/世表 | 结构化数据，直接提取 |

### 2.4 事件的定义与识别标准

一个"事件"被定义为**有完整起因-经过-结果（或至少经过-结果）的行为单元**，必须包含：

- 一个或多个行为主体（人物）
- 一个行为或状态变化
- 可选的时间、地点、因果关系

粒度的具体判断标准：

| 判断 | 标准 |
|------|------|
| 能否用4–8字概括？ | 能 → 粒度合适 |
| 能否写出2–4句描述？ | 不能（太简单） → 可能太细 |
| 描述超过8句？ | → 可能太粗，应拆分 |

几种特殊情况的处理：

- **世系传承**：连续的"X崩，子Y立"（无额外叙述）合并为一个"世系传承"事件
- **重复叙述**：同一事件在不同章节被提及（如001和002都提到"鲧治水"）→ 各章独立提取，不去重
- **太史公曰**：评论本身不是事件，但如果评论中**首次提到**某个史实，应当提取
- **模糊边界**：纯描述性内容（如禹划定九州的地理描写）视为一个大事件的子部分

### 2.5 识别实现：Claude API + SKILL方法论

事件识别的技术实现采用Claude API逐章处理，核心是一套**方法论驱动的提示词设计**：

**输入三要素**：
1. **已标注的章节原文**：文本已完成实体标注（含`@人名@`、`=地名=`、`%时间%`等标记），实体标注为事件识别提供了结构化的上下文
2. **`SKILL_事件识别.md`方法论文档**（311行）：定义事件的定义、粒度标准、11种类型体系、各体裁处理策略、输出格式规范、质量标准
3. **章节类型提示**：根据当前章节属于本纪/表/书/世家/列传，微调提取策略（如本纪侧重年表式编年提取，列传侧重人物生平转折点）

**识别流程**：

```text
已标注原文(.tagged.md)
    │
    ├─ 实体标注提供上下文 ──→ @人物@ =地点= %时间% &朝代&
    │
    ▼
Claude API（嵌入SKILL_事件识别.md）
    │
    ├─ 1. 通读全文，识别叙事单元
    ├─ 2. 按粒度标准切分事件边界
    ├─ 3. 为每个事件分配类型、提取要素
    ├─ 4. 生成概览表 + 详情记录
    │
    ▼
事件索引文件（NNN_章名_事件索引.md）
    │
    ▼
validate_events.py 后处理验证
    │
    ├─ 检查事件ID格式和连续性
    ├─ 检查必须字段完整性
    └─ 生成统计汇总
```

**输出格式**：每章生成一个事件索引文件，包含三部分：

1. **概览表**：事件ID / 事件名称 / 类型 / 时间 / 地点 / 主要人物 / 朝代
2. **详情记录**：每事件含事件描述（2–4句）、原文引用（保留标注）、段落位置
3. **关键事件链**：如"商朝建立：契受封 → 先公世系 → 汤建商"

**质量保障**：
- 长章节（如秦始皇本纪）按section拆分，分批处理后合并
- 支持断点续传（progress.json），中途失败可恢复
- 概览表格与详细记录一一对应，自动验证

最终产出：**130章 × 3,092个事件**，平均每章23.8个事件。

| 章节类型 | 典型事件数 |
|---------|-----------|
| 本纪（短） | 30–50 |
| 本纪（长） | 80–150 |
| 表 | 10–30 |
| 书 | 15–40 |
| 世家 | 20–100 |
| 列传 | 10–60 |

## 三、自动年代标注

### 3.1 三级确定性标注体系

事件提取完成后，需要为每个事件标注公元纪年。设计了三级确定性标注体系：

| 标注格式 | 含义 | 来源 | 示例 |
|---------|------|------|------|
| `（公元前XXX年）` | 精确纪年 | 年表直接对应 | （公元前770年） |
| `[公元前XXX年]` | 确定推算 | reign-year换算 | [公元前636年] |
| `[约公元前XXX年]` | 近似推断 | 上下文推断 | [约公元前2600年] |

纪年标记（如`%秦昭王十三年%`）独立保留，与公元年标注并存：

```
%秦昭王十三年% （公元前294年）
```

### 3.2 自动推断脚本及其系统性缺陷

初始年代标注由`write_inferred_years.py`脚本自动完成，采用"最近锚点"策略：找到章节中最近的已知纪年，赋给周围事件。这种方法存在严重的系统性错误：

| 缺陷类型 | 偏差范围 | 示例 |
|---------|---------|------|
| **单锚点塌缩** | 全章共享同一年份 | 001五帝本纪66事件全标前2290年 |
| **通论章节多时代塌缩** | 跨千年事件共享单一锚点 | 028封禅书上古至汉全标前209年 |
| **锚点计算错误** | 216–368年 | 046田和代齐偏差216年 |
| **前元/中元/后元混淆** | 7–13年 | 011孝景本纪28处混淆 |
| **年号/帝王混淆** | 12–300年 | 053萧何卒年错误 |
| **追叙误用当前时间** | 20+年 | 列传开头早年经历被标为后期年份 |

这些缺陷中，格式错误可用脚本自动检测，但**语义错误**（如"二年"指哪位君主的二年）必须由AI理解上下文才能判断。这就是Agent反思管线的核心价值所在。

### 3.3 Ground Truth：年表作为外部锚点

整个年代审查体系的根基是一个关键假设：**年表中的精确纪年是ground truth**。

《史记》自身使用王朝纪年（"秦昭王十三年""汉元光元年"等），不提供公元纪年。要将王朝纪年换算为公元纪年，必须有一套外部参照系。本项目使用的年表数据（`kg/tables/`）来自现代学术考证，包含帝王元年/卒年、重大事件的公元对照。

年代推断的逻辑链如下：

```text
                    外部年表（ground truth）
                    ┌──────────────────┐
                    │ 帝王元年/卒年     │
                    │ 重大事件公元年     │
                    │ 45+个基准锚点     │
                    └────────┬─────────┘
                             │ 直接对应
                             ▼
               ┌─────────────────────────┐
               │ 精确纪年（公元前XXX年）    │ ← 年表有载，直接采用
               └────────────┬────────────┘
                            │ 作为基准，向外推理
                            ▼
               ┌─────────────────────────┐
               │ 确定推算 [公元前XXX年]     │ ← reign-year换算、世系回推
               └────────────┬────────────┘
                            │ 证据更弱时
                            ▼
               ┌─────────────────────────┐
               │ 近似推断 [约公元前XXX年]   │ ← 上下文推断、活跃期约束
               └─────────────────────────┘
```

换言之：**精确年份来自外部年表，其余年份从精确年份推理而来**。推理方法包括：

- **Reign-year换算**：已知"秦昭王元年=前306年"（年表），则"秦昭王十三年"= 306 - (13-1) = 前294年 → 确定推算`[]`
- **世系链回推**：已知"文侯卒年前746年"（年表），逆推穆侯、殇叔在位期 → 确定推算`[]`
- **人物活跃期约束**：已知人物出生/去世年（年表），事件必须落在该窗口内 → 近似推断`[约]`
- **前后锚点插值**：两个精确年份之间的事件按叙事顺序线性分配 → 近似推断`[约]`

三级确定性标注直接反映了**距离ground truth的推理链长度**：
- `（）`精确 = 零步推理（年表直接对应）
- `[]`确定推算 = 一步推理（从年表锚点换算）
- `[约]`近似 = 两步以上推理（间接推断）

这一设计决定了审查工作流的优先级：**先查年表（确认或修正精确年份），再以精确年份为基准检查周围事件的推算是否正确**。

## 四、大模型作为通用推理机器

### 4.1 为什么用大模型而非专用脚本

年代推断看似是数值计算问题（"秦昭王十三年 = 306 - 12 = 前294年"），用脚本即可。但实际场景远比纯换算复杂——它需要**语义理解、多源证据综合、矛盾检测、不确定性判断**等能力。这些正是大模型作为通用推理机器的优势。

脚本能做的是**模式匹配**：看到"%秦昭王十三年%"就查表换算。脚本做不到的是**语义推理**：

- 判断"二年"指的是哪个"二年"（哪位帝王？元年还是中元？）
- 发现"前333年苏秦说魏襄王"不可能，因为魏襄王前318年才即位
- 识别"明年"在古汉语中是链式递推还是并列叙述
- 判断《史记》"景公三年"与年表"景公十七年"哪个更可信

大模型的本质角色是：**给定ground truth（年表）和一套推理规则（SKILL），对每个事件执行多步推理并输出带置信度的判断。** 它不是纪年换算器，而是一个能阅读古文、理解叙事结构、交叉验证多源证据的通用推理引擎。

### 4.2 六种推理类型与具体示例

大模型在年代审查中执行的推理远不止简单换算。以下是六种核心推理类型，每种附带完整的推理链：

**推理类型1：年表直接查证（最简单，~800次）**

```
输入：事件"长平之战"，当前标注[约公元前262年]
推理：查年表 → 前260年条明确记载"秦白起破赵长平"
结论：修正为（公元前260年），确定性从"约"升级为精确
```

这是最基础的推理——查表比对。但它的价值在于Agent能识别"长平之战"对应年表中的哪一条记载，这需要语义理解。

**推理类型2：Reign-year换算（数学推理，~400次）**

```
输入：%定公十四年%，当前标注（公元前498年）
推理：
  (1) 查年表：鲁定公元年 = 前509年
  (2) 换算：十四年 = 509 - (14-1) = 前496年
  (3) 原标前498年与"十四年"矛盾（前498年 = 定公十二年）
结论：修正为（公元前496年）
```

看似简单的减法，但Agent必须（a）识别"定公"是鲁定公而非其他国的定公，（b）找到正确的元年基准，（c）执行换算并检测矛盾。

**推理类型3：人物共处窗口约束（逻辑推理）**

```
输入：卫灵公闻濮水琴声（与晋平公同场），当前标注[约前530年]
推理：
  (1) 查年表：卫灵公元年 = 前534年
  (2) 查年表：晋平公卒年 = 前532年
  (3) 两人共处窗口 = 前534—前532年（仅3年）
  (4) 前530年时晋平公已去世2年 → 矛盾
结论：修正为[约前533年]（窗口中值）
```

这是一种**约束满足推理**：从多个独立条件推导出一个可行的时间窗口，排除不满足约束的标注。

**推理类型4：世系链回推（多步推理）**

```
输入：晋穆侯命子之谶，当前标注[约前808年]
推理：
  (1) 已知终点：文侯卒年 = 前746年（年表）
  (2) 文侯在位35年 → 文侯元年 = 前780年
  (3) 殇叔在位4年 → 殇叔元年 = 前784年
  (4) 穆侯在位27年 → 穆侯元年 = 前811年
  (5) 穆侯七年生太子仇 = 811 - (7-1) = 前805年
结论：修正为[约前805年]
```

这是一种**链式回溯推理**：从一个已知锚点逆向推导多步，每步依赖上一步结果。脚本难以实现因为世系关系嵌在叙事文本中。

**推理类型5：多源证据综合（证据权衡）**

```
输入：苏秦说韩宣王，当前标注（前333年）
推理：
  (1) 《史记》定苏秦活动于前334—前333年
  (2) 查年表：韩宣惠王元年 = 前332年 → 前333年未即位，矛盾
  (3) 查年表：魏襄王元年 = 前318年 → 同传"说魏襄王"在前333年也矛盾
  (4) 查年表：前311年条记苏秦赴燕昭王
  (5) 出土文献（马王堆帛书）：苏秦活跃于前311—前284年
  (6) 结论：三位君主在位期均与《史记》原标年份矛盾
     → 以年表+出土文献为准，全传下移约25年
```

这是最复杂的推理类型：需要同时处理多个独立证据源、检测系统性矛盾、做出权威性判断。

**推理类型6：古汉语语义消歧（语言理解）**

```
输入：原文"後六年窦太后崩(前135)。其明年征文学。明年上初至雍"
推理：
  (1) 第一轮理解：链式递推 → 前135→前134→前133年
  (2) 第二轮发现：纪年标记%元光元年% = 前134年
  (3) 若取前133年，与标记矛盾
  (4) 重新分析"明年"语义：两个"明年"可理解为并列叙述
     （均指前135年之明年 = 前134年），非链式递推
  (5) 验证：后续事件012-011有独立标记%元光二年%=前133年
     → 前134→前133年构成合理时间序列
结论：修正为（前134年）
```

这要求Agent理解古汉语"明年"的歧义性，并通过交叉验证选择正确的语义解读。

### 4.3 SKILL作为新Ontology

传统知识工程中，ontology（本体论）是预先定义的概念体系——由领域专家设计，自上而下。本项目的SKILL文档走了一条相反的路：**自下而上、从数据中涌现的ontology。**

SKILL文档包含四个层次：

```
┌─────────────────────────────────────────────────┐
│ 第四层：审查检查清单（12项）                        │
│   操作规程——做什么、按什么顺序做                    │
├─────────────────────────────────────────────────┤
│ 第三层：推理逻辑规则（12条）                        │
│   推理方法——怎么从已知推未知                        │
├─────────────────────────────────────────────────┤
│ 第二层：错误模式（26条）                            │
│   领域知识——古籍年代标注会出什么错                   │
├─────────────────────────────────────────────────┤
│ 第一层：关键基准年表（45+锚点）                     │
│   事实基础——ground truth数据                       │
└─────────────────────────────────────────────────┘
```

与传统ontology的关键区别：

| 维度 | 传统ontology | SKILL作为ontology |
|------|-------------|------------------|
| 构建方式 | 专家预先定义 | Agent从数据中发现，自下而上涌现 |
| 知识来源 | 领域文献、专家访谈 | 130章 × 2轮 = 260次实战审查 |
| 更新机制 | 人工维护 | 自动追加（new_patterns自动入库） |
| 验证方式 | 同行评审 | 下一章/下一轮即验证（发现不准则修正） |
| 表示形式 | OWL/RDF等形式化语言 | 自然语言规则+案例（可直接嵌入提示词） |
| 执行者 | 程序代码 | 大模型作为通用推理机器 |

**SKILL是大模型可执行的ontology。** 传统ontology需要专门的推理引擎（如Pellet、HermiT）来执行推理；SKILL则利用大模型本身作为推理引擎——大模型读取自然语言规则，在具体场景中应用这些规则，输出结构化判断。

这种"自然语言ontology + 大模型推理引擎"的组合，开辟了一种新的知识工程范式：
- **无需形式化**：规则用自然语言表达，附带真实案例，大模型直接理解
- **自我生长**：新模式在审查过程中自动发现并入库
- **即学即用**：新规则追加到SKILL后，下一章立即生效
- **可传承**：第一轮的SKILL直接传给第二轮，"经验"零损耗

## 五、Agent反思管线设计：Bootstrap知识发现

### 5.1 设计思路：从零开始的自举过程

核心问题：如何让AI系统性地发现并修正自己（或另一个AI系统）的年代标注错误——**而我们事先并不知道错误长什么样？**

这是一个典型的**bootstrap（自举）问题**：没有预先定义的错误分类体系，没有标注好的训练数据，没有专家预设的审查规则。一切知识必须从审查过程本身中涌现。

设计答案：**SKILL滚动积累 + Agent逐章反思 + 脚本自动应用**的三层架构。核心理念是让Agent在审查过程中同时完成两件事：（1）修正当前章节的错误；（2）提炼错误模式反哺后续章节。知识在过程中自我生长。

```
┌─────────────────────────────────────────────────────────┐
│                    SKILL文档（滚动积累）                    │
│  错误模式(26) + 推理逻辑(12) + 基准年表(45+) + 检查清单(12) │
└──────────────────────┬──────────────────────────────────┘
                       │ 嵌入提示词
                       ▼
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│ 提示词生成    │→│  Agent反思    │→│  结果吞入     │
│ --prompt NNN │  │ 逐事件审查   │  │ --ingest NNN │
└──────────────┘  └──────────────┘  └──────┬───────┘
                                           │ 新模式回写SKILL
                                           ▼
                                   ┌──────────────┐
                                   │  修正应用     │
                                   │ apply_fixes  │
                                   └──────────────┘
```

### 5.2 三个关键脚本

**（1）`run_review_pipeline.py`** — 管线主控脚本（1,382行）

三种模式：
- `--prompt NNN`：为第NNN章生成审查提示词，嵌入年表数据、事件索引、SKILL规则
- `--ingest NNN /tmp/reflect_NNN.json`：吞入Agent反思结果，更新pipeline状态，提取new_patterns追加到SKILL
- `--report`：生成全局统计报告

**（2）`generate_review_prompts.py`** — 提示词批量生成（478行）

为130章逐一生成审查提示词，每个提示词包含：
- 该章事件索引全文
- 对应时段的年表数据
- 完整SKILL文档（26条错误模式+12条推理逻辑+45+基准年+12项检查清单）
- 7步审查工作流指令

**（3）`apply_reflect_fixes.py`** — 修正应用脚本（149行）

将Agent输出的JSON修正写回事件索引Markdown文件。关键技术细节：

```python
# 年代标注剥离与替换
def strip_years(text):
    """移除所有年代标注，保留纪年标记如%九岁%"""
    t = re.sub(r'（公元前\d+年(?:—前\d+年)?）', '', text)
    t = re.sub(r'\[约?公元前\d+年(?:—前\d+年)?\]', '', t)
    return t.strip()

# JSON字段兼容（不同轮次Agent输出字段名不一致）
for c in raw:
    if "suggested_year" not in c:
        if "corrected_time" in c:
            c["suggested_year"] = c["corrected_time"]
        elif "new" in c:

```

### 5.4 SKILL滚动积累：Bootstrap的核心机制

这是本管线最关键的设计，也是bootstrap知识发现的实现方式。

**冷启动悖论**：要审查年代错误，需要知道错误模式；要知道错误模式，需要先审查出错误。SKILL滚动积累打破了这个鸡生蛋的循环：

1. **第1章反思（冷启动）**：SKILL几乎为空，Agent仅凭年表数据和基本推理能力"裸审"。发现了"单锚点塌缩"这一最严重的结构性缺陷——001章66事件全标前2290年。
2. **第1章吞入**：`new_patterns`自动追加到SKILL文档。此时SKILL从0条变为2条。
3. **第2章反思**：SKILL已含"单锚点塌缩"模式。Agent立刻在002夏本纪中识别出相同问题（禹事件全标前2150年），修正效率显著提高。
4. **第N章反思**：SKILL已含前N-1章所有模式。Agent越来越"有经验"，不仅修正更准确，还能发现更细微的错误。

**知识增长曲线**：

```
SKILL模式数  25 ┤                                    ●●●●●●●●●●●
              20 ┤                        ●●●●●●●●●●●
              15 ┤                ●●●●●●●●
              10 ┤        ●●●●●●●
               5 ┤   ●●●●
               0 ┤●●
                 └──────────────────────────────────────────────
                  001  020  040  060  080  100  120  130 (章节)
                  冷启动──→ 快速积累──→ 逐渐饱和──→ 几乎无新增
```

这意味着：
- **早期章节**（001–010）：冷启动阶段，修正量大，新模式发现密集
- **中期章节**（030–080）：快速积累，SKILL日趋完善，修正越来越精准
- **后期章节**（100–130）：SKILL已成熟，修正集中在精细调校，几乎无新模式
- **跨轮次继承**：第二轮继承第一轮完整SKILL（25条模式），从"热启动"状态开始。这等于让一个"新手审查员"直接继承了"老手"的全部经验。

**这就是bootstrap的力量：起点为零，过程即学习，知识自我生长。**

### 5.5 审查工作流（7步）

每章Agent按以下7步执行审查：

1. **通读概览表**：检查时间列的单调性（本纪应递增，列传按叙事顺序）
2. **逐事件年表查证**：在年表中查找该事件，有载则确认/修正
3. **Reign-year换算校核**：对含纪年标记的事件，验证`基准元年 - (N-1)`
4. **确定性等级审查**：年表有载的应为`（）`精确标注，不应使用`[约]`
5. **概览-详情一致性**：概览表时间列与详情section的时间/年代推断行同步
6. **SKILL模式匹配**：逐条检查26个已知错误模式是否在本章触发
7. **输出corrections和new_patterns**

### 6.4 25条错误模式

第一轮从实战中提炼出25条错误模式，分四大类：

**结构性缺陷（6条）**：单锚点塌缩、确定性标注不足、通论章节多时代塌缩、原文精确纪年被覆盖、跨千年共享汉代锚点、年代推断行位置错位。

**换算逻辑缺陷（7条）**：前元/中元/后元混淆、景帝元年误标武帝元年、同名君主跨国混淆、魏惠王纪年off-by-one、田齐纪年系统性偏移22年、即位年与元年错位、统一前后称号语境不分。

**叙事结构缺陷（7条）**：祖先事件误用后裔年代、复合总结段落单点标注、概览详情大面积分裂、秦历岁首十月跨年误标、前后君主纪年衔接错位、封王年替代核心事件年、"N岁不得调"导致升迁序列压缩。

**史料分歧类（5条）**：锚点本身计算错误、锚点年份引用错误、《史记》系年与现代学术定年偏差、传说性人物跨世纪活动期矛盾、概览表与详情年份大面积不一致。

### 6.6 典型案例

**案例1：单锚点塌缩——五帝本纪（001）**

001章66个事件全部标为前2290年（黄帝生卒中点）。修正44处，按帝王世系重新分配：黄帝[约前2600年]→颛顼[约前2500年]→帝喾[约前2430年]→尧[约前2350—前2250年]→舜[约前2250—前2200年]。

**案例2：出土文献推翻传世文献——苏秦系年（069）**

《史记》将苏秦定在前334年，但1973年马王堆帛书《战国纵横家书》出土后学界认定苏秦活跃于前311—前284年。矛盾验证：069章中苏秦"说韩宣王""说魏襄王""说齐宣王"，三位君主在前333年均尚未即位。全传21个事件下移约25年。

**案例3：世系链回推——晋穆侯命子之谶（039-004）**

从已知终点文侯卒年前746年逆推：文侯在位35年→元年前780年→殇叔在位4年→穆侯在位27年→穆侯元年前811年→穆侯七年生太子仇=前805年。原标[约前808年]修正为[约前805年]。

**案例4：《史记》vs《左传》分歧——下宫之难（043-006）**

赵世家记"景公三年"=前597年，年表记前583年"讨赵氏"=景公十七年，两者差14年。内部验证："居十五年"后赵氏复位，从前597年+15年=前582年≈前583年，吻合。采用年表前583年，因史料分歧用`[约]`标注。

## 十一、方法论贡献与反思

### 11.1 揭示的LLM年代推断五大缺陷

本项目通过1,441处修正，系统性揭示了LLM在古籍年代推断中的缺陷：

1. **锚点处理缺陷**：单锚点塌缩、多时代章节无法分段处理
2. **换算逻辑缺陷**：改元混淆、同名混淆、即位年/元年边界
3. **确定性判断不足**：几乎不区分精确/推算/近似三级标注
4. **叙事结构理解不足**：复合段落、追叙、总结性叙述无法正确处理
5. **史料分歧盲区**：不知道《史记》与年表/出土文献存在系统性偏差

### 11.2 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 审查单位 | 逐章（非逐事件） | 章内事件有上下文关联，逐章审查可利用时间序列一致性 |
| 年代权威 | 年表优先 | 年表经现代学术考证，比《史记》原文更可靠 |
| 确定性三级 | `（）` > `[]` > `[约]` | 区分证据强度，避免"过度自信"或"过度保守" |
| SKILL积累 | 滚动追加 | 每章发现的新模式立即惠及后续章节 |
| 批量执行 | 5章并行 | 平衡效率与Agent上下文窗口限制 |

### 11.3 过程中的意外发现

- **苏秦系年推翻**：1973年出土文献使《史记》苏秦传全部年代失效，需整传下移25年
- **田齐22年系统性偏差**：《史记》与年表对田齐威王/宣王存在稳定的22年偏差
- **"明年"语义歧义**：古汉语"明年"可作链式递推或并列叙述两种理解，需上下文判断
- **下宫之难14年分歧**：《史记》与《左传》对同一事件的年代记载相差14年，是史学界的著名争议

### 11.4 局限性

- 41个事件（1.3%）仍未能定年，多为远古传说或缺乏任何锚点的事件
- 近似标注`[约]`的精度仍有提升空间，需更多跨章节交叉验证
- 未与《汉书》等其他史料进行系统性交叉对照

## 十二、技术壁垒：质量检查为什么难以复制

### 12.1 "提取容易，审查难"

用大模型从古籍中提取事件、生成结构化数据，技术门槛已大幅降低——任何人调用API即可完成。但**提取只是起点，质量检查才是真正的技术壁垒**。

本项目的核心发现是：大模型自动标注的年代有**46.6%需要修正**（1,441处修正 / 3,092个事件）。换言之，如果不做审查，近一半的年代标注是不可靠的。而且这些错误不是随机噪声，而是**系统性的、有规律的、需要领域知识才能发现的**。

这就是壁垒所在：**不是谁都能发现这些错误，即使发现了也不是谁都能系统性地修正。**

### 12.2 质量检查的三层壁垒

**第一层壁垒：你需要知道错误长什么样**

没有SKILL文档，一个全新的审查者（人或AI）面对001章66个事件全标"前2290年"，能发现问题吗？也许能——这是个明显的异常。但面对042章"郑灵公"被标为前424年（偏差181年），能发现吗？必须知道"同名君主跨国混淆"这个错误模式，并且能区分郑灵公和秦灵公。

26条错误模式就是26种"你需要知道的陷阱"。没有这套知识，审查者只能靠运气发现错误。

**第二层壁垒：你需要正确的推理方法**

发现错误只是第一步，修正需要正确的推理。以下是一些非平凡推理的例子：

| 推理任务 | 需要的能力 | SKILL中对应规则 |
|---------|-----------|----------------|
| "秦昭王十三年"换算为公元年 | 查表+减法 | 推理逻辑2：Reign-year换算 |
| 判断"二年"指哪位帝王 | 叙事上下文理解 | 错误模式4：年号/帝王混淆 |
| 判断苏秦活跃年代 | 出土文献知识 | 推理逻辑12：出土文献优先 |
| 判断景帝"前元""中元""后元" | 汉代纪年制度知识 | 错误模式3：前元/中元/后元混淆 |
| 卫灵公见晋平公的可能时间 | 多人物在位期交集 | 推理逻辑4：人物活跃期约束 |

12条推理逻辑就是12种"怎么从已知推未知"的方法。缺少任何一条，相应类型的错误就无法修正。

**第三层壁垒：你需要ground truth**

所有推理的根基是**年表数据**（45+个关键基准年）。没有"秦昭王元年=前306年"这个锚点，所有秦昭王纪年的换算都无法进行。没有"齐威王元年=前356年"这个锚点，就无法发现《史记》对田齐有22年系统性偏差。

基准年表不是随便查Wikipedia就能凑齐的——它需要经过现代学术考证的、覆盖先秦至西汉的完整帝王元年/卒年数据集。

### 12.3 SKILL作为质量壁垒的量化

三层壁垒对应SKILL的三个组成部分，可以量化其对修正的贡献：

```
                        修正能力 = f（错误模式 × 推理方法 × 基准数据）

                  ┌──────────────┐
                  │  基准年表     │  → 决定能否换算（45+锚点）
                  │  (ground truth)│     缺失 → 无法做任何reign-year换算
                  └──────┬───────┘
                         │
                  ┌──────┴───────┐
                  │  推理逻辑     │  → 决定能否推导（12条方法）
                  │  (how)       │     缺失 → 发现错误但无法修正
                  └──────┬───────┘
                         │
                  ┌──────┴───────┐
                  │  错误模式     │  → 决定能否发现（26条模式）
                  │  (what)      │     缺失 → 错误从眼前溜走
                  └──────────────┘
```

**缺少任一层，质量检查就失效。** 这是本项目最重要的结论之一：高质量古籍结构化的真正壁垒不在提取（extraction），而在审查（validation）。审查壁垒 = 错误模式 × 推理方法 × 基准数据。SKILL文档将这三层壁垒编码为一份可传承、可执行的知识体系。

### 12.4 可复制性分析

| 环节 | 可复制性 | 壁垒 |
|------|---------|------|
| 事件提取 | 高（API调用即可） | 低 |
| 自动年代标注 | 高（脚本运行即可） | 低 |
| 质量审查 | **低**（需SKILL+年表+推理能力） | **高** |
| SKILL构建 | **极低**（需130章冷启动+迭代积累） | **极高** |

最难复制的是SKILL本身的构建过程。它不是"写一份规则文档"那么简单——它是从130章 × 2轮 = 260次实战审查中涌现的领域知识，每一条模式都对应真实的错误案例。要在另一部古籍上达到同等质量，必须重新经历完整的bootstrap过程。

但一旦SKILL建成，它就成为**可传承的质量基准**——第二轮热启动的效率远高于第一轮冷启动（修正量从1,010降至431，修正精度从大幅偏差收窄到1–5年微调），证明了SKILL的传承价值。

## 结语：Agent反思冷启动——一种普适的知识发现方法论

### 从年代审查到一般方法论

本项目的核心贡献不是3,092个事件或1,441处年代修正——这些是具体成果。真正的贡献是验证了一种**普适的知识发现方法论**：

> **Agent反思冷启动**：大模型作为通用推理引擎，从零开始审查结构化数据，在审查过程中同时发现错误模式（what）和推理方法（how），将发现的知识编码为自然语言ontology（SKILL），通过滚动积累实现知识自我生长，最终收敛为一套可传承、可执行的领域规范。

这一方法论的本质是：**让AI在做事的过程中学会怎么做事**。它不需要预先定义错误分类、不需要标注训练数据、不需要领域专家手写规则。一切从冷启动开始，知识从实践中涌现。

传统知识工程的路径是：

```
专家定义ontology → 编写规则 → 程序执行 → 人工修正
```

Agent反思冷启动的路径是：

```
大模型裸审 → 从错误中发现模式 → 模式自动入SKILL → SKILL指导下一轮审查
    ↑                                                      │
    └──────────── 知识闭环，自我生长 ←────────────────────────┘
```

### 本项目中的其他验证场景

年代审查不是这一方法论的唯一应用场景。本项目的知识图谱构建管线中，至少还有四个任务天然适用同一范式：

**场景1：实体标注质量审查**

《史记》130篇57万字的实体标注（11类实体：人名、地名、官职、时间等）同样由大模型完成，同样存在系统性错误。例如：
- "蚕食"的"蚕"被误标为🌿动植物🌿（动词用法误标为名词）
- "秦"在"秦灭六国"中是&朝代&，在"秦地"中是=地名=（多义词歧义）
- 嵌套标注遗漏（`$@安国君@$`中官职与人名的嵌套关系）

可以用同样的冷启动方式：Agent逐章审查标注 → 发现"动词误标""多义词歧义""嵌套遗漏"等错误模式 → 积累为`SKILL_实体标注审查.md` → 后续章节自动避免同类错误。

**场景2：人名消歧**

《史记》中3,797个人名有87个歧义短名（"武王"可指周武王/秦武王/楚武王等7国武王），涉及261条消歧映射。消歧规则同样不是预先定义的，而是在逐章处理中发现的：
- 第一批章节发现"同谥号跨国歧义"模式
- 后续发现"同国同谥号歧义"（齐桓公小白 vs 齐桓侯午）
- 再后续发现"通称歧义"（"太子"在不同段落指不同人）

这些模式的积累过程与年代审查SKILL的冷启动完全同构。当前`SKILL_实体消歧.md`已包含三层歧义结构的完整分类——它也是从130篇实践中涌现的，而非预先设计的。

**场景3：事件关系发现**

3,092个事件之间存在因果、先后、并发、包含等关系。这些关系类型同样可以通过冷启动发现：Agent审查事件序列 → 发现"同一战役的攻守两方视角"重复 → 发现"跨章节同一事件的不同叙述" → 发现"因果链跨越世代传递"（如赵氏孤儿的仇恨链横跨三代人）。每种关系模式一旦发现，就能指导后续章节的关系提取。

**场景4：年表与正文的系统性比对**

《史记》正文与其自身附带的年表之间存在系统性偏差（如田齐22年偏差、苏秦25年偏差），这些偏差本身就是冷启动中发现的知识——没有人预先告诉Agent"注意田齐威王的纪年可能有22年偏差"，它是从046章的审查中自主发现的。同样的方法可用于《史记》与《汉书》《左传》《竹书纪年》的系统性比对。

### 方法论的三个核心条件

Agent反思冷启动适用的前提条件是：

| 条件 | 说明 | 年代审查中的对应 |
|------|------|----------------|
| **可验证的ground truth** | 有外部参照系判断对错 | 年表数据（45+基准年） |
| **可编码的错误模式** | 错误有规律、可归类、可复现 | 26条错误模式 |
| **大模型可执行的推理** | 发现和修正需要语义理解而非纯计算 | 6种推理类型 |

满足这三个条件的任务，都可以使用冷启动方法论。不满足的（如纯主观判断、无ground truth的任务）则不适用。

### 更广泛的适用领域

超越本项目，这一方法论在以下领域具有直接适用性：

- **其他古籍知识图谱**：《汉书》《资治通鉴》等纪传体/编年体史书的结构化处理，SKILL可部分复用（共享汉代纪年规则、推理逻辑），部分需冷启动（新朝代纪年规则）
- **法律文书结构化**：合同条款提取与合规审查，错误模式从实际案例中涌现
- **医学文献标注**：从医案中提取症状-诊断-处方关系，推理规则从误标中学习
- **代码审计**：安全漏洞模式的冷启动发现与SKILL积累

共同特征是：**大模型先做粗活（提取/标注），然后自己审查自己（反思），在审查中学会审查（SKILL积累），越审越准（收敛）。**
