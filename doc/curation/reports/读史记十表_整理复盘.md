## 二、实际工作流与偏离

**原方法论预设**（01i 流水线）：

```
勘探 → 清洗 → 结构化 → 句读 → OCR 错字标注 → 校验
```

**实际过程**：
- 阶段 0 勘探：快速扫过文件结构，定位 10 卷 + 提要 + 总论边界（按 `grep "读史记十表卷"`）
- 阶段 1 清洗：口头处理（去重复 `<史部,...>` 页标 + 折叠空行 + 剥 `　　` 缩进），未脚本化
- 阶段 2 结构化：与阶段 3 句读合并，边读边打 Markdown 层级
- 阶段 3 句读：按卷循环 Edit，每卷 ~800-1500 字的 prose，LLM 直接逐卷产出
- 阶段 4 OCR 错字标注：**边句读边标**，"防"字每出现必思考候选
- 阶段 5 引号作用域修复：**未预料的独立阶段**——首次 Write 输出残留大量半角引号，后期才发现需要专门修复
- 阶段 6 校验：仅做了 grep 式扫描，没有实现 `verify_ocr_preserve.py`

**最大偏离**：阶段 5（引号修复）原本不在预设流水线上，是执行过程中发现的 gap。已反馈到 01i §六作为独立阶段。

## 三、关键工程细节（有价值但不入 Skill）

### 3.1 引号修复算法的四次迭代

依次尝试的策略：

| 版本 | 策略 | 结果 |
|-----|-----|-----|
| v1 | 全局按顺序偶/奇位置 → 左/右 | 1426 个 `"` 平均转换，但因语义漏闭导致后续全错位 |
| v2 | "翻转对" 正则（`"..."` 反向 → 正向） | 反而扩散错误，10 轮后错位更多 |
| v3 | 上下文感知（前后字符判断左/右） | 规则覆盖不全，714 左/497 右不平衡 |
| v4 | **【】作用域栈 + 逐行重置** ✓ | 714 对完美配对，0 行不平衡 |

关键领悟：`【】`必须独立作用域，不是普通字符。逐行重置 outer counter 可限制单个语义 bug 的污染半径。

### 3.2 Write 工具的半角引号陷阱

当我用 Edit/Write 工具写入大段中文含引号的 prose 时，即使输入是全角 `"` `"`，输出常常被规范化为半角 `"`。这是工具层的字符转义行为，需要**后处理修复**，不能只靠写入时小心。

对下次项目：Write 完成后，**必须跑一次 §六作用域栈算法**，不要依赖 Write 时就对。

### 3.3 LLM 偏差的具体样本

三类偏差的真实触发例（已归纳入 01i §七）：

- **夹注吞并**：`主文【《索隠》曰：某者某也】继续主文` → LLM 倾向输出 `主文，【《索隠》曰，某者某也】，继续主文`（把夹注当成插入语加逗号）
- **引号层级错配**：纯双引号嵌套 `""内套""` 而非 `"外'内'外"`
- **专名连读**：`周鲁齐晋秦楚宋卫陈蔡曹郑燕吴`（表头）被 LLM 读为连续名词链

### 3.4 表格恢复的现实

OCR 对年表格式几乎完全丢失——原书的"周 秦 魏 韩 赵 楚 燕 齐"七列表格在 OCR 里只剩散乱的列表。本次未尝试还原，仅保留卷首的列名提示。表格还原需要对照影印 PDF 逐表重建，是独立的大工作量。

## 五、给下次的自己

1. **先跑字频统计再估工作量**：1% 抽样即可发现高频 OCR 错字模式。《读史记十表》的"防"字全书出现百次以上。
2. **别忽略引号修复是独立阶段**：即使方法论里写了"全角优先"，实际执行后仍会有几千处半角残留。
3. **【】作用域栈算法可直接复用**：伪代码在 01i §六。下一本书用它就够。
4. **超过 10 万字考虑工具化**：继续靠对话处理《读史记十表》级别的书是可行的，但 50 万字以上应先写 gj.cool API 包装脚本。
5. **项目复盘 ≠ 方法论**：记录"这次做了什么、踩了什么坑"到 `logs/curation/reports/`，可复用的规律升到 `skills/`。
