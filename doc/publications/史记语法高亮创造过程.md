## 1. 缘起：一个大胆的想法

这个项目的前身可以追溯到2025年2月。最初的尝试是手工编写RDF/TTL知识图谱——从高祖本纪的两个小节开始，手动创建本体定义文件`ontology.ttl`。这种纯手工方式虽然严谨，但进展极其缓慢：130篇、52万字的《史记》，靠人工标注恐怕要数年之功。

到了2026年1月，AI的能力已经发展到了一个关键拐点。Claude Code这样的AI编程助手不仅能理解古文语义，还能直接编写代码、操作文件系统。一个念头浮现出来：

**能不能让AI像IDE给代码加语法高亮一样，给古文也加上"语法高亮"？**

### 2.3 人工纠错的价值（09:14）

> "岳" 在"岳曰"中是人名

这是一个典型的人工纠错案例。古文中"四岳"既可指官职（四方诸侯之长），也可指具体人物。AI识别出了这个词，但判断错了类型。这个纠正帮助Claude理解了古文中人名的复杂性——同一个字在不同上下文中可能有不同的语义角色。

### 2.4 标记体系的设计革命（09:51）

处理到第二章时，我意识到一个根本问题：

> "原始文件的语义结构变得难以阅读了。需要更可读的标记……文档应该保留最小的语义标签，样式/渲染应该由一个单独的渲染函数或CSS来完成。"

这条指令触发了一个关键的架构转型——从内联样式的HTML标注，转向**内容与样式分离**的设计：

**之前**（内联HTML样式）：
```html
<span style="color: #8B4513; text-decoration: underline;">黄帝</span>
```

**之后**（轻量Token标记）：
```markdown
@黄帝@居=轩辕之丘=，娶於&西陵&之女。
```

这个设计突破至关重要。Token标记系统的好处是：
1. **人类可读**：标记符号简短，不影响阅读原文
2. **机器可处理**：正则表达式即可完成Token到HTML的转换
3. **内容与样式分离**：样式由CSS统一控制，修改样式不需要修改内容
4. **最小侵入**：保留原文字符，标注只是"包裹"

这一天诞生了11种Token标记符号的完整映射表：

| 标记符号 | 实体类型 | CSS类名 | 颜色 |
|---------|---------|---------|------|
| `@人名@` | 人名 | `.person` | #8B4513 棕色 |
| `=地名=` | 地名 | `.place` | #B8860B 金黄 |
| `$官职$` | 官职 | `.official` | #8B0000 深红 |
| `%时间%` | 时间 | `.time` | #008B8B 青色 |
| `&朝代&` | 朝代/氏族 | `.dynasty` | #9370DB 紫色 |
| `^制度^` | 制度/典章 | `.institution` | #4682B4 钢蓝 |
| `~族群~` | 族群/部落 | `.tribe` | #2F4F4F 深灰 |
| `*器物*` | 器物/书名 | `.artifact` | #CD853F 秘鲁棕 |
| `!天文!` | 天文/历法 | `.astronomy` | #483D8B 深蓝紫 |
| `?神话?` | 神话/传说 | `.mythical` | #8B008B 深洋红 |
| `〘动植物〙` | 动植物 | `.flora-fauna` | #228B22 森林绿 |

### 3.2 段落编号系统：致敬Doug Engelbart（05:03-13:39）

| 时间 | Commit | 内容 |
|------|--------|------|
| 05:03 | `245904c` | 段落序号渲染为小黄框，实体类型mouseover提示 |
| 12:38 | `baa5a63` | 改善列表渲染 |
| 12:52 | `6fdda1b` | **Purple Number致敬** |
| 13:39 | `15b6870` | 段落号码可点击，hash URL可分享 |

**Purple Numbers**（紫色编号）是对计算机先驱Doug Engelbart的致敬。Engelbart在1960年代提出了"增强人类智力"的愿景，其中包括给文档的每个段落分配永久、唯一的编号，使得精确引用和知识链接成为可能。

在本项目中，每个段落都有一个可点击的编号锚点：
```html
<a href="#pn-1" id="pn-1" class="para-num" title="点击复制链接">1</a>
```

最初设计为紫色背景（致敬Purple Numbers），后改为淡米色小框，视觉更柔和：
```css
.para-num {
    background-color: #FFF8E1;  /* 淡米色 */
    color: #6D4C41;             /* 深棕色 */
    font-size: 0.85em;
    border-radius: 3px;
    cursor: pointer;
}
```

## 6. 设计哲学

### 6.1 "语法高亮是当代的标点符号"

这个类比是整个项目的灵魂。古代的原始文本是没有标点符号的，句读（断句）是读书人的基本功。现代标点符号的普及，让"人人可读"成为可能。

语法高亮做了同样的事——通过颜色和样式，让读者一眼就能区分人名、地名、官职、时间，大幅降低古文阅读的认知负担。

### 6.2 克制优雅的视觉设计

CSS样式设计遵循"克制"原则：
- 颜色不过于鲜艳，使用深沉的暗色系（棕、暗红、深蓝紫）
- 背景色透明度极低（0.03），不干扰阅读
- 虚线边界而非实线，提供视觉提示但不喧宾夺主
- 字体使用"Noto Serif SC"（思源宋体），契合古籍的庄重感
- 行高设为2，给每行文字留出呼吸空间

### 6.3 大模型NER优于关键词匹配

项目明确选择了用大模型进行实体识别，而非基于关键词列表的字符串匹配，原因是：

- 古文多义词极多："和"可以是人名、动词、形容词
- 复合实体需要语义理解："夏后帝启"、"西伯昌"
- 动词性用法不应标记："蚕食"中的"蚕"、"龙颜"中的"龙"
- 上下文决定语义角色：同一个"岳"字，可能是人名也可能是官职

### 6.4 内容与样式分离

Token标记系统的设计，让`.tagged.md`文件同时服务于两个目的：
1. **人类阅读**：标记符号简短不干扰，可以直接用文本编辑器阅读
2. **机器渲染**：正则表达式一遍扫描即可转换为HTML

这种"双模"设计是实用主义工程哲学的体现。

## 7. 技术架构全貌

### 7.1 处理管道

```

    ↓ 大模型NER + 人工审核
标注文件(.tagged.md)
    ↓ render_shiji_html.py (正则替换Token→HTML span)
HTML文件(.html)
    ↓ publish_to_docs.sh
GitHub Pages 发布
```

| `docs/css/shiji-styles.css` | 11类实体样式+页面排版 |

| `build_entity_index.py` | 实体索引生成器 |
| `entity_aliases.json` | 实体别名映射（586条） |
| `disambiguation_map.json` | 语义消歧映射（644条） |

### 7.3 渲染器核心逻辑

`render_shiji_html.py`的核心是一组有序的正则替换规则（`ENTITY_PATTERNS`），处理顺序至关重要：

1. `**粗体**` 必须在 `*器物*` 之前处理（避免冲突）
2. 外层标记（`$官职$`）先于内层标记（`@人名@`）处理
3. `@人名@` 放在最后，因为人名最常作为内层标注（如`$@安国君@$`）
4. 所有正则排除 `<>` 和 `"` 字符，避免匹配已生成的HTML标签
