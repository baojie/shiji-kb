## 二、文本标注

### 2.1 标注规范（v2.5，〖〗格式）

**18类实体**使用 `〖TYPE content〗` 非对称格式标注：

| 类型 | 符号 | 说明 | 示例 |
|------|------|------|------|
| 人名 | `〖@X〗` | 历史人物、传说人物 | `〖@刘邦〗`、`〖@项羽〗` |
| 地名 | `〖=X〗` | 城邑、山川、国名 | `〖=咸阳〗`、`〖=泰山〗` |
| 官职 | `〖;X〗` | 正式任命的职衔、封号 | `〖;丞相〗`、`〖;太尉〗` |
| 身份 | `〖#X〗` | 社会角色、地位（非官职） | `〖#天子〗`、`〖#太后〗` |
| 时间 | `〖%X〗` | 年号纪年、月份、时长 | `〖%元年〗`、`〖%三月〗` |
| 邦国 | `〖'X〗` | 统一王朝、诸侯国、外邦 | `〖'汉〗`、`〖'齐〗` |
| 氏族 | `〖&X〗` | 宗族、家族、姓氏集团 | `〖&刘氏〗`、`〖&嬴氏〗` |
| 数量 | `〖$X〗` | 非时间计量：军队/距离/金额 | `〖$三万人〗`、`〖$千里〗` |
| 制度 | `〖^X〗` | 典章制度、礼法规范 | `〖^郡县制〗`、`〖^封禅〗` |
| 族群 | `〖~X〗` | 民族、部落 | `〖~匈奴〗`、`〖~三苗〗` |
| 器物 | `〖•X〗` | 礼器、兵器 | `〖•鼎〗`、`〖•印绶〗` |
| 天文 | `〖!X〗` | 星象、历法 | `〖!彗星〗`、`〖!闰月〗` |
| 神话 | `〖?X〗` | 传说、神话事件 | `〖?黄帝〗`、`〖?女娲〗` |
| 生物 | `〖+X〗` | 名词性生物 | `〖+龙〗`、`〖+百穀〗` |
| 典籍 | `〖{X〗` | 古籍书名 | `〖{春秋〗`、`〖{尚书〗` |
| 礼仪 | `〖:X〗` | 礼仪制度、宗庙祭祀 | `〖:宗庙〗`、`〖:籍田〗` |
| 刑法 | `〖[X〗` | 刑罚、法律条文 | `〖[腰斩〗`、`〖[族诛〗` |
| 思想 | `〖_X〗` | 哲学概念、文体体裁 | `〖_仁义〗`、`〖_本纪〗` |

**消歧语法**：`〖TYPE 显示名|规范名〗`
- 示例：`〖@台|吕台〗`、`〖@桓公|齐桓公〗`、`〖=函谷|函谷关〗`
- 显示短名，链接/索引用规范名
- 适用于所有18种实体类型
- 优先级高于 `disambiguation_map.json`

**段落编号**：`[1]` `[1.1]` `[1.1.1]`（不重用，永久引用锚点）

**核心原则**：
1. **只添加标记符号**，不改变原文字符（汉字、标点、引号、空格等）
2. **严禁嵌套标注**（如 `〖#〖#text〗〗`）
3. **Markdown结构元素不标注**（标题、编号、块标记等）

详见 [skills/SKILL_03a_实体标注.md](../../skills/SKILL_03a_实体标注.md)

### 10.1 Issue 管理

使用 GitHub Issues 跟踪需求和问题，按7类标签分类：

| 类别 | 说明 | 示例 |
|------|------|------|
| `enhancement` | 新功能、新特性 | 添加人物关系图谱可视化 |
| `bug` | 程序错误、数据错误 | 实体消歧规则误判 |
| `documentation` | 文档改进 | 补充事件提取说明 |
| `question` | 技术问题、设计讨论 | 如何处理表格中的嵌套关系 |
| `research` | 调研、探索性工作 | 调研知识图谱存储方案 |
| `refactor` | 代码重构、流程优化 | 重构实体标注脚本 |
| `maintenance` | 维护性工作 | 清理临时文件 |

详见 [skills/SKILL_10a_TODO和Issue管理.md](../../skills/SKILL_10a_TODO和Issue管理.md)

### 10.2 TODO 管理

项目根目录的 `TODO.md` 用于任务跟踪，与 Issue 互补：
- **Issue**：适合需要讨论、跨时间跨人的需求/问题
- **TODO**：适合当前短期任务清单

**注意**：`TODO.md` 是文件，不是 `TodoWrite` 工具。

### 10.3 每日工作日志

路径：`logs/daily/YYYY-MM-DD.md`

**生成方法**：
```bash
python logs/daily/generate_log.py 2026-04-05
```

**日志规范**：
- 时间范围：X日 07:00 ~ X+1日 07:00（凌晨0-7点算前一天）
- 必须包含微信通知格式（全角中点·，以"提交N次代码"结尾）
- 无提交日使用"太史公曰"格式 + 32字赞文（8个四字句）
- 生成日志后同步更新 `CHANGELOG.md`

详见 [skills/SKILL_10b_每日工作日志维护.md](../../skills/SKILL_10b_每日工作日志维护.md)

### 10.4 Git 版本管理

**关键规范**：

**绝对禁止**：
```bash
# ❌ 会覆盖工作区文件，造成不可逆数据丢失
git checkout <commit> -- <file>
git restore <file>
git reset --hard
```

**提交规范**：
- 只在用户明确要求时 commit
- 只提交缓存区（staged）内容
- 使用 HEREDOC 格式编写 commit message
- 首行总结（≤50字）+ 空行 + 按模块分组列出变更

**文件恢复安全方案**：
1. 查看差异：`git diff <commit> -- <file>`，手动编辑修复
2. 编写脚本比较差异，只提取需要修复的部分
3. 使用 `git stash` 保护现有修改后再恢复

详见 [skills/SKILL_10c_Git代码版本管理规范.md](../../skills/SKILL_10c_Git代码版本管理规范.md)

## 十三、文档组织规范

### 13.1 `doc/` vs `docs/` 的区分

**关键区别**：

| 特征 | `doc/` | `docs/` |
|------|--------|---------|
| **定位** | 内部工作文档 | 公开展示文档 |
| **受众** | 项目团队、开发者 | 外部用户、访客 |
| **成熟度** | 可包含草稿、WIP | 必须已定稿、审核通过 |
| **发布** | 不同步到 GitHub Pages | 同步到 GitHub Pages |
| **版本控制** | 可频繁修改 | 需要版本管理 |

**判断标准**：问自己"这个文件可以公开给不了解项目的人看吗？"
- 能 → `docs/`
- 不能 → `doc/`

### 13.2 `doc/` 目录结构

```
doc/
├── spec/                  # 技术规范
├── workflow/              # 工作流程文档（本文档所在）
├── methodology/           # 方法论研究
├── entities/              # 实体相关报告
├── predicate/             # 动词/谓词相关报告
├── events/                # 事件相关报告
├── analysis/              # 分析报告
├── pronunciation/         # 正音与拼音相关
└── reports/               # 其他报告
```

### 13.3 报告分类规则

| 目录 | 存放内容 | 示例 |
|------|---------|------|
| `doc/entities/` | 实体标注、消歧、别名等相关报告 | 实体标注统计报告、实体反思报告 |
| `doc/predicate/` | 动词标注、动词分类、谓词相关报告 | 动词标注迁移报告、动词分类体系 |
| `doc/events/` | 事件提取、年代反思等相关报告 | 事件年代反思总结 |
| `doc/analysis/` | 词频统计、推理案例等分析报告 | 词频统计表、推理案例集 |
| `doc/pronunciation/` | 正音与拼音标注相关 | 多音字词表、拼音规则 |

**重要**：
- 动词有关的报告应该放在 `doc/predicate/` 下面，不要放在 `doc/entities/` 下
- 实验性、未定稿的内容应放在 `labs/` 而非 `doc/`

### 13.4 `docs/` 目录结构（发布目录）

```
docs/
├── index.html            # 项目主页
├── chapters/             # 已发布的130篇章节HTML
├── entities/             # 实体索引页（18类 + 总览 + 编年）
├── original_text/        # 原始文本（供读者查看）
├── css/                  # 样式文件
├── js/                   # 前端脚本
├── app/                  # Web应用（地铁图等）
└── assets/               # 公开资源（图片、字体）
```

**严格禁止放入 `docs/` 的内容**：
- 工作日志、进度记录
- 实验性功能说明
- 内部讨论文档
- 草稿、待审阅内容
- 任何未经审核的内容
