## 二、新增工具 [audit_explicit_sets.py](../../../kg/entities/scripts/audit_explicit_sets.py)

对每个 `(name, expected_cat)` 组合，在 **chapter_md + 三家注** 中按类别特征模式扫描：

**方法**：
- **正证据**：支持 expected_cat 的模式（如 `登X` 支持山脉）
- **反证据**：暗示其它 cat 的模式
- **支持度** = 正证据分 / (正+反)

**判定**：
- `OK` — 支持度 ≥ 40%
- `WEAK_SUPPORT` — 0 < 支持度 < 40%
- `CONTRADICTS` — 全是反证据
- `NO_EVIDENCE` — 完全无证据（罕见地名）

**输出**：[第七轮_白名单审计.md](第七轮_白名单审计.md)（45 KB）

## 四、关键反思：审计大量 "CONTRADICTS" ≠ 大量误归

发现 **184 条 CONTRADICTS**，但多数是**信号质量问题**而非真正错误：

### 情形 A：多标签本质
如 `河东` [CONTRADICTS 建议 城邑]：河东既是区域又是郡，也是常被"攻河东"的战场。primary 区域 不错，但应该有多标签。我的正则只扫"区域"特征模式（X之地/民），对这类"被动承受"地名捕捉不足。

### 情形 B：信号规则覆盖不全
如 EXPLICIT_BUILDING 里的 `玉堂`：三家注有明确的建筑证据，但 `祠X` / `立X` / `起X` 模式可能没触发。

### 情形 C：名字泛化
如 `西方/东方/北方/南方`：方位词，非具体地点，但某些原文中做指代位置（"军东方"）被误匹配为"城邑"。

→ **审计是质量监测而非自动纠错**，需人工判读 CONTRADICTS 是真错还是假警报。

## 六、审计揭示的更深层问题

### 6.1 正则规则粒度不够

我的 CATEGORY_SIGNALS 仅捕捉**高频典型模式**。但历史地名语境极其多样：
- 河东既可以 "攻河东"（城邑）又可以 "定河东"（区域）
- 上林既是苑（建筑）又是苑中地名（类城邑）
- 敖仓 `据敖仓` `守敖仓` 被误匹配为关隘/城邑

**改进方向**：
- **接受多标签**成为常态（现在 343 条多标签，可能需要 600+）
- **context-aware**：不同章节（禹贡 vs 汉书地理志 vs 列传）的同名地名应区别对待

### 6.2 审计证据 vs 历史事实

审计基于**当前 tagged.md 的模式出现**，不是历史地理事实。因此：
- 如果某 name 在 `chapter_md` 中根本无强特征（只出现一两次做背景词），审计会给 NO_EVIDENCE 或 CONTRADICTS
- 但历史事实上它可能 100% 确定是某类（如"霸陵"明确是陵）
- **审计补充** L1 白名单决策，不替代它

### 6.3 规则冲突热点

`EXPLICIT_ANCIENT_CITY` + `KNOWN_HAN_XIAN` 有很多重叠（许多古邑在汉代为县）。审计将两者视为不同类别评分。实际上它们是**同一名字的不同时代表述**，应考虑合并或标多标签。
