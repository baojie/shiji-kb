> 文件：[`kg/entities/scripts/confidence_report.py`](../../../kg/entities/scripts/confidence_report.py)
> 首次引入：第二轮反思（2026-04-18）
> 当前版本：R7 定稿（结构自 R2 内部迭代后基本稳定）

---

## 一、目的与定位

### 1.1 解决什么问题

地名分类器 [`classify_places.py`](../../../kg/entities/scripts/classify_places.py) 对每个 place 输出一组类别（primary + secondary），但**不反映"这些分类有多可靠"**。

置信度函数为每个 `(name, category)` 二元组赋一个 **0.00–1.00** 的分数，用于：

1. **审计低置信度** —— 识别需要人工校对的条目
2. **UI 可视化** —— 高置信度徽章实色显示，低置信度半透明
3. **优先级排序** —— 后续迭代先处理低置信度
4. **质量监测** —— 每轮迭代后总体置信度分布变化

### 1.2 不做什么

- **不决定类别**：分类由 `classify()` 确定，置信度只**事后打分**
- **不自动纠错**：置信度低 ≠ 分类错，可能只是"证据稀疏"
- **不做加权投票**：不是概率，是"证据链最强强度"

## 二、架构

### 2.1 调用时机

```
classify_places.py
  ├─ classify() 对每个 place 给出 cats 列表
  ├─ build_context_hints() + build_sanjia_hints() 构建证据
  ├─ build_peer_groups() 构建 peer 图
  └─ [同步调用] confidence_report.score_evidence()
       ↓
     写 place_confidence.json  ← HTML 渲染时读取
```

### 2.2 输入签名

```python
score_evidence(
    name: str,              # place 名字
    cat: str,               # 目标类别（每个 cat 单独打分）
    refs: list[tuple],      # [(chapter_id, para_num), ...] from entity_index
    hints: dict,            # {name: {cat: hit_count}} from L3 扫描
    cp_mod: module,         # classify_places 模块引用
    all_places: set,        # 所有 place 名字集合
    peer_map_cache: dict,   # peer 邻居缓存
) → list[(rule_name, conf, note)]
```

### 2.3 输出

证据列表（每条：规则名 + 权重 + 说明），调用方取 `max(conf)` 作为该 `(name, cat)` 的置信度。

| 规则 | 权重 | 触发 |

| `L5_PEER_PROP` | 0.50 | peer 图邻居（顿号并列的地名）|

### 兜底

```
conf = 0.30   # 仅间接证据，不触发任何规则（罕见）
```

## 四、聚合方法：MAX

```python
max_conf = max(e[1] for e in evidence_list) if evidence_list else 0.30
```

**取最强证据，不累加、不加权平均**。

### 为什么选 MAX 而非其他？

| 方法 | 优缺点 |
|------|--------|
| **MAX**（当前） | 简单、可解释、稳定；但多源收敛不加分 |
| SUM | 分数会爆炸，需要归一化 |
| OR-gate (`1-(1-p1)(1-p2)`) | 概率语义正确，但**假设证据独立**，而我们的证据常高度相关 |
| 加权平均 | 稀释强证据，不合理 |

**MAX 的核心理念**：
> "只要我找到一条强证据，这就足够可信。追加弱证据不增加可信度上限。"

## 五、权重定义原则

### 5.1 锚点

- **1.00** = 人工精心维护的核心白名单（KNOWN_COMMANDERIES：43 条秦汉郡名）
- **0.95** = 手工维护 + 多次审核的白名单
- **0.80** = 自动提取 + 人工筛选
- **0.65** = 强模式单次命中 / 中等后缀
- **0.40** = 弱模式单次命中
- **0.30** = 兜底（无任何规则触发，仅靠间接继承）

### 5.2 相对关系

- L1（手工白名单） > L2.5（自动白名单） > L4（后缀） > L3（动词） > L5（传播）
- 原因：**信息浓度递减**：手工核对 → 结构性数据 → 词法模式 → 语法模式 → 图结构

### 5.3 权重选择的直觉

一条 0.85 的证据 ≈ "我有 85% 把握这是对的"。不是严格的概率语义，是**有序的信任度**。

## 六、边界情况

### 6.1 多标签评分

每个 `(name, cat)` **独立**评分。所以 `长安 → [县, 城邑]` 会分别评分：
- `(长安, 县)`: L1_EXPLICIT_ANCIENT_CITY(0.80) + L2_5_HANSHU_XIAN(0.90) → 0.90
- `(长安, 城邑)`: L1_EXPLICIT_ANCIENT_CITY(0.80) → 0.80

这对同一地名会看到**多个置信度**，是预期行为。

### 6.2 无证据兜底

如果没触发任何规则（极罕见），返回 `(0.30, "仅间接证据")`。

典型场景：通过 peer 传播得到的 cat，但 peer 本身也无强证据。

### 6.3 冲突忽略

当前**不检测冲突**。如果 L1 强证据说"山脉"，同时 L3 弱证据说"水域"，我们对两个类别分别打分：
- (name, 山脉): 0.95 （L1）
- (name, 水域): 0.40 （L3）

调用者看到两个条目，自己判断。

## 七、已知局限

### 7.1 L3 所有源混在一起

**问题**：三家注的 "X，山名" 和 chapter 的 "渡X" 都记为 `L3_HINT`。实际三家注质量远高。

**影响**：单次三家注命中只得 0.40，可能是被低估的。

**修复**（未实施）：拆成 `L3_SANJIA`（0.85）和 `L3_VERB`（0.40-0.65）。

### 7.2 单规则统治

**问题**：3 条 0.80 证据 ≡ 1 条 0.80 证据（都 = 0.80）。

**影响**：置信度无法反映"多源收敛"。

**修复**（未实施）：多源加分，如 `+0.03 × 额外强证据数`，最多加 0.10。

### 7.3 权重倒置

**问题**：`HANSHU_XIAN`（0.90，自动提取有 ~5% 噪声）> `KNOWN_HAN_XIAN`（0.85，手工精选）。

**影响**：轻微。手工清单被低估。

**修复**（未实施）：交换两者权重。

### 7.4 无冲突检测

**问题**：L1 说山脉，L3 说水域，我们独立打分不做冲突惩罚。

**影响**：同一地名两个高冲突 cat 都显高 conf，用户看到可能困惑。

**修复**（未实施）：
```python
if alt_score > 0.5 * chosen_score:
    conf *= 0.9
```

## 十、扩展点

### 10.1 增加新规则

1. 在 `WEIGHT` 表加一行 `'L*_NAME': (conf, note)`
2. 在 `score_evidence()` 相应位置加检查逻辑
3. 对应更新 [SKILL_03h](../../../skills/SKILL_03h_地名地段分类.md) 累积日志

### 10.2 权重调参

避免频繁改动（破坏 T1-T4 档位基准）。如需调整：
- 首选**加新规则**而非调旧权重
- 新权重落在相邻档位间（如 0.65 与 0.80 间可加 0.70/0.75）

### 10.3 新分类类别

R4 新增"虚构"类别时只加了 `L1_EXPLICIT_FICTIONAL` 一行。新类别只需：
- `WEIGHT` 表加 `L1_EXPLICIT_{TYPE}`
- `score_evidence()` 加 L1 检查条件
