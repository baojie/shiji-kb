## 一、问题域与适用场景

### 1.1 三种工作场景的对比

| 场景 | 问题形式 | 典型例 | 主工具 | 备注 |
|-----|---------|-------|-------|------|
| **A. 跨类型迁移** | 一类实体整体搬到另一类 | 〖^制度〗→〖[刑法〗（五刑/官刑等）| migrate_TYPE_fixes.py（always/context/new_candidates 三阶段）| 早期 v2.1-v2.2 主要模式 |
| **B. 同类细分** | 一级类稳定后划分子类 | place → 水域/山脉/关隘/郡/县/城邑…14 子类 | classify_TYPE.py（L1-L5 分层）| **本文档主题** |
| **C. 未分类降低** | 降低 U 未分类比例 | person 未分类 2642→1319 | 多轮反思 + 白名单批量 | 与 B 强耦合 |

B 和 C 通常一起出现：细分体系建起来后，自然有一批未命中任何子类的实体。本方法论把 B+C 作为统一工作流处理。

### 1.2 何时启动"同类细分"（触发条件）

三项同时满足才值得启动：

1. **一级类型标注稳定**：覆盖率 > 95%，跨类型迁移（场景 A）已清零。
2. **细分有下游价值**：HTML 筛选 / 本体构造 / 统计分析 等任何下游能直接消费子类标签。
3. **源头冲突已浮现**：观察到源头标注的边界模糊（如"官职 vs 人名"、"地名 vs 建筑"），细分是消化冲突的有效工具。

不满足这三条时，先做 SKILL_03c（按章反思）和 SKILL_03f（边界错误）再来。

### 1.3 细分 vs 其他几种近义操作

- **细分子类（subclass）**：本文档主题。一级 `place` 下分 14 个地段。**修改对象：分类器的 categories.json**。
- **细分类型标签（tag）**：源头标注层引入新 marker（如〖◆邦国〗从〖&氏族〗拆出）。**修改对象：tagged.md 文件本身**。属于跨类型迁移（场景 A）。
- **细分修饰标签（decoration）**：不改变分类体系，只加辅助标记（如〘※成语〙）。**修改对象：独立的新标注层**，与本文档方法论独立。

本文档只处理第一种。

## 三、Step 1：分类体系设计

### 3.1 子类数量建议

- **10-20 类为宜**。少于 10 说明粒度太粗（一个子类混多种语义），多于 20 说明细分过度（UI 筛选疲劳、人工维护成本高）。
- 如果单一分类维度覆盖不全，不如建两级层次（L1/L2）。**人名走了这条路**（L1 4-5 大类 → L2 细分 20+ 类）。
- **预留"误标 / 特殊"子类**（2-5 个）：任何分类工作最后都会沉淀出"边界不清、需要源头修正"的实体，直接给它们专门的 bucket。
  - 地名：`cat-mis`（误标）+ `cat-fiction`（虚构地名）+ `cat-split`（复合待拆分）= 3 个
  - 官职：`cat-mis` + `cat-person-mis` + `cat-identity-mis` + `cat-shihao-mis` + `cat-split` = **5 个**
  - 人名：低置信度用 confidence < 0.5 标识而非独立子类

### 3.2 多标签策略

**多标签不是 bug，是本质**（核心教训 #3）。历史实体天然多面：
- `司徒`：汉三公 + 上古周官
- `咸阳`：城邑 + 县
- `护军`：军职 + 列卿
- `韩信`：诸侯君主 + 将相 + 宗室

设计时明确：
1. **是否允许多标签**：强烈建议允许。强行合并=信息丢失。
2. **多标签比例预期**：10-20% 为健康。低于 5% 说明规则过严或强合并；高于 30% 说明 merge 逻辑松散（虚假叠加）。
3. **多标签排序**：按"主标优先级"升序。priority 最小的类在前（人名用这种方式，`["诸侯君主", "将相", "宗室"]`）。
4. **JSON 结构**：`{name: [cat, cat, ...]}` 而非 `{name: cat}`。

### 3.3 CSS 类命名约定

- 统一前缀 `cat-`（与 `verb-`、`lang-` 等其他标注层区分）
- 内容用英文或拼音，便于 CSS 选择器和调试：`cat-river`、`cat-jun`、`cat-jiachen`
- 误标类明确标出：`cat-mis`、`cat-person-mis`

### 3.4 JSON 输出格式

**categories.json**（分类结果）：
```json
{
  "长安": ["县", "城邑"],
  "元里": ["乡里", "城邑"],
  "轘辕": ["关隘"]
}
```

**confidence.json**（置信度）：
```json
{
  "长安": {"县": 0.90, "城邑": 0.80, "建筑": 0.55},
  "元里": {"乡里": 0.75, "城邑": 0.65}
}
```

两个文件同源——同一批分类逻辑，分别输出"类别赋值"和"每个类别的置信度评分"。

### 3.5 分类维度的选择

不同实体类型适用不同维度：

| 实体 | 维度 | 理由 |
|-----|------|-----|
| 地名 | **地段/行政级别** | 用户需要按"山/水/郡/县"筛选 |
| 官职 | **职能分级** | 三公/九卿/列卿/地方官的历史意义明确 |
| 人名 | **身份 × 时代 × 国家** | 三维交叉才能落到叶节点（如"秦国—秦末—谋臣"）|
| 器物 | 建议按**用途**（兵器/乐器/仪器/日用） | 多标签常见（剑=兵器+象征） |
| 典籍 | 建议按**经史子集 × 主题** | 经部的《诗经》既是经又是诗集 |
| 时间 | 按**时期/纪年体系** | 干支/王侯纪年 |

### 3.6 Step 1 的验收

- [ ] 子类数在 10-20 之间（或合理解释为什么超出）
- [ ] 每个子类有 2-5 条代表词（放 SKILL_03X 文档）
- [ ] CSS 类命名统一
- [ ] JSON 输出格式确定（单标签 vs 多标签）
- [ ] 预留了"误标/特殊"子类（2-5 个）
- [ ] 草拟了 SKILL_03X 文档的第一版骨架

## 四、Step 2：L1-L5 分层决策框架

### 4.1 五层的定义与设计哲学

**核心原则**：**信息浓度递减，置信度递减**。越往下的层级，信号越弱、误判率越高。

```
信息浓度 ▲
    │
L1  │  显式白名单（人工精选）              conf 0.80-1.00
    │
L2  │  章节上下文（结构信号）              conf 0.85
    │
L2.5│  权威文献自动提取（汉书等）          conf 0.85-0.95
    │
L3  │  共现模式（动词 + 实体）             conf 0.40-0.65
    │
L4  │  后缀启发式（词形规则）              conf 0.55-0.80
    │
L5  │  并列/亲属传播（结构规则）           conf 0.50-0.75
    │
    ▼ 置信度
```

### 4.2 每层详解

#### L1 显式白名单

- **定义**：手工维护的 `EXPLICIT_XXX` set，收录确信分类正确的规范名（canonical form）。
- **维护**：不放 surface form（如"昭王"歧义大），只放 canonical（"秦昭王""燕昭王"各自一条）。
- **规模**：起点 100-200 条，全覆盖阶段扩至 500-900 条。
- **分类**：**按子类**建一个白名单（如 EXPLICIT_MOUNTAIN、EXPLICIT_RIVER）。
- **代码模板**：
  ```python
  EXPLICIT_MOUNTAIN = {'泰山', '嵩山', '华山', '终南', '首阳', ...}
  EXPLICIT_RIVER = {'黄河', '渭水', '泾水', '汉水', ...}
  ...
  def classify_l1_whitelist(canonical):
      results = []
      if canonical in EXPLICIT_MOUNTAIN: results.append('山脉')
      if canonical in EXPLICIT_RIVER: results.append('水域')
      ...
      return results
  ```

#### L2 章节上下文（结构信号）

- **定义**：利用古籍章节的专题性——某章大比例出现 X 类实体，就把该章未分类实体默认判为 X 类。
- **例**：
  - 地名：侯者年表（017-021）→ 县（汉代侯国 = 县级）
  - 官职：匈奴列传（110-111）→ 外邦职
  - 人名：仲尼弟子列传（067）→ 学者文士
- **阈值**：某实体在该章出现次数 / 该实体总出现次数 ≥ 50% 才触发。
- **注意**：L2 是**观察相关性**而非**因果分类**。必须作为兜底——见 §4.4 merge 优先级中 L2 在最后。

#### L2.5 权威文献自动提取

- **定义**：从《汉书·地理志》《汉书·百官公卿表》《仲尼弟子列传》等权威源头自动抽取条目，加入白名单。
- **脚本模板**：`extract_hanshu_dili.py`、`extract_hanshu_baiguan.py`
- **产出**：`hanshu_dili.json`（1049 县 + 104 郡）、`hanshu_baiguan.json`（116 项官名）
- **权重**：高于手工白名单（0.85-0.95 vs L1 的 0.80-1.00），因为自动提取覆盖全面，但有 ~5% 噪声（所以与手工精选的 EXPLICIT_KNOWN_* 并用）。

#### L2.5bis 别名继承（仅人名）

- **定义**：对已有多 surface 映射的实体，当 canonical 已分类时，所有 surface 自动继承。
- **代码**：
  ```python
  # 若"秦昭王"已归帝王，则它的 surface "昭王"/"昭"也归帝王
  for surface, canonical in aliases.items():
      if canonical in categorized:
          inherit[surface] = categorized[canonical]
  ```
- **唯一只有人名才需要**（地名/官职的别名少）。

#### L3 共现模式

- **定义**：基于上下文动词 + 实体模式的分类推断。
- **例**：
  - 地名：`登X` → 山脉；`渡X` / `济X` → 水域；`都X` → 城邑
  - 官职：`拜为X` / `封X` / `立X为王` → 对应职类
  - 人名：`X为丞相` / `X即位` → 对应身份
- **投票机制**：扫描全库，对每个实体累积共现动词的票数。单票不采纳（噪声大），≥ 2 票才采纳；refs ≤ 3 的低频实体可 1 票即采。
- **权重**：
  - 强（≥ 3 次命中）：0.65
  - 中（2 次）：0.55
  - 弱（1 次）：0.40

#### L4 后缀启发式

- **定义**：基于词形后缀的规则匹配。
- **例**：
  - 地名：`X山` → 山脉；`X水`/`X江`/`X河` → 水域；`X郡` → 郡；`X县` → 县；`X宫` → 建筑
  - 官职：`X令` → 县乡吏；`X将军` → 军职；`X尉` → 军职；`X大夫` → 歧义（爵/官/身份）
- **权重**：
  - 强后缀（`山`/`水`/`郡`）：0.80
  - 中后缀（`邑`/`城`/`宫`）：0.65
  - 弱后缀（`阳`/`阴`/`平`）：0.55

#### L5 并列/亲属传播

- **定义**：利用文本中的并列结构（顿号、并举）或亲属关系推断。
- **例**：
  - `〖=A〗、〖=B〗、〖=C〗` 三者并列 → A 已分类为山脉，则 B、C 同类
  - `〖@父名〗之子〖@子名〗` → 子继承父的"帝王/诸侯"身份
- **严格性**：只匹配**顿号**并列（不扩展到逗号，误判多）；亲属关系需要明确标记（`之子/之弟/其母`）。
- **权重**：0.50-0.75。

### 4.3 三种实体的 L1-L5 实现差异对比

| 层 | 地名 | 官职 | 人名 |
|---|-----|------|-----|
| L1 | 12 个 EXPLICIT_*（MOUNTAIN/WATER/...）| 10 个 EXPLICIT_*（SANGONG/JIUQING/...）| 按身份分的 EXPLICIT_*（EMPEROR/RULER/...）|
| L2 | 侯者年表 → 县、河渠书 → 水/建筑 | 侯者年表 → 爵位、匈奴传 → 外邦职 | 专题列传（067 → 学者、匈奴 → 外邦人）|
| L2.5 | 汉书地理志（1049 县 + 104 郡）| 汉书百官公卿表（116 项）| rulers.json（656 君主）+ 私家文档 |
| L2.5bis | 无 | 无 | **别名继承**（3322 条别名）|
| L3 | 24+ 动词模式（登/渡/城）| 20+ 动词模式（拜/封）| 15+ 模式（为 X / 即位）|
| L4 | 30+ 后缀规则 | 后缀 + 诸侯国消歧动态规则 | 封号后缀（单于/夫人/公子）+ 国名+三字名 |
| L5 | 顿号并列 + 拆分继承 | 并列（少用）| 并列 + 家族/同僚关系 |

### 4.4 merge 优先级：L1 > L2.5 > L3 > L4 > L2

这是**官职工作流第四轮反思**最重要的发现。初版用 L2 作为中层规则导致大量虚假多标签，改为**L2 仅兜底**后虚假多标签从 30 降到 16。

```
优先级链：
L1 (手工白名单)
  > L2.5 (权威文献)
    > L3 (上下文)
      > L4 (后缀)
        > L2 (章节兜底，仅当 final 为空时应用)
```

### 4.5 4 个关键 Guard 条件

#### Guard 1：误标子类命中早返回
```python
if final and set(final) <= {CAT_MIS, CAT_PERSON_MIS, CAT_IDENTITY_MIS, CAT_SHIHAO_MIS, CAT_SPLIT}:
    return final  # 不再叠加其他分类
```
**原因**：误标是源头问题，不能叠加其他分类。

#### Guard 2：特殊规则强制覆盖
```python
# 汉诸侯国官：X (诸侯国) + 官 → 强制郡国长吏
if _is_vassal_official(canonical) or canonical in EXPLICIT_JUNLI_VASSAL:
    return [CAT_JUNLI]
```
**原因**：级别纠正必须排除上层 L1 的泛配。

#### Guard 3：L3 不覆盖具体类别
```python
_has_specific = final and not (set(final) <= {CAT_GENERIC})
if l3_cat and l3_cat not in final and not _has_specific:
    final.append(l3_cat)
```
**原因**：L3 规则如"将兵"会把丞相误分为军职，必须检查 L1 是否已给出具体类别。

#### Guard 4：L2 仅兜底
```python
if l2 and not final:  # final 必须完全为空
    final.append(l2)
```
**原因**：章节相关性 ≠ 因果分类，最后才用。

### 4.6 Step 2 的验收

- [ ] L1 白名单规模 ≥ 200 条
- [ ] L2 章节规则覆盖至少 8 个专题章节
- [ ] L2.5 权威文献脚本可重跑（extract_*.py）
- [ ] L3 模式库 ≥ 15 条
- [ ] L4 后缀规则 ≥ 10 条
- [ ] L5 并列规则实现（即便暂未触发）
- [ ] merge 函数包含 4 个 Guard
- [ ] R1 输出：未分类 35-45%

## 五、Step 3：多轮反思范式

### 5.1 轮次节奏（以地名 7 轮为模板）

| 轮 | 主题 | 起点 | 终点 | 主要动作 |
|---|------|------|------|---------|

| **R2** | 全覆盖 + 源头修正 | 35-45% | 0-10% 未分类 | 模式挖掘、EXPLICIT 大扩充（+200-400）、**细分误标子类**、**源头修正** |
| **R3** | 证据富化 + 规则微调 | 低置信 200+ | 低置信 100+ | build_evidence_bundles.py、规则微调、移入/移出少量条目 |

| **R7** | 白名单健康审计 | - | - | audit_*.py：死词/冲突/孤立词 |

### 5.2 R1 冷启动详解

**目标**：35-45% 未分类。

**关键动作**：
1. 脚本骨架：`classify_TYPE.py` 复用 `classify_places.py` 模板（1300 行）
2. 手工白名单：每个子类列 10-30 条核心代表词
3. L4 后缀：把最明显的后缀规则写入（`X山` → 山脉等）
4. 先跑一次，看 distribution：
   - 若某子类 < 5 条 → 粒度太细，考虑合并
   - 若某子类 > 300 条 → 粒度太粗，考虑拆分

**信号诊断**：
- 未分类 > 60%：L1 白名单太少或 L4 后缀规则太严
- 未分类 < 20%：L4 规则过宽，可能有误判（抽样检查）

### 5.3 R2 全覆盖 + 源头修正 详解（最重要的一轮）

**目标**：未分类 → 0-10%，同时修复源头标注问题。

#### 5.3.1 未分类上下文导出

```bash
python scripts/dump_blank_TYPE.py > doc/entities/TYPE反思/第一轮_上下文.md
```

该脚本对每个未分类实体导出：
- canonical name + refs 数量 + 总出现次数
- 每个 ref 处的上下文（前后 30-50 字）

人工阅读 Top 100-200 未分类，按"共同特征"分组。

#### 5.3.2 模式挖掘（批量规则提炼）

每组特征 → 一条规则或一批白名单：

| 共同特征 | 归类动作 | 数量级 |
|---------|--------|-------|
| "X（之 / 为）官职" | L3 新模式 | +20-50 条命中 |
| "X 单于"、"X 贤王" | EXPLICIT_FOREIGN 扩充 | +10-30 |
| "丞相何"、"卫将军青" | EXPLICIT_NEED_SPLIT（官职+人名）| +30 |
| "杞湣公"、"杞隐公" 系列 | EXPLICIT_RULER 批量（春秋国君）| +14 |
| 谥号裸出现（"哀"、"武"）| EXPLICIT_SHIHAO_MIS | +98 |

#### 5.3.3 源头修正（最易被忽略但最根本）

很多"未分类"本质是**源头标注错误**。R2 必须回溯源头：

- **复合实体拆分**：`〖=新丰鸿门〗` → `〖=新丰〗〖=鸿门〗`（地名 R2 拆 20 处，影响 25 章）
- **误标剔除**：`〖=乾封〗`（修辞虚指）、`〖=善置〗`、`〖=湘君〗`（神灵非地）等（地名 R2 剔除 29 处）
- **边界调整**：`〖;卫将军青〗` → `〖;卫将军〗〖@卫青〗`

脚本模板：`fix_source_mislabeled.py`。

#### 5.3.4 细分误标子类（官职案例的专属收获）

用户在第二、三轮反思中根据实际误标模式要求新增子类：
- `cat-person-mis`：官号+人名代人（如 `老上单于`）
- `cat-identity-mis`：本应 `〖#〗` identity 的词（`太后`、`太子`、`布衣`）
- `cat-shihao-mis`：裸谥号（`哀`、`武`、`纣`）
- `cat-split`：待拆分的复合体

**判断信号**：剩余未分类中发现**系列化的边界问题**（20+ 条共同特征），且这些问题**无法通过现有子类描述**。此时细分子类是最诚实的解。

### 5.4 R3 证据富化详解

**目标**：为低置信度条目构建证据包，帮助人工做最终决策（不是自动修正）。

**核心教训**：**低置信度 ≠ 误分类**。可能只是证据稀疏。冒然修正会引入噪声。

#### 5.4.1 evidence bundles 构建

```bash
python scripts/build_evidence_bundles.py > kg/entities/data/TYPE_evidence_bundles.json
```

每个低置信度实体的 bundle 包含：
- refs 全部出处（[章节ID, 段落号]）
- context_windows：前后 30 字的上下文（取前 6 处）
- cooccur_tags：同句的官职/人名/动词
- sanjia_notes：相关的《集解》《索隐》《正义》注释

文件大小：3-5 MB 对 1000-2000 低置信度条目。

#### 5.4.2 规则微调（而非大改）

R3 只做小调整：
- 根据 evidence 移动 5-20 条条目（明显证据指向新类）
- 新增 5-10 条 L3 模式（已发现但初版漏掉的）
- 不改 merge 逻辑、不大批量改白名单

R3 的置信度分布应基本不变（地名 R3: 135→134，说明 R2 后已接近饱和）。

### 5.5 R4 梯度资源策略

**核心洞察**：不同置信度的条目值得投入不同资源。

| 档位 | conf 范围 | 条数占比 | 策略 |
|-----|---------|---------|-----|
| T1 | ≥ 0.90 | 50%+ | 抽样 5% 验证即可 |
| T2 | 0.70-0.89 | 25% | 本地 evidence 扫描，扩 L1 白名单 |
| T3 | 0.50-0.69 | 15% | 轻量 LLM 推理（Haiku）批量验证 |
| T4 | < 0.50 | 5-10% | Agent + WebSearch 深研 |

**执行顺序**：T4 → T3 → T2 → T1（高价值优先）。

**收益递减**：T4 深研每条耗 1-5 分钟但修正价值高；T1 抽样每条 5 秒但修正率 < 5%。

### 5.6 R5 UI 置信度可视化

把置信度以视觉形式暴露给用户：

```css
.entity-chip {
    opacity: calc(0.40 + 0.60 * var(--conf));
}
```

- conf = 1.0 → opacity = 1.0（完全不透）
- conf = 0.5 → opacity = 0.70（半透）
- conf = 0.3 → opacity = 0.40（淡出，几乎看不到）

hover tooltip："置信度 X（越接近 100 越可靠）"

这比显示数字更直观，也不干扰阅读。

### 5.7 R6 系统性概念质保

**核心洞察**：**系统性概念混淆比单条值错更有价值**。

做法：对高置信度（≥ 0.90）条目抽样 5%，人工检查。寻找的不是单条错误，而是 **cohort 级概念混淆**。

**地名 R6 经典发现**：EXPLICIT_HAN_VASSAL_KINGDOMS 白名单里有 9 条本该归"诸侯王国（郡级）"的条目被误归"区域"。修正一次就解决一批。

**官职 R6 经典发现**：L2 中层应用导致的虚假多标签（30 → 16）——一次 merge 逻辑修改清掉 14 条。

### 5.8 R7 白名单健康审计

见 §七 Step 5 详述。

## 六、Step 4：置信度函数

详见 [SKILL_08f 置信度函数设计](../../skills/SKILL_08f_置信度函数设计.md)。本节只列适用于细分分类的要点。

### 6.1 WEIGHT 权重表模板

```python
WEIGHT = {
    # L1 最强
    'L1_KNOWN_CORE': (1.00, '核心手工白名单'),
    'L1_EXPLICIT_STRONG': (0.95, '类型专属白名单'),
    'L1_EXPLICIT_WEAK': (0.80, '次强白名单'),
    
    # L2.5 权威文献
    'L2.5_HANSHU_AUTO': (0.90, '汉书自动提取'),
    'L2.5_HANSHU_MANUAL': (0.85, '汉书手工精选'),
    
    # L2 章节兜底
    'L2_CHAPTER_MAJOR': (0.85, '侯者年表 / 匈奴传等'),
    
    # L3 共现模式
    'L3_HINT_STRONG': (0.65, '模式命中 ≥ 3 次'),
    'L3_HINT_MEDIUM': (0.55, '模式命中 2 次'),
    'L3_HINT_WEAK': (0.40, '模式命中 1 次'),
    
    # L4 后缀
    'L4_SUFFIX_STRONG': (0.80, '强后缀（山/水/郡）'),
    'L4_SUFFIX_MEDIUM': (0.65, '中后缀（邑/城/宫）'),
    'L4_SUFFIX_WEAK': (0.55, '弱后缀（阳/阴/平）'),
    
    # L5 传播
    'L5_SPLIT_INHERIT': (0.75, '拆分继承'),
    'L5_PEER_PROP': (0.50, '并列传播'),
    
    # 兜底
    'FALLBACK': (0.30, '无规则触发'),
}
```

### 6.2 聚合方法：MAX

不累加、不平均，只取最强证据：

```python
def compute_confidence(evidence_list):
    if not evidence_list:
        return 0.30
    return max(WEIGHT[ev][0] for ev in evidence_list)
```

**理由**：
- "只要找到一条强证据，就足够可信"
- 避免弱证据稀释强证据
- 实现简单，解释直观

**已知局限**（4 条）：
1. **L3 源混在一起**：三家注与章节动词都是 L3，未拆分。未来可分为 L3_SANJIA(0.85) 和 L3_VERB(0.40-0.65)。
2. **多源收敛无加权**：3 条 0.80 证据仍然等于 1 条 0.80。
3. **权重倒置**：自动提取 > 手工精选（L2.5_HANSHU_AUTO 0.90 > L2.5_HANSHU_MANUAL 0.85），但自动提取有 5% 噪声。
4. **无冲突惩罚**：L1 说山脉、L3 说水域时，两个都显高 conf。未来可加 `if alt_score > 0.5 * chosen_score: conf *= 0.9`。

## 七、Step 5：evidence + 白名单审计

### 7.1 evidence bundles：低置信度的救命稻草

见 §5.4.1。核心脚本 `build_evidence_bundles.py`，输出 JSON 供：
- 人工审核（把 JSON 渲染成 HTML 表，每行一实体）
- LLM 批量推理（T3 档）
- Agent 深研（T4 档）的上下文输入

### 7.2 白名单审计：audit_explicit_sets.py

扫描所有 EXPLICIT_XXX 集合，做三项检查：

#### 检查 1：死词

**定义**：白名单有但 entity_index 中无（从未在文本出现）。

**典型原因**：
- 预防性添加
- 拼写错误（`中书` 实际应是 `中书令`）
- 源头已修正（词被删除）

**处理**：删除明显拼写错误，保留可能预防性的（标注为"预防性"注释）。

**规模**：通常 5-10% 白名单条目会是死词（官职 71/~900 条）。

#### 检查 2：冲突

**定义**：同一词出现在多个 EXPLICIT_XXX 中。

**分类**：

| 冲突类型 | 例 | 处理 |
|---------|-----|-----|
| 合理多标签 | `司徒` ∈ SANGONG + ANCIENT | 保留，文档中解释"时代分化" |
| 族内冲突（误标族）| 同在 MIS 和 PERSON_MIS | 选主类，删从类 |
| 跨族冲突 | 同在 JIUQING 和 IDENTITY_MIS | 人工审查，改源头或去弱类 |

**官职案例**：35 条冲突 → 11 条（其中 16 合理多标签保留）。

#### 检查 3：孤立词

**定义**：refs < 2 的白名单条目（在文本中几乎未出现）。

**处理**：
- refs = 1 的冷僻词：保留（可能是合法但低频词）
- refs = 0：视为死词

### 7.3 审计输出模板

```markdown
# {Type} 白名单审计报告

## 死词（{N} 条）
- EXPLICIT_XXX: `词1`, `词2`, ...

## 冲突（{N} 条）

### 合理多标签（{N} 条）
- `司徒` ∈ SANGONG + ANCIENT —— 时代分化（汉三公 + 周官）

### 族内冲突（{N} 条）
- `X` ∈ MIS + PERSON_MIS —— 应归 PERSON_MIS

### 跨族冲突（{N} 条）
- `Y` ∈ JIUQING + IDENTITY_MIS —— 应改源头标注为 identity

## 孤立词（{N} 条）
- EXPLICIT_XXX: `词1` (refs=1), ...
```

### 7.4 审计的哲学：监测而非自动纠错

**CONTRADICTS（审计不支持）不等于错归**。可能的原因：
1. **多标签本质**（60%）：河东既是区域又是郡又是战场
2. **信号规则覆盖不全**（30%）：玉堂有建筑证据但 CATEGORY_SIGNALS 没覆盖"玉 + 堂"
3. **名字泛化**（10%）：方位词"西方"指代位置而非具体地点

因此审计只生成**报告**，人工决定是否改。不建议做自动纠错脚本（地名 R7 验证：CONTRADICTS 184 条中只有 5 条真错，其余都是其他原因）。

### 8.2 可选：TTL → Markdown → HTML 三层管道（概念树）

**适用**：需要概念分类树（如人名的 L1/L2/L3 层次）。

**三层结构**（人名案例）：

```
kg/taxonomy/person.ttl  ←── 权威源，手工编辑 L1-L3 类结构
      │
      │ build_taxonomy.py
      ↓
docs/kg/person_taxonomy.md  ←── Markdown 分类树
      │
      │ rebuild_person_tree_html.py
      ↓
docs/kg/person_tree.html  ←── 交互式 HTML（details/summary + 搜索 + 实例芯片）
```

**TTL 关键属性**：
```turtle
per:帝王 a owl:Class ;
    rdfs:subClassOf per:王室 ;
    rdfs:label "帝王"@zh ;
    :order 1 .              # 时序控制

per:禹 a per:夏 ;
    rdfs:label "禹"@zh ;
    :count 42 .             # 文本中出现次数
```

**为何要自动生成而非手写**：
1. 数据同步：每次重分类或补实例都要改 TTL + HTML，手写两处 source of truth 会漂移
2. 计数维护：各类人数、主标/总标要实时同步，手写易错
3. 颗粒度调整：拆分/合并 owl:Class 时脚本自动重生树

**详见 SKILL_06e 概念分类树构建**。

## 十、六条核心教训

### 教训 1：源头修正优先（R2 必做）

修正源标注比修正分类结果更根本。地名 R2 拆了 20 处复合地名、剔除 29 处误标，这些工作**不能留到后面**——后面的分类逻辑在错误数据上叠加，会越改越乱。

**检查信号**：R2 若没做过任何源头修正，说明审查不够细。

### 教训 2：低置信度 ≠ 误分类

**冒然把低置信度条目自动改类是最常见的 R3 错误**。低置信度可能仅是：
- 文本中提及太少（refs < 3）
- 三家注未注明
- 该子类的 L3 模式还未覆盖全

R3 应构建 evidence bundles 让人看证据，**不是**自动改。

**检查信号**：R3 如果自动改了 > 20 条，回头看是否有误改。

### 教训 3：多标签是本质，不是 bug

历史实体天然多面。设计时允许多标签（预期 10-20% 比例）。虚假多标签的根源是 **merge 逻辑错**（L2 当中层叠加），不是多标签本身。

**处理方式**：
- 保留合理多标签
- 每条多标签能写一句话说明"为什么两个类都对"（写进 SKILL 文档）
- merge 逻辑 Guard 修虚假叠加

### 教训 4：L2 章节规则只作兜底

L2 是**观察相关性**，不是**因果分类**。某词在侯者年表 50% 出现 ≠ 该词就是爵位，可能只是该章背景。

**必须**：`if l2 and not final: final.append(l2)`（见 §4.4 merge 链 + §4.5 Guard 4）。

### 教训 5：系统性概念混淆 > 单条值错

一次 R6 抽样发现 9 条 HAN_VASSAL_KINGDOMS 混归 vs 逐条深研 R4 改 17 条——前者**投入 1 小时，修正 9 条 cohort**，后者**投入 5 小时，修正 17 条单点**。

**策略**：R4/R5 之后一定安排一轮 R6 抽样审计，找 cohort 级问题。

### 教训 6：细分子类比强行合并更诚实

官职 R2/R3 发现 260 条难分的根源是**源头标注的边界模糊**（官职 vs 人名 vs 身份 vs 谥号）。此时不要强合并成一个大 `CAT_MIS`，应细分 5 个 mis 子类。

**决策标准**：
- 如果观察到 ≥ 20 条系列化的边界问题
- 这些问题无法被现有子类描述
- 则新增子类比强合并更诚实

## 附 C：已知局限与未来方向

### 已知局限

1. **置信度函数的 MAX 聚合不反映多源收敛**（§6.2 局限 2）
2. **权重倒置**（§6.2 局限 3）
3. **L3 源头未拆分**（§6.2 局限 1）
4. **无冲突惩罚**（§6.2 局限 4）
5. **evidence bundles 人工审阅成本高**：3.3 MB JSON 难以在浏览器快速浏览，需专门 HTML 审阅器
6. **CONTRADICTS 审计的假阳性率高**（§7.4）：184 条中只有 5 条真错

### 未来方向

1. **统一的 classify_ENTITY.py 框架**：把三个分类脚本的共性抽到 `kg/entities/scripts/common/classify_base.py`，新实体类型只需填配置
2. **evidence bundles 审阅器**：HTML 表格 + 筛选 + 标注决定的界面
3. **自动化的 cohort 检测**：R6 的人工抽样可部分自动化——统计每个白名单的"内部置信度分布熵"，熵高的白名单优先审查
4. **跨实体分类的一致性检查**：人名 ⊆ 对应官职 / 地名 → 封国（相互约束）
5. **L3 拆分为 L3_SANJIA + L3_VERB** 两个子层，分别赋权
6. **增加加权平均选项**：对"多源收敛"场景（≥ 3 条 0.80 证据）做小幅加分
