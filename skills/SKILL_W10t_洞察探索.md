# SKILL W10t: 洞察探索（H20）

> "洞察是知识库的增值。W14 是矿，H20 是选矿站——筛出有价值的洞察，排好队，等待深挖。"

---

## 一、何时执行

| 触发场景 | 说明 |
|---|---|
| W14 输出新洞察假设时 | 立即接收，写入队列 |
| 质量迭代主循环中发现意外关联 | 即时记录 |
| 用户提出某种假设需要验证 | 写入 P1 队列 |
| 每 30 轮迭代周期检查一次 insight_queue | 清理/重排优先级 |

**重要**：H20 只负责**接收和调度**，洞察发现和验证由 `SKILL_W14_洞察发现.md` 执行。

---

## 二、洞察来源渠道

### 渠道1：W14 自动输出

W14 在分析实体关联时，自动产生洞察假设并写入：

```bash
cat wiki/logs/butler/insight_queue.md
```

### 渠道2：质量迭代主循环偶发发现

在处理其他 H 类任务时，偶然发现：
- 两个看似无关的人物有意外关联
- 某个模式反复出现（如某种策略屡败屡用）
- 数据统计出现异常值（某时代人物特别密集）

### 渠道3：用户提出假设

```
用户："我注意到《史记》中的刺客都与侯国有关，能验证吗？"
→ 记录为洞察假设，加入 P1 队列
```

---

## 三、执行步骤

### Step 1：接收洞察假设

无论来源，将洞察假设格式化为标准条目：

```markdown
## 洞察假设 #NNN

**假设**：[一句话陈述假设，如"战国纵横家普遍有出身低微→权贵的成长轨迹"]

**来源**：W14 自动 / 用户提出 / 主循环偶发

**证据线索**：
- 已观察到：[支持假设的初步证据]
- 待验证：[需要检索的数据/引文]

**优先级**：P1 / P2

**PN 证据数**：N 条（已知）

**状态**：open / in_progress / verified / rejected
```

### Step 2：分配优先级

| 优先级 | 条件 |
|---|---|
| P1 | 已有 ≥2 条 PN 证据支持；用户直接提出；对理解史记有重要意义 |
| P2 | 仅有 1 条证据线索；边缘话题；纯偶发观察 |

### Step 3：写入 insight_queue.md

```bash
# 追加到队列文件（Append-Only 原则）
cat >> wiki/logs/butler/insight_queue.md << 'EOF'

## 洞察假设 #NNN
[格式化条目内容]
EOF
```

### Step 4：维护队列（每 30 轮）

```bash
# 检查队列状态
cat wiki/logs/butler/insight_queue.md | grep "^**状态**" | sort | uniq -c

# 清理 verified/rejected 的老条目（归档而非删除）
# 将已完成条目移至 insight_archive.md
```

---

## 四、成功标准 / 完成条件

- [ ] 新洞察假设在当轮写入 insight_queue.md
- [ ] 每个条目包含完整字段（假设/来源/证据/优先级/PN数/状态）
- [ ] P1 条目自动通知 W14 准备执行
- [ ] 本轮未执行任何洞察验证（那是 W14 的工作）
- [ ] 每 30 轮清理一次队列（归档 verified/rejected）
