## 一、核心主线：发现误标 → 诊断 → 纠错

```
┌─────────────────────────────────────┐
│ 多轮反思（R1-R7）                    │
│  ├─ R2 未分类驱动                    │  → 发现
│  ├─ R3 低置信 + evidence             │   
│  ├─ R6 高置信 cohort 抽样            │  
│  ├─ R7 白名单审计（死词/冲突）       │  
│  └─ 跨工序反馈（03c / 03f）          │  
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│ 误标分类诊断                         │  → 诊断
│  ├─ 源头错（tagged.md 标错类）       │   
│  ├─ 分类器错（规则未覆盖）           │  
│  └─ 边界模糊（源头标注体系本身模糊）  │  
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│ 按类型选择纠错手段                   │  → 纠错
│  ├─ 源头批量修（fix_source）         │   
│  ├─ 规则调整（白名单/Guard/L3）      │  
│  └─ 新增误标子类（cat-*-mis）        │  
└─────────────────────────────────────┘
```

误标不是一次找完的，是多轮反思里不断浮现的。每轮发现一类新误标就追加一种发现途径和对应纠错手段。

## 二、如何发现误标（8 种途径，按产率排序）

### 1. 未分类驱动（R2 主力，产率最高）

**做什么**：导出未分类实体的上下文，按共同特征分组。

**脚本**：`python kg/entities/scripts/dump_blank_TYPE.py > doc/entities/TYPE反思/第N轮_上下文.md`

**发现什么**：
- 共现特征显示它本不属于此类（如官职中"老上单于"的共现词是"击/围/击破"而非"拜/封"）
- 复合实体（如"新丰鸿门"应拆分）
- 源头修辞虚指（如"乾封"在"封"字前是动词不是地名）

**三条工作流实例**：
- 官职 R2：260 条未分类 → 发现 102 条人名误标（"X单于"、"丞相+人名"）
- 地名 R2：52 条完全空白 → 29 条误标剔除 + 20 处复合地名拆分
- 人名 R2：2103→1803 → 白名单扩 80 条（"杞湣公/杞隐公"系列）

### 2. 白名单冲突（R7 必做）

**做什么**：扫描同一词是否出现在多个 EXPLICIT_* 白名单中。

**脚本**：`python kg/entities/scripts/audit_official_whitelists.py`

**诊断**：
- **族内冲突**（如同在 MIS + PERSON_MIS）→ 误配，选主类删从类
- **跨族冲突**（如同在 JIUQING + IDENTITY_MIS）→ 需改源头或去弱类
- **合理多标签**（如 `司徒` ∈ SANGONG + ANCIENT）→ 保留并在文档解释

**产率**：官职 R7 一次发现 35 条冲突，修 24 条误配。

### 3. 白名单死词（R7）

**做什么**：找出白名单有但 entity_index 无的条目。

**含义**：
- 拼写错误（如 `中书` 应是 `中书令`）
- 源头已删除
- 预防性条目（保留）

**产率**：官职 71 条死词 / 900 白名单条目。

### 4. 高置信度抽样（R6，产率低但单条价值高）

**做什么**：对 conf ≥ 0.90 的条目抽样 5%，人工检查。

**发现什么**：**cohort 级概念混淆**（一批相同根源的错归）。

**实例**：
- 地名 R6 发现 9 条 HAN_VASSAL_KINGDOMS 被混归"区域"
- 官职 R4-R5 发现 14 条虚假多标签源自 L2 中层应用

**一次 R6 发现的 cohort 常等于 R4 逐条深研 20 条的价值**。

### 5. 共现统计异常

**做什么**：对每个实体计算其共现词 distribution，与所声称类型的典型共现对比。

**手动做法**：在 `kg/entities/data/entity_index.json` 里看某实体的 refs，到 tagged.md 里看它的上下文。

**实例**：
- 官职 "太后" 的共现词中"生/子/母/废"占大多数 → 应为 identity
- "老上单于" 共现"击/围/破" → 应为 person name

### 6. 低置信度 + evidence（R3）

**做什么**：对 conf < 0.5 的条目构建证据包，人工审阅。

**脚本**：`python kg/entities/scripts/build_evidence_bundles.py`

**关键教训**：**低置信度 ≠ 误标**。大多是证据稀疏（refs 少、三家注无）。抽样 20 条验证："证据稀疏"远多于"真错归"。只有人工审阅后确认的才作为误标处理。

**产率**：地名 R3 从 135 条低置信中仅修 5-10 条真错归。

### 7. 跨工序反馈

**做什么**：收集其他反思工作流发现的类型错误。

**来源**：
- [SKILL_03c 按章反思](../SKILL_03c_按章反思.md)：某章发现 "XX" 实为官职不是人名
- [SKILL_03f 边界错误](../SKILL_03f_实体边界错误综合反思.md)：边界切分错导致类型错
- [SKILL_03b 人名消歧](../SKILL_03b_人名消歧.md)：消歧过程发现类型误判

把这些记录的类型错误汇总到本 skill 的某一轮反思，批量处理。

### 8. 权威文献对照

**做什么**：把分类结果与《汉书·地理志》《汉书·百官公卿表》等对照。

**发现什么**：
- 汉书地理志说是"县"的，我们标为"郡" → 错
- 汉书百官表说是"九卿"的，我们归"列卿" → 错（如"长乐卫尉"）

**补救**：修 `extract_hanshu_*.py`，让自动提取权重高于手工白名单。

## 三、发现后如何纠错（三类误标三种手段）

先做误标分类诊断，再选手段。**同一个"看起来错了"的实体，诊断为哪一类决定了纠错的动作**。

### 诊断 A：源头错（tagged.md 中的类型标记错）

**判断**：该实体**应该是别的类**（不是 official 而是 person）。

**手段**：
- **批量同类** → 用 [SKILL_03e1 跨类型迁移](SKILL_03e1_跨类型迁移.md) 的三阶段法（always/context/new_candidates）
- **零散几条** → `python scripts/fix_source_mislabeled.py` 或 Edit 工具手修
- **复合实体拆分** → `fix_source_mislabeled.py` 脚本（如"新丰鸿门"→"新丰"+"鸿门"）

**必跑的 lint**：
```
python scripts/lint_text_integrity.py   # 原文字符未变
python scripts/lint_markdown.py          # 标注格式无破坏
```

**严格遵守** [CLAUDE.md](../../CLAUDE.md) 的源头编辑规则：
- 严禁引入半角引号 `""`（原文用 `""`）
- 严禁嵌套标注
- 严禁全角/半角符号转换

### 诊断 B：分类器错（源头对，但分类规则未覆盖或误触发）

**判断**：实体类型标对了（是 official），但归错子类（本是九卿被归三公）。

**手段**（按优先级）：

| 问题 | 纠错动作 | 例 |
|-----|--------|-----|
| 缺白名单条目 | 加 `EXPLICIT_正确类` | `长乐卫尉` 加入 EXPLICIT_JIUQING |
| 错误白名单条目 | 从 `EXPLICIT_错误类` 移除 | `列侯` 从 EXPLICIT_JUE 移到 identity_wordlist |
| Guard 未触发 | 改 merge 优先级或 Guard 条件 | 汉诸侯国官 `_is_vassal_official()` |
| L3 过度匹配 | 改 L3 模式或加 guard | "将兵"+丞相 → 加 L3 guard 不覆盖具体类 |
| L2 中层叠加 | 改 L2 为兜底（只在 final 空时触发）| 官职 R4 清 14 条虚假多标签 |
| 权重倒置 | 调 WEIGHT 表 | L2.5_HANSHU_AUTO(0.90) vs _MANUAL(0.85)|
| 规则覆盖漏 | 新增 L3 共现模式 | 新增"为 X 官" 模式覆盖 40 条 |

**验证**：重跑 `classify_TYPE.py`，统计新分布 vs 旧分布，看是否修正了目标 cohort 且未引入新错。

### 诊断 C：边界模糊（源头标注体系本身模糊，分不清）

**判断**：这个词**两种都对/都不完全对**。常见于源头 `〖;X〗` 标注无法区分：
- 官职 vs 人名（`老上单于`）
- 官职 vs 身份（`太后`/`太子`/`布衣`）
- 官职 vs 谥号（`哀`/`武`裸出现时）

**手段**：新增"误标子类"作为 bucket，让这类条目**先聚集**，再跨工序解决。

**官职实例**（R2/R3 新增的 5 个误标子类）：
- `cat-person-mis`：官号+人名代人（如 `老上单于`）
- `cat-identity-mis`：本应 identity（`太后`/`太子`）
- `cat-shihao-mis`：裸谥号（`哀`/`武`）
- `cat-need-split`：官职+人名复合体（`丞相何`/`卫将军青`）
- `cat-mis`：其他无法归类的误标

**后续跨工序协商**：
- 与 [SKILL_03a 实体标注](../SKILL_03a_实体标注.md) 讨论源头标注体系改进
- 把 `cat-need-split` 用 `fix_source_mislabeled.py` 拆分到 person + official
- 把 `cat-identity-mis` 改源头为 `〖#〗` identity 标签

### 诊断的判定树

```
实体被发现疑似误标
    ↓
该实体应属哪个类型？
├─ 明确属别的类型 → 诊断 A（源头错）→ 跨类型迁移
├─ 类型对但子类错 → 诊断 B（分类器错）→ 改规则
└─ 两种都对/都不完全对 → 诊断 C（边界模糊）→ 新增误标子类
```

## 四、典型案例：官职中"太后"的纠错路径

**发现**（R2 未分类驱动 + 共现统计）：
- "太后"出现 230 次，分类器归入 `cat-regent` 或 `cat-mis`
- 共现词：生/废/即位/母/皇 —— 不像官职（官职共现常是 拜/封/立）

**诊断**（C：边界模糊）：
- "太后"不是官职也不是人名，是 `〖#身份〗`
- 源头把 `〖;太后〗` 标错了

**纠错**（两步）：
1. 分类层面：新增 `cat-identity-mis` 子类，把"太后/太子/皇后"等归入
2. 源头层面：运行 [SKILL_03e1 跨类型迁移](SKILL_03e1_跨类型迁移.md)，把这批词 `〖;X〗` → `〖#X〗`

**验证**：
- `official.html` 中 identity-mis 子类清零
- `identity.html` 中新增"太后"等 230 条
- `scripts/lint_text_integrity.py` 通过

## 五、多轮反思节奏

| 轮 | 主题 | 主要动作 | 发现什么误标 |
|---|------|---------|-----------|
| **R1** | 起点（若需建体系） | 10-20 子类 + L1-L5 初版 | 无 |
| **R2** | 未分类驱动 + 源头修正 | dump_blank + 模式挖掘 + 白名单扩充 + 源头修正 | **最多**（诊断 A+C） |
| **R3** | 证据富化 | build_evidence_bundles | 少量（诊断 B） |
| **R4** | 梯度资源策略 | T4→T3→T2→T1 | 中等（诊断 B） |
| **R5** | UI 可视化 | opacity 映射 | 无（视觉化） |
| **R6** | 系统性概念质保 | 高置信抽样找 cohort | 少量但价值高（诊断 B） |
| **R7** | 白名单审计 | audit_*.py 死词/冲突 | 批量（诊断 A+B）|

**何时结束**：
- 未分类 < 5%，降幅 < 0.5 pt
- 低置信抽样 20 条，都是"证据稀疏"而非"错归"
- 白名单审计冲突 < 5%
- 新增反思投入 > 2 小时但修正 < 10 条

## 六、L1-L5 分层决策框架（纠错时的调整对象）

R2-R7 的纠错大多是**调整 L1-L5 规则**，而不是直接改分类结果。

| 层 | 规则 | 置信度 | 纠错时常做的事 |
|----|------|-------|------------|
| L1 | 显式白名单（EXPLICIT_XXX） | 0.80-1.00 | 加/移除条目（最常见纠错动作） |
| L2 | 章节上下文 | 0.85 | 只做兜底（Guard 4） |
| L2.5 | 权威文献自动提取 | 0.85-0.95 | 修 extract 脚本，调权重 |
| L2.5bis | 别名继承（仅人名） | 0.85 | 补 aliases 映射 |
| L3 | 共现动词 / 身份后缀 | 0.40-0.65 | 加新模式 / 加 guard 防误触发 |
| L4 | 词形后缀启发式 | 0.55-0.80 | 调后缀集合 |
| L5 | 并列 / 亲属传播 | 0.50-0.75 | 严格化（仅顿号不含逗号）|

**merge 优先级**：**L1 > L2.5 > L3 > L4 > L2**（L2 只作兜底）

**4 个关键 Guard**：
1. 误标子类命中 → 早返回，不叠加
2. 特殊规则（如诸侯国官）→ 强制覆盖
3. L3 不覆盖 L1/L2.5 已给出的**具体**类
4. L2 仅在 final 完全为空时应用

## 八、六条核心教训

1. **源头修正优先**：诊断 A 不能留到后面（R2 必做）
2. **低置信度 ≠ 误标**：R3 大多是证据稀疏；冒然自动改是最常见的 R3 错误
3. **多标签是本质**：10-20% 合理；虚假多标签是 merge 逻辑错（诊断 B 的 L2 层）
4. **L2 章节规则只作兜底**：中层应用会生成大量虚假多标签
5. **系统性概念混淆 > 单条值错**：R6 一次抽样常胜 R4 逐条深研
6. **细分子类比强行合并更诚实**：边界模糊且系列化（≥ 20 条）时新增 cat-*-mis
