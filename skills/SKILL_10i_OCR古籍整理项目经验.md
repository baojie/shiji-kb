# SKILL 10i: OCR 古籍整理项目的经验沉淀

> **定位**：SKILL_01i 是方法论（应如何做），本 Skill 是 postmortem（做下来学到了什么）。面向下一次启动类似项目的自己或协作者。

## 一、案例概况

**对象**：清·汪越撰、徐克范补《读史记十表》（《钦定四库全书》影印本 OCR）

**规模**：

**耗时**：一次对话内完成从方法论制定到全书处理（十卷 + 提要 + 总论）

## 二、关键经验（非预先可知的部分）

### 2.1 "防"字是 OCR 灾难的放大镜

四库全书影印本 OCR 中，**单一"防"字至少对应 13 种以上原字**：

| 原字 | 例 | 占比（粗估） |
|-----|-----|-----------|
| 榷 / 议 | "商防"（商议） | 高频 |
| 幾 | "刘知防"（刘知幾） | 高频 |
| 微 | "钩稽防义" / "其防微矣" | 高频 |
| 会 | "书防盟" / "防诸侯" | 高频 |
| 灭 | "秦防六国"（秦灭六国）、"防郑"、"防鲁" | 高频 |
| 毁 / 堕 | "防封地" | 中 |
| 护 | "回防者" | 中 |
| 桓 | "防子"（在赵/韩世系语境） | 中 |
| 弑 | "三晋防其君"、"项羽防义帝" | 中 |
| 策 / 议 | "偃防"（主父偃议） | 中 |
| 略 / 旨 | "大防"（大略/大旨） | 中 |
| 究 / 就 | "诚防本纪"（诚究） | 低 |
| 亡 / 殂 | "义帝既防之后" | 低 |
| 陷 / 守 | "苛守荥阳防" | 低 |
| 补 / 合 | "炼石防鼇" | 低 |

**教训**：
- 一个字的 OCR 失真可以形成**跨语境多义词**——单靠正则无法订正
- 标注策略不能一次性定最终字，只能**先标可疑，后期多版本互校订正**
- 若启动新书的 OCR 清洗，**先用 1%-5% 抽样跑"单字多义"统计**，再估工作量

### 2.2 LLM 句读的三类系统偏差

全书处理过程观察到：

1. **夹注吞并**：LLM 倾向于把 `【小字注】` 合并入正文句子，需显式提示"夹注独立成段，不共享外部标点"。
2. **引号层级错配**：嵌套引用（如"太史公曰：...《索隐》曰：..."）常出现外层双/内层双并存，应要求外"内'外"三层交替。
3. **专名连读**：把"周"、"鲁"、"齐"之类单字邦国误并为双字词（如"周鲁"作地名）。需显式提示古文多单字邦国。

**对策**：模板化的**前置提示（prefix prompt）**比事后校对效率高得多。模板写入 SKILL_01i §4.3。

### 2.3 输出结构的隐形约束

**Markdown 结构**对最终可读性的影响远比预期大：

- **三级标题足够**（书名 / 卷 / 篇题），四级反而难导航
- **夹注保持 `【】`** 比改成 footnote 更贴近原书阅读体验
- **存疑条目**独立成段，以 `**存疑**：` 引导，明显提升可检索性
- **卷首原序**用 `>` blockquote 与"读 XX 表"区分，视觉上分开"原文"与"评注"

**教训**：结构决策一旦开始，不要中途改。第一卷定好的结构后续全部沿用。

### 2.4 单次对话内能做到什么

**做到的**：
- 130 篇级别章节的句读 + 排版（10 万字级）
- 方法论 + 经验沉淀
- OCR 可疑字汇总

**没做 / 需下一轮的**：
- **OCR 可疑字的版本互校订正**（需找到其他版本的《读史记十表》对校，已超单次对话边界）
- **表格正文恢复**（原书年表正文因 OCR 仅剩散条目；需对照影印 PDF 逐表还原）
- **字符完整性脚本验证**（SKILL_01i §3.1 提到的 `verify_ocr_preserve.py` 尚未实现）
- **HTML 渲染**（当前只生成 Markdown，未纳入 docs/ 的发布管线）

**教训**：**分阶段交付**比一次性做完更稳。阶段 1（句读 + 排版 + 可疑字表）可在单对话内完成；阶段 2（多版本订正 + 表格恢复 + 工具化验证 + 发布）应拆成独立任务。

## 三、工作量估算（用于下一次类似项目）

**案例实测**：《读史记十表》约 10 万字，单对话内完成句读 + 排版 + 方法论 + 经验沉淀。

**按规模外推**（以"清代学者注释体古籍"为参考）：

| 规模 | 建议策略 |
|-----|---------|
| <2 万字 | 单对话完成方法论 + 全书处理 |
| 2-10 万字 | 单对话方法论 + 全书处理（本案例） |
| 10-50 万字 | 分卷 / 分章多轮处理，每轮 <3 万字 |
| >50 万字 | 先工具化（gj.cool API / 本地模型批处理脚本），再人工 + LLM 抽样复核 |

**不建议**：>30 万字的书仍用"对话式手动分段"处理——应转向脚本化。

## 六、一句话总结

**OCR 古籍整理的本质是"分层降噪"**——每一层只解决一种不确定性（噪声 / 结构 / 句读 / 错字），不要在一步里同时解决多种，否则错误互相污染无法回溯。
