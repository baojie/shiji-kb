## 二、搜索能力分类

设计新搜索功能前，先确定它属于哪一类——这决定了索引形态、匹配算法和 UI：

### 2.1 字面检索

- **全文 AND 子串**（已实现）：最基础的兜底，实现简单、召回可控。
- **别名扩展**（规划中）：补字面检索的最大洞——换个称呼就找不到。OR 展开变体 × AND 跨 token。依赖实体别名表。

### 2.2 近似检索

- **拼音模糊**：解决"记得音不记得字"。中文独有难点（多音字、方言音）。
- **错别字容错**：编辑距离或 n-gram。中文字符级编辑距离信噪比差，通常要结合拼音。

### 2.3 语义检索

- **反向搜索**（描述→实体）：依赖向量嵌入或 LLM 查询改写。最大痛点是索引维护成本与结果可解释性。
- **同义词扩展**：介于字面与语义之间，需要同义词表。

### 2.4 结构化浏览

- **分面过滤**：非搜索、近检索。依赖实体的结构化属性（时代、籍贯、身份、类别）。
- **图浏览**：依赖关系三元组。探索性而非查找性。

## 三、数据依赖表

每类搜索都要"弹药"，没有对应数据就不要上对应能力：

| 搜索能力 | 必需数据 | 当前位置 |
|---------|---------|----------|
| 全文检索 | 去标注后的段落正文 | `chapter_md/*.tagged.md` + `strip_markup` |
| 别名模糊 | surface → variants | `kg/entities/data/entity_index.json` / `entity_aliases.json` |
| 拼音模糊 | 段落 / 实体的拼音串 | 暂缺（需引入 pypinyin，注意多音字） |
| 反向搜索 | 实体简介 / 向量嵌入 | 暂缺（需生成实体描述 + embedding） |
| 分面过滤 | 实体类型、时代、地域、身份 | `kg/entities/data/person_categories.json` 等（部分就绪） |
| 图浏览 | 关系三元组（TTL） | `kg/entities/data/*.ttl` 已有基础 |

## 四、关键设计原则

以下原则**不可轻易破坏**，破坏前请升级 SPEC：

1. **客户端懒加载，零服务器**。索引一次构建、浏览器请求一次、缓存复用。禁止引入服务端搜索。
2. **索引体积 < 5 MB**。超过就必须分片或换形态（如倒排压缩）。
3. **索引与原文的一致性**：`strip_markup` 必须把标注清理干净，否则搜不到或误搜。每次新增标注符号都要同步 `_ENTITY_PFX`。
4. **AND 跨 token、OR 于变体组内**。这是字面与别名扩展共用的查询代数，不要引入 token 级 OR（会让用户困惑召回率）。
5. **多义别名透明展示**。模糊扩展后必须告诉用户"展开了什么"，否则用户会怀疑结果。
6. **非检索能力（分面、图）不混入搜索框**。保持搜索框契约单一；浏览类功能走独立入口（如 KG 页面）。

## 五、多义/歧义的处理底线

别名索引会遇到的几个已知坑（样例见 SPEC §5.1）：

- `轩辕` = 黄帝 ∪ 星宿
- `赤帝` = 尧 ∪ 神话 ∪ 概念
- `楚` = 国名 ∪ 地名 ∪ 多人姓氏

**底线**：
- 用户 opt-in 才展开（不要默认开启）。
- 展开结果必须透明（UI 上显式展示变体集）。
- 不在搜索阶段做类型过滤——类型是分面浏览的职责。
