# SKILL 06e: 概念分类树构建 — Bottom-Up 归纳与颗粒度调整

> **核心理念**：分类不是先验的，是从数据中归纳的。  
> 先有实例，再有类；先看数量，再定颗粒度。

---

## 一、何时使用

| 场景 | 信号 |
|------|------|
| 新一批实体已标注，需要建立分类体系 | 索引中有大量无分类实体 |
| 某叶子类实例数 > 50，内部有明显子结构 | 拆分信号：子结构可通过朝代/职能/地域切分 |
| 某叶子类实例数 < 10，且与邻类高度相关 | 合并信号：单独列类没有辨别力 |
| 分类树 MD 与 TTL 不一致 | 有人手工编辑了 MD，需要以 TTL 为准重生 |
| 需要补全缺失的实体（如某朝代帝王未入树） | 缺口分析发现高频实体未入树 |

**不适用**：从零设计一个全新实体类型的本体（用 SKILL_06a）。

---

## 二、Bottom-Up 构建流程（5步）

### Step 1. 实例统计与分桶

从 `entity_index.json` 提取目标实体类型，按出现频次分桶：

```bash
python kg/taxonomy/scripts/person_gap_report.py
```

| 频次桶 | 处理策略 |
|--------|---------|
| ≥ 50 | 必须入树；出现次数高说明文本中确实重要 |
| 20–49 | 必须入树 |
| 10–19 | 应入树 |
| 5–9 | 建议入树，上下文验证后加入 |
| 2–4 | 择优：确认非别名后加入 |
| 1 | 留待批量脚本处理；人工成本过高 |

**关键动作**：区分"真实新实例"与"别名"——后者不需要新分类条目，只需在 `entity_aliases.json` 建立映射。

### Step 2. 归纳初始分类

对 Step 1 筛出的高频实体，自底向上归纳分组。归纳信号按优先级：

1. **显式归属**：白名单（rulers.json / shihao_index.json）直接确定类别
2. **章节信号**：主要出现在专题列传（刺客/酷吏/货殖等）→ 直接对应类
3. **名称后缀**：单于/夫人/公子/太子 → 对应特定类
4. **共现模式**：与已知类成员高度共现 → 同类推断
5. **历史常识**：帝王世系表（如夏本纪17王、商本纪序列）→ 补全空白

**多信号原则**：至少两个独立信号才可信。单信号分类历史准确率约 12%（见 SKILL_06a §5 记录）。

### Step 3. 颗粒度调整

这是 06e 的核心步骤。**颗粒度信号**：

#### 拆分信号（leaf class 实例数 > 50）

当某叶子类超过 50 个实例，检查内部是否有可切分维度：

```
信号A：时间维度 — 早/中/晚期（如秦国历代君主：秦孝公前/商鞅变法后/统一后）
信号B：职能维度 — 文臣/武将/外交（如汉臣可拆）
信号C：地域维度 — 关中/山东/荆楚（如战国策士分布）
信号D：历史阶段 — 如"汉初"/"汉中后"已拆分
```

拆分阈值建议：
- 子类实例数 ≥ 10 → 值得单独列类
- 子类实例数 < 10 → 合入父类或"其他"桶

#### 合并信号（leaf class 实例数 < 10）

当叶子类过小，考虑：
- 合并到邻近类（功能相似、时代相近）
- 提升为父类的实例（不单独设子类）
- 保留但标注"稀疏类"（<10，可能扩充）

#### 实操判断表

| 现状 | 建议操作 | TTL 修改 |
|------|---------|---------|
| 某类 > 100 且内部异质性高 | 必须拆分 | 新建子类，移动实例 |
| 某类 50–100 且有清晰切分维度 | 建议拆分 | 同上 |
| 某类 10–50 | 保持，继续观察 | 只新增实例 |
| 某类 < 10，与邻类功能重叠 | 合并 | 改 rdf:type，删子类 |
| 某类 < 5，历史上确实独特 | 保留为稀疏类 | 加注释说明 |

### Step 4. TTL 追加与校验

**追加格式**：

```turtle
per:少康 a per:夏 ; rdfs:label "少康"@zh ; :count 8 .
per:帝槐 a per:夏 ; rdfs:label "帝槐"@zh ; :count 3 .
```

**校验三步**：

```python
# 4.1 语法
from rdflib import Graph
g = Graph(); g.parse('kg/taxonomy/person.ttl', format='turtle')
print(f'{len(g)} triples')  # 不报错即通过

# 4.2 冗余检查（重跑 gap report）
python kg/taxonomy/scripts/person_gap_report.py
# TTL 冗余 < 100 为合格

# 4.3 没有实例游离在父类（本级）中未入子类
```

### Step 5. 重生分类树 MD

```bash
python kg/taxonomy/scripts/build_taxonomy.py kg/taxonomy/person.ttl
# → kg/taxonomy/person_taxonomy.md
```

对应 HTML（如有）同步重跑：
```bash
python kg/taxonomy/scripts/rebuild_person_tree_html.py
```

## 三、颗粒度设计原则

### 垂直粒度（层次深度）

| 层级 | 示例（人物树） | 原则 |
|------|-------------|------|
| L0 | 人物 | 顶层锚点，不挂实例 |
| L1 | 王室 / 臣 / 诸子百家 | 大类，<100 类覆盖全域 |
| L2 | 帝王 / 诸侯 / 后妃 / 宗室 | 中类，每类 20–200 实例 |
| L3 | 夏 / 商 / 周 / 汉 | 叶类，每类 10–80 实例 |
| L4 | （仅当 L3 > 80 才拆） | 如"汉初"/"汉中后" |

**原则**：树深不超过 4 层；实例只挂在叶节点。

### 水平粒度（同层类数）

- 同一父类下子类数量建议 3–12 个
- < 3 个说明粒度太粗（考虑下移）
- > 15 个说明粒度太细（考虑合并或引入中间层）

### 正确案例（汉代帝王拆分）

```
汉（帝王）→ 汉（5人，高祖/惠帝/文帝/景帝/武帝等）
```
汉代帝王本可单独列出每一任，但 史记 时段只到武帝，所以合并为一类（约 10 人）是合适颗粒度。

### 反例（过细导致空类）

```
# 错误：为 1-2 人建类，无扩充空间
per:帝皋 a owl:Class ; rdfs:subClassOf per:夏 .  # ❌ 太细
```

正确做法：1-3 人的小类直接挂在父类下作为实例，不单独建子类。

## 四、缺口补全工作流

当发现某朝代/某类别大量缺失（如夏朝帝王）：

1. **查历史序列**：从史书（如夏本纪）获取完整列表
2. **比对索引**：`entity_index.json` 中查找各名称的出现频次
3. **频次筛选**：≥ 3 次出现的必须入树
4. **别名去重**：同一人的多种称谓只建一个 TTL 实例
5. **TTL 追加**：按频次排序插入
6. **重生 MD**：运行 `build_taxonomy.py`

**夏朝补全示例**（已实施）：

| 帝王 | 在史记中的名称 | 索引频次 | 状态 |
|------|-------------|---------|------|

| 帝相 | 后帝相 | 5 | 已修正归类 |
| 帝槐 | 帝槐 | 3 | 已补入 |
| 帝芒 | 帝芒 | 3 | 已补入 |
| 帝泄 | 帝泄 | 3 | 已补入 |

## 五、验证清单

每次分类树扩充后必须通过：

- [ ] `entity_index.json` 高频桶（≥ 10 次）缺口 ≤ 5%
- [ ] `person_gap_report.md` 重生，缺口数量下降
- [ ] TTL 用 `rdflib` 解析无错，三元组数量增加符合预期
- [ ] `person_taxonomy.md` 由 `build_taxonomy.py` 重生（不手编）
- [ ] TTL 冗余（索引中不存在的实例）< 100
- [ ] 新增实例的 rdfs:label 在 `build_person_taxonomy.py` 中能正确渲染
- [ ] `git diff kg/taxonomy/person.ttl` 只含新增三元组
