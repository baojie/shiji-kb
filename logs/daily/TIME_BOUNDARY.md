# 工作日时间划分说明

## 📅 基本规则

本日志系统采用 **早上7:00** 作为工作日分界线。

## 🕐 时间范围

每个日志文件对应的时间范围：

```
YYYY-MM-DD.md 包含的提交时间：
┌─────────────────────────────────────────────┐
│  (前一天) 07:00:00  至  (当天) 07:00:00     │
└─────────────────────────────────────────────┘
```

### 具体示例

```
2026-03-17.md
├─ 开始: 2026-03-16 07:00:00
└─ 结束: 2026-03-17 07:00:00 (不含)

包含的提交：
✓ 2026-03-16 08:44  (白天工作)
✓ 2026-03-16 16:59  (下午工作)
✓ 2026-03-16 21:24  (晚上加班)
✓ 2026-03-17 00:37  (深夜加班)
✓ 2026-03-17 03:41  (凌晨加班)
✓ 2026-03-17 06:59  (7点前)
✗ 2026-03-17 07:00  (属于次日)
✗ 2026-03-17 18:48  (属于次日)
```

## 🎯 设计理念

### 为什么选择7:00而不是0:00？

1. **符合人体作息**
   - 凌晨0-7点通常仍在工作或休息
   - 将这段时间计入"前一天"更自然

2. **避免工作切分**
   - 深夜加班不会被切到两天
   - 连续工作的完整性得以保持

3. **方便次日统计**
   - 早上7点后即可统计完整的前一天工作
   - 适合在上午发送微信群通知

## 📊 时间表

| 时间段 | 归属 | 场景 |
|--------|------|------|
| 前日 07:00-12:00 | 当日 | 上午工作 |
| 前日 12:00-18:00 | 当日 | 下午工作 |
| 前日 18:00-24:00 | 当日 | 晚上加班 |
| 当日 00:00-07:00 | 当日 | 深夜/凌晨加班 |
| 当日 07:00-次日 07:00 | 次日 | 下一工作日 |

## 💡 使用建议

1. **推荐工作流**
   ```bash
   # 次日早上7点后，生成前一天的日志
   python scripts/generate_daily_log.py
   ```

2. **注意事项**
   - 在当天7点前提交的代码会计入当天日志
   - 在当天7点后提交的代码会计入次日日志
   - 深夜加班无需担心日志被分割

3. **查看时间范围**
   ```bash
   # 查看某日日志的实际提交时间
   git log --since="2026-03-16 07:00" --until="2026-03-17 07:00" --oneline
   ```

```
归属日志: 2026-03-17.md ✓ (全部计入同一天)
```

### 案例3：跨7点提交
```
提交1: 2026-03-17 06:50  → 2026-03-17.md
提交2: 2026-03-17 07:10  → 2026-03-18.md
说明: 两次提交虽然只差20分钟，但跨越了7点分界线
```

## 🔄 与传统0点划分的对比

| 特性 | 0点划分 | 7点划分 ✓ |
|------|---------|----------|
| 深夜加班 | 分割到两天 | 归入同一天 |
| 凌晨工作 | 单独计入新一天 | 延续前一天 |
| 统计时机 | 需等到次日白天 | 当日早上即可 |
| 作息习惯 | 违反直觉 | 符合直觉 |
| 工作完整性 | 可能破坏 | 保持完整 |

## 📞 常见问题

**Q: 为什么不是6点或8点？**
A: 7点是一般的起床/上班时间，作为分界线最合适。

**Q: 如果我在早上6点提交代码？**
A: 会计入当天的日志，这通常是深夜工作的延续。

**Q: 能改成0点划分吗？**
A: 可以修改 `generate_daily_log.py` 中的时间计算逻辑，但不推荐。
