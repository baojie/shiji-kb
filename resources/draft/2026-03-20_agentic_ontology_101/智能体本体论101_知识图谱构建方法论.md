# 智能体本体论101：AI驱动的知识图谱构建方法论指南

**鲍捷，与Claude (Anthropic)合作**

*基于《史记》知识库项目：130章、57.7万字、18类实体、102,851条标注、3,197个事件、7,637条关系——一人+AI智能体团队6周构建。文本完整性：91.5%章节通过验证，800+处错误收敛。*

> 本文档更新Noy & McGuinness的经典文献《本体开发101：创建第一个本体的指南》(斯坦福，2001)，使其适应AI智能体时代。传统"本体论101"教人类使用Protégé自顶向下设计本体；本指南教人类如何与AI协作，通过迭代反思和混合自动化，自底向上**生长**本体。

## 第一部分：范式转移

### 1. 为什么需要智能体本体论？

Noy & McGuinness (2001)列举了开发本体的五个理由：

1. 共享对信息结构的共同理解
2. 实现领域知识的重用
3. 使领域假设显式化
4. 分离领域知识与操作知识
5. 分析领域知识

这五条至今有效。但**如何**构建本体已根本改变：

| | 本体论101 (2001) | 智能体本体论 (2026) |
|--|---------------------|---------------------------|
| **起点** | Protégé中的空白模式 | 原始文本语料 |
| **方向** | 自顶向下设计 | 自底向上涌现 |
| **规模** | 数十类、数百实例 | 数千类、数万实例 |
| **迭代速度** | 每轮数周 | 每轮数小时 |
| **核心瓶颈** | **2001: 设计** | **2026: 反思/演化** |

**核心转变：** 2001年，难点在于**设计**本体。2026年，难点在于**反思和演化** AI提取的本体。本体不再是预先设计的蓝图——它是从数据中生长的有机结构，由人类修剪和校准。

### 2. 关于本项目

我们通过《史记》知识库展示这些原则——从司马迁《史记》(约公元前94年)中提取的完整知识图谱，覆盖中国2,600年历史：

**数据规模：**
- **文本：** 130章，57.7万字古汉语
- **实体：** 15,190个唯一实体，102,851条标注，18类
- **事件：** 3,197个历史事件，98.7%有公元前纪年
- **关系：** 7,637条事件关系（4类自动计算 + 5类LLM推断）
- **质量：** 5轮反思，12,200+处修正收敛

**方法论：**
- **14个元技能：** 通用知识工程方法（OTF+JIT+Bootstrap、反思循环、柳叶刀法等）
- **40+管道技能：** 古籍处理的领域特定技术
- **6个项目管理技能：** 文件组织、Git工作流、CHANGELOG、日志、TODO/Issue跟踪
- **时间投入：** ~150小时人工 + ~30亿tokens AI
- **人机分工：** 人类设定愿景，AI执行批量工作，人类把控质量
- **质量保证：** 130章系统验证后，91.5%文本完整性

这不是玩具示例。这是生产级知识库，具有交互式阅读器、地铁图时间线、跨章推理功能——所有方法均记录为可复用的SKILL文件。

## 第二部分：三大方法论支柱

区分智能体本体工程与传统方法的三大核心创新：

### 3. OTF + JIT + Bootstrap：元方法

#### 3.1 On-The-Fly (OTF)：边干边总结

**传统方法（事后总结）：**
```
第1-90天：执行任务（不总结）
第91天：开始总结
  ├─ 问题1：细节遗忘，总结质量低
  ├─ 问题2：错误模式重复90次，返工成本高
  └─ 问题3：无法应用于项目本身（已结束）
```

**OTF方法：**
```
第1天：处理5个样本
  ├─ 发现模式A（消歧问题）
  └─ 立即总结 → 更新规则库

第2天：处理30个样本
  ├─ 应用第1天规则（80%自动化）
  ├─ 发现新模式B（格式问题）
  └─ 立即总结 → 开发检查工具（1小时）

第3-10天：处理全部130章
  ├─ 应用累积的规则+工具
  ├─ 错误率从20% → 5%
  └─ 发现最终模式C、D → 立即总结
```

**核心原则：** 不要等到结束再总结。当你看到模式重复3次，立即提取并编码为规则/工具/SKILL文档。

*史记案例：* 实体标注期间，仅处理10章就发现"单字国名"错误模式（赵/韩/魏被误标为人名）。立即创建检测脚本。该模式在全部章节中出现1,200+次——早期捕获节省了数周手动修正。

#### 3.2 Just-In-Time (JIT)：交付最小可行版本

**传统方法（大爆炸）：**
```
第1-4周：设计完美模式（50属性本体）
第5-8周：执行一次（发现模式不匹配）
第9周：大规模返工（重构本体，重新录入数据）
→ 总计：9周，质量不确定
```

**JIT方法：**
```
第1周：
  JIT-v0.1：最小本体（4类）+ 标注5章
  ├─ 交付：初始数据（60%质量）
  └─ 反馈：需要"诸侯国"类型

第2周：
  JIT-v0.5：本体演化（4→8类）+ 标注30章
  ├─ 交付：扩展数据（75%质量）
  └─ 反馈：需要"器物"类型

第3周：
  JIT-v1.0：本体稳定（18类）+ 标注130章
  ├─ 交付：完整数据（85%质量）
  └─ 反馈：系统性错误模式

第4周：
  JIT-v1.5：第一轮反思
  ├─ 交付：高质量数据（92%质量）
  └─ 反馈：收敛

→ 总计：4周，质量92%
```

**核心原则：** 不要在第一次迭代就追求完美。快速交付最小可行版本，收集反馈，迭代改进。质量通过连续精炼提升，而非前期完美主义。

*史记案例：* 事件纪年标注经历5个质量阶梯：
- P0（第1-2天）：3,092事件初始公元前纪年（60%准确率）
- P1（第1周）：首轮反思，1,010处修正（80%准确率）
- P2（第2周）：二轮反思，431处修正（88%准确率）
- P3（第3周）：三轮反思，465处修正（92%准确率）
- P4（第4周）：四轮反思，167处修正（95%准确率）
- P5（第5周）：五轮反思，46处修正（97%准确率，收敛）

每个阶梯都交付可用数据。早期阶梯使下游工作（关系提取）能在纪年完美前开始。

#### 3.3 Bootstrap：知识自生长（5阶段）

**传统视角：**
```
人类劳动 → 完成任务 → 项目结束
  ↓
下个项目：从零开始
```

**Bootstrap视角：**
```
人类劳动 → 产生中间知识（日志/脚本）
  ↓
中间知识 → 总结为SKILL文档
  ↓
SKILL文档 → 自动化工具（减少人类努力）
  ↓
工具使用 → 生成高阶知识（模式）
  ↓
模式 → 自动更新SKILL文档（总结本身自动化）
  ↓
SKILL文档 → 自演化（系统自治理）
```

**五个Bootstrap阶段：**

| 阶段 | 自动化率 | 关键产出 | 人类角色 |
|-------|----------------|--------------|------------|
| **阶段1：手工劳动** | 0% | 标注样本，对话日志 | 执行者 |
| **阶段2：脚本** | 20% | 检查/统计/转换脚本 | 执行者+脚本工 |
| **阶段3：SKILL文档** | 60% | 方法文档，Agent工作流 | 审核者 |
| **阶段4：自动总结** | 90% | Agent自动更新SKILL | 质量控制者 |
| **阶段5：元演化** | 99% | 元SKILL（生成方法的方法） | 异常处理者 |

*史记案例：* 实体标注Bootstrap轨迹：
- 第1周：手动标注（100%人工）
- 第2周：编写10个验证脚本（20%自动化）
- 第3周：记录SKILL_03，Agent读取并自动执行（60%自动化）
- 第5-8周：Agent自动发现新错误模式，更新SKILL（90%自动化）
- 第3月：提取元SKILL（00-META-01_反思），适用于所有标注任务（98%自动化）

**收敛证明：** 人类时间指数下降：
```
第1周：  40小时（100%人工）
第2周：  32小时（80%人工）
第3-4周：24小时（40%人工）
第5-8周：10小时（10%人工，跨4周）
第3-4月：5小时（1%人工，跨2月）
→ 总人类时间：111小时 vs. 估计400小时（传统方法）
```

### 4. 反思循环：质量收敛

将"AI输出（90%准确）"转化为"生产级数据（97%+准确）"的结构化迭代过程。

#### 4.1 反思的三个维度

| 维度 | 用例 | 示例 | 优势 | 局限 |
|-----------|----------|---------|-----------|------------|
| **按章** | 章节特定上下文错误 | SKILL_03c按章实体审核 | 捕获局部上下文 | 遗漏跨章模式 |
| **按类型** | 全章节系统性错误 | 将"刑法"术语从institution→law类型迁移 | 一次修复应用全库 | 需先发现模式 |
| **全局** | 跨章一致性 | 第五轮事件纪年：跨章验证同一事件 | 发现矛盾 | 计算成本高 |

**选择原则：**
- 第1轮：**按章**建立基线，累积错误模式
- 第2-3轮：**按类型**系统修正（批处理）
- 最终轮：**全局**交叉验证和一致性检查

#### 4.2 收敛证明：事件纪年案例

对3,092个事件公元前纪年进行五轮Agent驱动反思：

| 轮次 | 修正数 | 影响章节 | 新模式 | 主要修复类型 |
|-------|-------------|-------------------|--------------|----------------|
| **R1** | 1,010 | 118/130 | 25 | 系统性年份错误 + 确定性升级 |
| **R2** | 431 | 105/130 | 1 | 确定性升级 + 微调 |
| **R3** | 465 | 70/130 | 0 | 收敛验证 + 残余清理 |
| **R4** | 167 | 68/130 | 0 | 最终验证 + 格式统一 |
| **R5** | 46 | 28/130 | 0 | 跨章验证 + 确定性升级 |

**指数衰减：** 修正量每轮减少约57%（R1→R2→R3），然后R4-R5加速下降。

**收敛标准：**
- 修正数 < 总数的5%
- 新模式数 = 0（连续2轮）
- 跨章矛盾数 < 10

**关键洞察：** AI错误不是随机的——它们是**系统性的**。一旦发现模式，可以一次修复数百实例。每轮累积模式，使下一轮更高效。

#### 4.3 四阶段工作流

每个反思任务遵循此结构：

```
┌─────────────────────────────────────────────────┐
│  阶段0：准备                                     │
│  ├─ 定义质量标准 & 错误模式                      │
│  ├─ 编写SKILL文档（方法论 + 已知模式）            │
│  └─ 准备检测工具（lint/grep/stats）              │
├─────────────────────────────────────────────────┤
│  阶段1：侦察                                     │
│  ├─ 抽样10-20章，识别主要错误类型                │
│  ├─ 统计错误分布（按章/类型）                    │
│  └─ 确定策略（按章/按类型/全局）                 │
├─────────────────────────────────────────────────┤
│  阶段2：执行                                     │
│  ├─ Agent批量审核（SKILL引导）                   │
│  ├─ 提取修正建议（JSON）                         │
│  └─ 应用修正（脚本自动化）                       │
├─────────────────────────────────────────────────┤
│  阶段3：验证                                     │
│  ├─ 运行lint检查（格式验证）                     │
│  ├─ 对比前后统计                                 │
│  ├─ 抽查5-10%（人工审核）                        │
│  └─ 决策：收敛 → 退出，或迭代 → 更新SKILL，下一轮│
└─────────────────────────────────────────────────┘
```

**输出：** JSON格式修正及理由，支持审计追踪和回滚。

### 5. 柳叶刀法：混合精度

以手术刀具命名，该方法倡导**细粒度任务分解**和**为每个子任务选择最优解决方案**（LLM/规则/代码）。

> *"手术刀般的精准，而非大锤端到端。"*

#### 5.1 任务分解：决策树

```
任务分析
  ├─ 有显式规则？
  │   ├─ 是 → [规则 + 代码]（如格式验证、类型过滤）
  │   └─ 否 ↓
  ├─ 结构化匹配问题？
  │   ├─ 是 → [代码算法]（如实体共现、时间对齐）
  │   └─ 否 ↓
  ├─ 需要语义理解？
  │   ├─ 是 → [LLM]（如因果推理、情感分析）
  │   └─ 否 ↓
  ├─ 可安全自动化？
  │   ├─ 是 → [规则 + LLM联合]（基于规则的候选过滤 + LLM确认）
  │   └─ 否 → [人工审核]（保守回退）
```

#### 5.2 示例：事件关系提取

**任务：** 提取3,185个事件间的7,637条关系（5.07百万可能对）。

**按关系类型分解：**

| 关系类型 | 解决方案 | 数量 | 理由 |
|---------------|----------|-------|-----------|
| **cross_ref**（互引） | 规则 + 代码：名称相似度 ≥0.6 + 共享人物 | 294 | 确定性匹配 |
| **co_person**（共享人物） | 代码：跨章事件共享 ≥2关键人物 | 867 | 图算法，O(n²) |
| **co_location**（共享地点） | 代码：共享地点 + ≥1人物（噪声过滤） | 542 | 组合约束 |
| **concurrent**（同时） | 代码：同一公元前年份 + 共享 ≥1人物 | 173 | 时间对齐 |
| **sequel**（续接） | LLM：章内时间序列 | 1,623 | 语义理解 |
| **causal**（因果） | LLM：章内因果判断 | 407 | 复杂逻辑推理 |
| **part_of**（包含） | LLM：部分-整体事件层次 | 107 | 语义分析 |
| **opposition**（对立） | LLM：识别对立方行动 | 50 | 语义角色标注 |
| **跨章因果** | LLM二次处理：精炼自动生成候选 | 352 | 混合：自动过滤然后LLM |

**成本-效益分析：**

| 方法 | 精度 | 召回率 | 成本 | 时间 |
|----------|-----------|--------|------|------|
| 纯LLM（所有5.07M对） | 87% | 92% | $41,000 | 11小时 |
| 纯规则（仅确定性） | 95% | 68% | $50 | 5分钟 |
| **混合（4自动 + 5 LLM）** | **94%** | **89%** | **$380** | **1小时** |

**关键洞察：** 不要对所有事情使用LLM。对结构化任务使用代码（便宜100倍），为规则失败的语义任务保留LLM。

#### 5.3 三种协作模式

**模式1：规则 → LLM（过滤然后精炼）**
```
所有事件对（5.07M）
  ↓ [规则过滤]：共享实体/时间/地点
候选（25K）  ← 减少99.5%
  ↓ [LLM推理]：语义关系分类
最终关系（7,637）
```

**模式2：LLM → 规则（生成然后验证）**
```
LLM生成别名候选
  ↓ [启发式规则]
    - 长名必含短名
    - 字符重叠 ≥50%
    - 同章共现 ≥3次
  ↓ [自动确认]
安全别名对（595）
  ↓ [人工审核]
歧义案例（103）
```

**模式3：规则 → 代码 → LLM（分层回退）**
```python
def infer_event_year(event):
    # 层1：基于规则的查表（精确已知事件）
    if event.name in KNOWN_EVENTS:
        return KNOWN_EVENTS[event.name]

    # 层2：代码约束推理（区间缩窄）
    if event.persons:
        valid_range = [max(p.birth for p in event.persons),
                       min(p.death for p in event.persons)]
        if valid_range[0] <= valid_range[1]:
            return interpolate(valid_range)

    # 层3：LLM语义推理（回退）
    return llm_infer_from_context(event.description)
```

**覆盖率：** 规则（40%）+ 代码约束（30%）+ LLM（25%）+ 人工（5%）

## 第三部分：9阶段管道

完整知识工程工作流，从原始文本到交互应用。

### 6. 概览：四个语义层

```
        ┌─────────────────────────────────────────────┐
  L4    │  应用语义                                    │  阶段9
        │  历史研究、矛盾检测、模式挖掘                 │
        ├─────────────────────────────────────────────┤
  L3    │  知识语义                                    │  阶段6-8
        │  本体、推理规则、SKU                         │
        ├─────────────────────────────────────────────┤
  L2    │  图语义                                      │  阶段3-5
        │  实体、事件、关系、原子事实                   │
        ├─────────────────────────────────────────────┤
  L1    │  结构语义                                    │  阶段1-2
        │  章节、段落、句子、注释                      │
        └─────────────────────────────────────────────┘
```

每层建立在前一层之上。没有结构语义就没有图语义。没有知识语义就没有应用语义。

### 7. 阶段1-2：文本校勘与结构分析

**阶段1：校勘（SKILL_01）**
- **输入：** 维基文库、中国哲学书电子化计划等多个文本版本
- **过程：** 异文比对、错误修正、确立权威版本
- **输出：** 清洗的定本（`docs/original_text/`）

**阶段2：结构分析（SKILL_02）**
- **过程：**
  - 章节/小节分割，段落编号（紫色编号）
  - 对话切分，韵文检测
  - 注释对齐（裴骃、司马贞、张守节三层注释）
  - 句级语义关系（因果、并列、递进）
- **输出：** 带`§1`、`§2`...段落ID的结构化文本
- **关键创新：** 紫色编号（受Doug Engelbart启发）用于精确引用

*数据：* 130章，57.7万字，~12,000段落编号

### 8. 阶段3：实体构建（18类）

**类型系统演化：**

| 版本 | 类型数 | 触发因素 | 变更示例 |
|---------|-------|---------|----------------|
| v1.0 | 11 | 初始设计 | 人物/地点/官职/时间/朝代/机构/部族/器物/天文/神话/生物 |
| v2.1 | 13 | 数据揭示"朝代"混合3个概念 | 拆分：朝代 → 朝代 + 诸侯国 + 宗族 |
| v2.3 | 15 | "官职"混合任命与社会角色 | 拆分：官职 → 官职 + 身份 |
| v2.4 | 16 | "动植物"更名为清晰 | 重命名：动植物 → 生物 |
| v2.5 | 18 | 需要标记数量词 | 新增：数量、引文 |

**最终18类：** 人物、地点、官职、时间、朝代、诸侯国、宗族、机构、部族、器物、典籍、思想、身份、数量、生物、天文、神话、事件名

**工作流：**
1. AI批量标注（Claude，temperature=0）→ 102,851条初始标注
2. 按章反思（SKILL_03c）→ R1修复1,913处错误
3. 按类型反思（SKILL_03e）→ 迁移2,235项从官职→身份，400+项从机构→法律
4. 别名解析 → 合并595个别名组（如刘邦 = 沛公 = 汉王 = 高祖）
5. 消歧 → 解决644个歧义短名（如武王 → 周武王在第4章，秦武王在第5章）

**质量保证：**
- 文本完整性验证：移除所有标签必须逐字节恢复原文
- 标签对称性检查：所有`〖@X〗`开闭对必须匹配
- 跨文件引用检查：事件中的实体必须存在于实体索引

*最终数据：* 15,190个唯一实体，102,851条标注，100%文本完整性

### 9. 阶段4：事件提取（3,197事件）

**为什么用LLM提取事件？**
- 古汉语事件跨多段（如"鸿门宴"跨10+段）
- 事件粒度需要语义判断（"六国灭亡" = 1事件还是6事件？）
- 因果关系嵌入叙事结构，无法通过关键词提取

**混合方法：**
- **事件提取：** LLM（需要语义理解）
- **公元前纪年：** 规则（年号表 + 在位年可计算）
- **纪年验证：** LLM Agent（综合合理性判断）

**事件模式：**
```json
{
  "event_id": "001_003",
  "chapter": "001",
  "name": "黄帝战蚩尤",
  "event_type": "战争",
  "date_display": "约前2700年",
  "date_ce": -2700,
  "date_certainty": "估算",
  "persons": ["@黄帝@", "@蚩尤@"],
  "locations": ["=涿鹿="],
  "description": "黄帝与蚩尤战于涿鹿之野..."
}
```

**纪年收敛（5轮）：**
- R1：发现"单锚崩塌"模式（第001章66事件全标-2290，应跨-2700至-2230）
- R2：发现"确定性欠标"（43.6%修正从"~公元前XXX"升级为"公元前XXX"并附编年验证）
- R3：发现"锚点污染"（R2确定性升级引入新错误，先验证锚点质量）
- R4：跨章验证（同一事件在多章提及必须日期一致）
- R5：最终清理，收敛（<5%修正率）

*最终数据：* 3,197事件，98.7%有公元前纪年，跨-2700至-87

### 10. 阶段5：关系发现（7,637关系）

**9类关系，2种发现方法：**

**自动计算（代码，4类）：**
1. `cross_ref`（互引）：名称相似度 + 共享人物
2. `co_person`：跨章事件共享 ≥2人物
3. `co_location`：共享地点 + ≥1人物
4. `concurrent`：同一公元前年份 + ≥1人物

**LLM推断（5类）：**
5. `sequel`：章内时间续接
6. `causal`：因果关系
7. `part_of`：部分-整体层次
8. `opposition`：对立方冲突
9. *跨章因果（5子类）*：自动候选 → LLM精炼

**"地铁图"应用：** 3,197事件为"车站"，7,637关系为"换乘连接"。用户可像地铁系统一样导航历史。

*换乘示例：*
- 事件007_042"项羽自刎"（项羽自杀）在第007章
- `cross_ref` → 事件008_087"项羽之死"（项羽之死）在第008章
- 用户可"换乘"章节间看不同叙事视角

### 11. 阶段6-9：本体、推理、SKU、应用

**阶段6：本体构建（SKILL_06）**
- 从实体词汇构建类层次
- 生成OWL/RDF表示（未来工作）
- 当前：基于JSON的分类体系及继承

**阶段7：逻辑推理（SKILL_07）**
- **推断：** 从事件参与约束推断人物生卒年范围
- **姓氏推理：** 分离先秦人物的姓（surname）vs. 氏（clan name）（7轮迭代）
- **异常检测：** 单事实违反常识/制度/规则
- **矛盾挖掘：** 跨章不一致（如第007章说"数十"人被杀，第008章说同一战役"8万"人被杀——100倍差异）

**阶段8：SKU构建（SKILL_08）**
- **SKU** = 库存单位，借用库存管理术语
- 三类SKU：
  - **事实SKU：** 原子事实（谁何时何地做什么）
  - **程序SKU：** 操作知识（礼仪程序、战术）
  - **关系SKU：** 实体关系（家谱、联盟网络）

**阶段9：应用构建（SKILL_09）**
- **认知辅助阅读器：** 18类语法高亮、悬停注释、可点击实体索引
- **地铁图：** 交互式时间线，130线（章节）× 3,197站（事件）
- **游戏化：** SKU → 技能卡，事件 → 情节线，实体 → 角色（未来工作）

## 第四部分：经验与原则

### 12. 智能体本体论七规则

*本体论101陈述了三条基本规则。我们保留全部三条并新增四条：*

**规则0（元规则）：先提取，后设计**
- 不要预先设计完美本体。让AI产生粗糙版本，然后从错误中发现你的本体。
- *示例：* 从11类开始，基于数据驱动发现演化到18类。

**规则1（来自本体论101）：没有唯一正确方式建模领域**
- 仍然成立。最佳本体取决于你的应用目标。

**规则2（来自本体论101）：本体开发必然是迭代过程**
- 仍然成立，但现在形式化为具有可测量收敛的**反思循环**。

**规则3（来自本体论101）：概念应接近你领域中的对象**
- 仍然成立。我们的18类实体直接映射历史文本对象，非抽象类别。

**规则4（新）：质量通过反思收敛**
- AI输出90%准确率 + 5轮反思 → 97%+准确率
- 每轮发现更少错误（指数衰减）
- 边际收益 < 阈值时停止

**规则5（新）：分类原则从边界案例涌现**
- 不要从理论定义类。调试误分类，提取决策规则。
- *示例：* "齐桓公 = 特定人物（人物类）" vs. "吴王 = 通用角色（官职类）"的区分从边界案例分析涌现。

**规则6（新）：分离源与解释**
- 永不修改原文以添加元数据
- 使用覆盖架构：`source.md`（不可变）+ `aliases.json` + `dates.json` + `relations.json`（可变）

**规则7（铁律）：标注不得改变源文本**
- 验证：`strip_all_tags(annotated_text) == original_text`（逐字节）
- 任何字符增删改 = 失败

### 13. 要避免的反模式

**❌ 反模式1：过度依赖LLM**
- *症状：* 对所有任务使用LLM，包括简单的基于规则任务
- *后果：* 成本爆炸（$41,000 vs. $380），速度慢，结果不稳定
- *示例：* 使用GPT-4检查"事件A在事件B之后"而两者都有公元前纪年
  - 错误：LLM推理（$0.01成本，1秒，可能错误）
  - 正确：`year_A > year_B`（代码，0.001秒，确定性）

**❌ 反模式2：规则过度工程**
- *症状：* 为每个边界案例编写规则，规则库膨胀到数千条
- *后果：* 维护地狱，泛化差，迭代成本高
- *示例：* 为644个歧义短名编写644条单独规则
  - 错误：644条硬编码规则
  - 正确：分层策略（规则查表40% + 启发式45% + LLM 10% + 人工5%）

**❌ 反模式3：黑箱管道**
- *症状：* 每个模块独立开发，中间结果不可见
- *后果：* 难以调试，错误传播，失败无法归因
- *解决方案：* 每阶段输出中间文件（JSON/Markdown），支持断点续传

**❌ 反模式4：盲目端到端**
- *症状：* 尝试用单一大模型解决一切
- *后果：* 大量数据需求（10K+标注），高训练成本，可解释性差
- *我们的选择：* 模块化管道 + 混合自动化，非端到端深度学习

**❌ 反模式5：过早模式冻结**
- *症状：* 看到任何数据前就设计30类实体及每类50属性
- *后果：* 模式-数据不匹配，大规模返工
- *正确方法：* 最小可行本体（4-5类）→ 数据驱动演化 → 稳定在15-20类

### 14. 质量保证：五层检查

| 层 | 检查类型 | 工具 | 通过率要求 |
|-------|-----------|------|----------------------|
| **L1：格式** | 标签对称性、文本完整性 | `lint_markdown.py`、`lint_text_integrity.py` | 100% |
| **L2：类型** | 实体类型有效性 | `validate_entity_types.py` | 100% |
| **L3：交叉引用** | 事件中的实体引用必须存在于实体索引 | `validate_cross_refs.py` | 100% |
| **L4：约束** | 日期约束（事件年份 ∈ [人物生年, 人物卒年]） | `validate_constraints.py` | >95% |
| **L5：一致性** | 跨章同一事件日期一致性 | Agent全局反思 | >98% |

**Lint优先原则：** 每轮反思前运行L1-L3检查。先修格式错误后修语义错误（防止语义审核中的噪声）。

**文本完整性验证：** 使用`lint_text_integrity.py`系统验证后，130章中91.5%达到完美完整性（标注文本去标签后与原文完全一致）。剩余8.5%章节正在修正。验证强制执行铁律：`strip_all_tags(annotated_text) == original_text`（逐字节）。

## 第五部分：从一本书到一座图书馆

### 15. SKILL作为DNA：可复用方法论

**创新：** 方法不埋在代码注释或非正式笔记中。它们**显式记录为SKILL文件**——AI智能体可读取并执行的结构化Markdown文档。

**双层结构：**

**14个元技能（领域无关）：**
- `00-META-00`：OTF + JIT + Bootstrap（元元技能）
- `00-META-01`：反思
- `00-META-02`：迭代工作流
- `00-META-03`：冷启动
- `00-META-04`：柳叶刀法
- `00-META-05`：知识作为上下文压缩
- `00-META-06`：SKILL优化与演化
- `00-META-07`：可读性（数据格式设计）
- `00-META-08`：标注模式设计
- `00-META-09`：Agent提示工程
- `00-META-10`：质量控制
- `00-META-11`：数据直觉培养
- `00-META-12`：数据融合
- `00-META-13`：技能迁移

**6个项目管理技能（基础设施）：**
- `SKILL_10`：项目管理（概览）
- `SKILL_10a`：TODO与Issue管理
- `SKILL_10b`：每日工作日志维护
- `SKILL_10c`：Git版本控制标准
- `SKILL_10d`：CHANGELOG编写标准
- `SKILL_10e`：文件组织与目录结构

**40+管道技能（领域特定）：**
- `SKILL_01`：文本校勘
- `SKILL_02`：结构分析（a-h：分割、注释、统计等）
- `SKILL_03`：实体构建（a-e：标注、消歧、反思等）
- `SKILL_04`：事件构建（a-f：提取、纪年、验证等）
- `SKILL_05`：关系构建（a-e：发现、战争事件等）
- `SKILL_06`：本体构建
- `SKILL_07`：逻辑推理（a-c：生卒年推断、姓氏推理、异常检测）
- `SKILL_08`：SKU构建
- `SKILL_09`：应用构建（a-b：阅读器、游戏化）

**AI如何读取SKILL：**
- Agent提示包含：`阅读SKILL_XX.md理解任务方法论`
- SKILL文档包含：错误模式（表格格式）、质量标准、验证检查清单、JSON输出模式
- Agent输出：修正 + 新发现模式（下一轮自动追加到SKILL）

**Bootstrap循环：**
```
人类执行任务 → 总结为SKILL → Agent读取SKILL并执行
  ↓
Agent发现新模式 → 自动更新SKILL → Agent读取更新的SKILL
  ↓
多个SKILL共享模式 → 提取元SKILL → Agent读取元SKILL
  ↓
元SKILL应用到新领域 → 领域SKILL自动生成 → 系统自演化
```

*史记证明：* 60+个SKILL文件（14元 + 6项目管理 + 40+管道）是项目最有价值的输出——比15,190个实体或3,197个事件更可复用。这些SKILL可指导下一个项目（《汉书》、《资治通鉴》），方法复用80-90%。项目管理SKILL（10系列）编码Git工作流、每日日志、文件组织的最佳实践，适用于任何知识工程项目。

### 16. 扩展：从60万字到40亿字

**成本模型（优化后，学习曲线后）：**

| 目标语料 | 规模 | 估计成本 | 关键适应 |
|---------------|-------|----------------|----------------|
| **史记**（试点） | 57.7万字 | $1,000级（含试错） | 方法探索，创建60+ SKILL |
| **汉书**（下一个） | 74万字 | $200-400（80%方法复用） | 实体类型：+2（年号），提示微调 |
| **二十六史** | 4000万字 | $50K-100K优化 | 管道并行化，类型系统扩展 |
| **资治通鉴系列** | 600-700万字 | $10K-20K优化 | 编年体vs.纪传体结构，交叉引用至史书 |
| **四库全书**（长期） | 30亿字 | $1M-2M（如工业化） | 多体裁适应，众包QA |

**关键洞察：** 随SKILL稳定和自动化改进，每10万字成本指数下降：
- 史记（首项目）：~$1.7/10万字（探索成本）
- 汉书（二项目，方法复用）：~$0.3/10万字
- 稳态（工业化）：~$0.05/10万字

**不仅规模，还有深度：** 目标不是单纯数字化，而是**结构化知识提取**，支持：
- 跨语料矛盾检测（同一事件在多史书中，比较记载）
- 模式挖掘（哪些制度因素与王朝稳定相关？）
- 谱系重构（从分散文本提及自动生成家谱）

**愿景：** AI智能体维护、持续演化的知识网络，跨越中国古典文本3000年。开源方法（SKILL文件），开放数据（CC BY-NC-SA 4.0），使任何人能通过问答、可视化、推理探索历史智慧。
