# SKILL: 古籍事件提取与关系发现 — 从叙事文本到事件知识图谱

> 从《史记》130篇中提取3,185个事件和7,314条关系的实践中提炼，适用于纪传体/编年体古籍的事件结构化与关系发现。

---

## 一、核心概念

### 1.1 事件知识图谱的三个层次

```
层次1: 事件提取
  叙事文本 → 结构化事件（ID/名称/类型/时间/地点/人物/描述/原文）

层次2: 关系发现
  事件集合 → 关系网络（延续/因果/包含/对立/互见/共人/共地/同期）

层次3: 时空定位
  叙事年份（"元光六年"）→ 公元年（前129年）→ 时间线可视化
```

### 1.2 方法论选择

**为什么用LLM而不是规则提取？**

古籍事件不同于新闻事件：
- 一个"事件"可能跨越数段叙述（"鸿门宴"在原文中跨十余段）
- 事件粒度需要语义判断（"秦灭六国"是一个事件还是六个？）
- 因果关系深藏在叙事逻辑中，非字面可提取

**为什么用混合方法（自动+LLM）做关系？**

| 关系类型 | 方法 | 原因 |
|---------|------|------|
| 跨章互见 | 自动 | 字面相似度+共享人物可计算 |
| 共人/共地/同期 | 自动 | 实体集合交集可计算 |
| 延续/因果 | LLM | 需理解叙事因果逻辑 |
| 包含/对立 | LLM | 需理解事件层级和立场 |

自动方法产出5,126条关系（确定性强、成本低），LLM产出2,188条（语义理解、成本高）。

## 二、事件提取

### 2.1 事件定义

**一个事件 = 一个完整行动单元**：起因 → 执行 → 结果。

**粒度规则**：
- "X死，Y继位" = 1个事件（除非二者都有详细展开）
- 同一事件出现在多处，只在主要记述处提取一次
- "太史公曰"评论不是事件（除非包含新的史实）
- 过于琐碎的行动不单独成事件（"遣使者"若无后续不提取）

### 2.2 事件Schema

```markdown
### {章节号}-{序号} {事件名称}
- **事件类型**: 战争/继位/政治活动/改革/家族事件/建设/文化/经济/灾害/...
- **时间**: %纪年%（公元前NNN年）
- **地点**: =地名=
- **主要人物**: @人名@、@人名@
- **涉及朝代**: &朝代&
- **事件描述**: 2-4句概括
- **原文引用**: 保留实体标记的原文
- **段落位置**: [起始段]-[结束段]
```

**关键**：事件描述和原文引用中保留所有实体标记（`@`、`=`、`$`、`%`、`&`），便于后续提取人物、地点等关联。

### 2.3 11种事件类型

| 类型 | 说明 | 典型事件 |
|------|------|---------|
| 战争 | 军事冲突、攻伐 | 长平之战、垓下之战 |
| 继位 | 即位、禅让、篡位 | 秦始皇即位 |
| 政治活动 | 外交、会盟、朝见 | 鸿门宴会 |
| 政治改革 | 制度变革 | 商鞅变法 |
| 政治整顿 | 清洗、诛杀 | 项羽斩宋义 |
| 家族事件 | 婚姻、继嗣、家世 | 项氏家世渊源 |
| 建设 | 工程、迁都 | 修筑长城 |
| 文化活动 | 著述、教育、礼仪 | 孔子授徒 |
| 经济活动 | 商业、赋税、屯田 | 盐铁专营 |
| 自然灾害 | 水旱、地震、蝗灾 | 河决瓠子 |
| 改革 | 法律制度改革 | 约法三章 |

CHAPTER_TYPE_PROMPTS = {

**事件密度差异极大**：
- 本纪：平均63个事件/篇（叙事密集）
- 列传：平均20个事件/篇（聚焦人物）
- 表：平均5个事件/篇（表格为主）
- 书：平均3个事件/篇（论述为主）

### 2.5 LLM提取流程

```python
def extract_events(chapter_text, chapter_type):
    prompt = f"""从以下《史记》章节中提取所有历史事件。

## 输出要求
1. 概览表（ID/名称/类型/时间/地点/人物/朝代）
2. 每个事件的详细记录（7个字段）
3. 事件链分析（2-5条因果链）

## 事件粒度
- 一个完整行动单元 = 一个事件
- 保留原文中的实体标记（@人名@ =地名= 等）

## 体裁特点
{CHAPTER_TYPE_PROMPTS[chapter_type]}

## 章节原文
{chapter_text}
"""
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=16000,
        temperature=0,
        messages=[{"role": "user", "content": prompt}]
    )
    return response.content[0].text
```

## 四、公元纪年标注

### 4.1 挑战

古籍中的时间表述极其多样：

| 格式 | 示例 | 难度 |
|------|------|------|
| 年号年 | `%建元六年%` | 低（查年号表） |
| 君主年 | `@秦昭王@%四十八年%` | 中（查在位年） |
| 朝代年 | `%汉元年%` | 中（需确定指哪个"汉元年"） |
| 裸年 | `%三年%` | 高（需从上下文推断指谁的三年） |
| 嵌入式 | `%秦始皇二十六年%` | 中（需解析复合表达） |
| 死亡引用 | `@某王@崩` | 中（取在位末年） |
| 无时间 | `-` | 需从前后事件插值 |

### 4.2 分层解析策略

```
优先级1：显式君主+年份
  "@秦始皇@%二十六年%" → 查reign_periods["秦始皇"].start_bce=221
  → CE = -(221 - (26-1)) = -196? 不对...
  → CE = -(start_bce - (year_num - 1)) = -(246 - 25) = -221 ✓

优先级2：年号
  "%建元六年%" → 查eras["建元"].start_bce=140
  → CE = -(140 - (6-1)) = -135

优先级3：别名解析 + 章节上下文
  "文公%三年%" 在秦本纪中 → "秦文公" → 查reign_periods
  "文公%三年%" 在鲁世家中 → "鲁文公" → 查reign_periods

优先级4：year_ce_map查表
  对复杂/无法解析的年份，查预构建的 year_ce_map.json

优先级5：上下文推断
  "%三年%" 无君主 → 取前一个成功解析的君主 → 推算
```

### 4.5 世纪级标注

对五帝等传说时代，精确年份不可考，使用世纪级颗粒度：

```markdown
- **时间**: -（公元前2700年）     ← 黄帝
- **时间**: -（公元前2500年）     ← 颛顼
- **时间**: %七十年%（公元前2230年） ← 帝尧在位70年的推算
```

### 4.6 输出格式

标注追加到事件的时间字段后：

```markdown

- **时间**: %秦始皇二十六年%

# 标注后
- **时间**: %秦始皇二十六年%（公元前221年）
```

区间格式：`（公元前246年-前221年）`

## 五、质量验证

### 5.1 事件格式验证（validate_events.py）

| 检查项 | 规则 |
|--------|------|
| 事件ID格式 | `DDD-DDD`，前缀匹配章节号 |
| ID连续性 | 001, 002, 003...（警告跳跃） |
| 必填字段 | 事件类型、主要人物、描述、原文引用、段落位置 |
| 概览表 | 必须存在且列结构正确 |
| 统计段 | 必须存在 |

### 5.2 纪年验证（lint_ce_years.py）

三条核心规则：

**规则1：时序检查**
```
章内事件的CE年应递增。
若 current_year < previous_year → 时序倒转警告
容差：±3年（同年不同解读）
严重：差值 > 20年 → 错误
```

**规则2：大跨度检查**
```
连续事件间隔 > 150年 → 警告（可能标注错误或遗漏事件）
```

**规则3：已知日期校验**
```python
KNOWN_DATES = {
    "孔子出生": -551,
    "秦始皇统一": -221,
    "陈胜起义": -209,
    "垓下之战": -202,
    "长平之战": -260,
    ...
}
# 匹配事件名称 → 对比CE标注 → 差值>5年则报错
```

## 八、经验教训

### 8.1 事件提取

1. **体裁决定粒度** — 本纪平均63事件/篇，列传平均20个。用统一粒度标准会让本纪太粗或列传太细。
2. **事件链比孤立事件更有价值** — 提取时要求LLM同时输出2-5条因果链，这些链直接成为sequel/causal关系的初始数据。
3. **保留实体标记** — 事件描述中保留 `@` `=` 标记，后续自动化（共人/共地计算）直接从标记中提取，无需二次NER。

### 8.2 关系发现

> **详见 [SKILL_事件关系发现.md](SKILL_事件关系发现.md) 第十节"经验教训"**：15条关系发现经验的完整总结。

**核心要点**：章内用LLM、跨章用自动；sequel占37%反映叙事线性特征；cross_ref置信度分级（1.0+共人最可靠）；co_location需加共人过滤避免噪音。

### 8.3 纪年标注

8. **分层解析是关键** — 从最确定的（年号）到最模糊的（裸年），逐层降级。每层失败才进入下一层。
9. **章节上下文是消歧利器** — 秦本纪中的"文公"必定是秦文公，不需要复杂算法。
10. **世纪级精度也有价值** — 五帝时代无法精确到年，但"前2700年"的世纪级标注让可视化成为可能。没有这个，五帝事件无法放到时间轴上。
11. **已知日期校验很重要** — 用"秦统一=前221年"等硬编码日期做锚点，一旦算出偏差就说明解析逻辑有bug。

### 8.4 工程实践

12. **先提取后标注后关联** — 三个阶段严格顺序执行。事件提取完成后再标注纪年，纪年稳定后再发现关系。逆序会导致频繁返工。
13. **关系合并需要去重** — 自动方法和LLM可能发现同一对关系，合并时按(source, target)去重，优先保留LLM结果（有reason字段）。
14. **插值算法让可视化完整** — 77%的事件有CE年，剩余23%通过线性插值获得X坐标，确保地铁图没有"断站"。
