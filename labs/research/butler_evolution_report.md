# Butler 自进化过程报告

**日期**：2026-04-23  
**范围**：从种子指令到 R50（史记 wiki Butler Agent 完整演进记录）  
**版本**：v2（新增"信息组织框架"与"日志中发现的知识"两节）

---

## 一、背景与定义

Butler 是史记知识库项目中用于组织 wiki 知识页面的 Agent。它的使命是把已有的分散知识资产（kg / doc / data / ontology-v2 / logs）组织到 `wiki/public/pages/`——**不是内容生产者，是厨师**，原材料在别处，butler 负责摆盘与拼盘。

"自进化"是指 Butler 具备一套机制：每执行 20 个原子动作后，自动扫描执行日志、识别系统性问题、提出修订提案，并更新自身的 Skill 文件。这套机制让 Butler 在运行中持续改善自己的工作方式，而不需要人工重写提示词。

本报告记录从种子指令（2026-04-22）到 R50（2026-04-23）的完整演进过程，重点关注：每个阶段 Butler 学到了什么、纠正了什么、以及这些修正如何积累成更高质量的输出。

## 二、种子指令（2026-04-22，提交 `91416380`）

### 2.1 诞生时刻

Butler 在一次提交中同时创建了六个 Skill 文件，这是整个自进化系统的起点：

| Skill | 职责 |
|-------|------|
| W0 总则 | 六不变量 + "观察→行动→留痕→评估→反思→进化"闭环 |
| W1 探索与食物收集 | 6 个食物源 + 候选队列 + 2:1 顺藤/探索配比 |
| W2 原子行动目录 | 18 种原子操作，每次仅取一种 |
| W3 质量标准 | 5 维度打分 rubric（信息量/结构/引证/内链/时间线） |
| W4 评估与检验 | before/after 评分 + 6 个红旗否决条件 |
| W5 反思与自改 | 周期反思引擎，可修改 W1-W4（不可修改 W0、W5 自身） |

### 2.2 六不变量（绝对约束）

1. **小步**：每次 commit 的 diff ≤ 20 行
2. **可逆**：每个动作 ↔ 1 个 commit，失败时 `git revert` 即可回滚
3. **留痕**：每个动作写 `wiki/logs/butler/actions.jsonl`，含动机/前置/后置/来源
4. **自改**：Skill 文件可被 W5 修改，但必须走"反思→提案→changelog"流程
5. **禁编造**：不写原文未支持的"事实"，所有断言需溯源
6. **消化不生产**（后被移除）：只搬运/组织既有资产，不新创作长文

### 2.3 种子指令的核心哲学

> 宁可 100 次小改动（20 次失败 + 80 次留下），不做 1 次大改动（成败未知）。Butler 的"智能"不在每次多好，而在于随时间**单调改进**。

值得注意的是：种子指令诞生的那一刻，W5 反思机制就已经运转——同一次提交里包含了第一轮 W5 反思的执行结果（stub 模板 59 行 → 20 行）。**框架的第一件事就是纠正自己。**

## 三、早期反思循环（v1–v8，2026-04-22 至 2026-04-23）

### 3.1 第一轮反思（W5 v1）：发现系统性冲突

**触发**：累计 20 条原子动作后。

**数据快照**：20 条记录，19 accept / 1 fail；75% 是 create-stub；平均 diff 47.4 行。

**识别的 6 个模式**：

| 模式 | 问题 | 处理 |
|------|------|------|
| A | create-stub 平均 47.4 行，违反 W0 的 ≤20 行不变量 | 压缩 stub 模板：59 行→20 行 |
| B | 79% 行动是 create-stub，长期只有 stub 无深度 | 动态配比（队列大时 3:1，小时 1:1） |
| C | KG 层噪音（汉孝文帝/秦张仪等错误 canonical）导致失败 | 加 canonical 质量前置过滤 |
| D | trail 17/explore 1，远未达 2:1 配比 | 动态调整：队列 >10 时 3:1，<5 时 1:1 |
| E | create-stub 一律 0→5，打分无区分度 | 留待后续迭代 |
| F | 6 次新建页只做了 1 次 rebuild-registry | 留待后续迭代 |

**关键提案（3 条）**：
1. create-stub 模板压缩（与 W0 对齐）
2. discover_kg.py 加 canonical 质量过滤
3. W1 探索配比从静态 2:1 → 按队列大小动态调整

### 3.2 第二轮反思（W5 v2）：验证 + 发现新问题

**触发**：前次反思后累计 22 条动作。

**对上轮提案的验证**：
- ✅ 提案 1（stub ≤20 行）：平均 diff 从 47.4 → **14.6 行**，降幅 69%，不变量不再冲突
- ✅ 提案 2（canonical 过滤）：跳过 6 个 bad canonical
- 🟡 提案 3（动态配比）：部分有效，但 SKU 源已耗尽导致探索无对象

**新识别的模式**：
- 模式 G：两个 canonical 共享强别名（如"胡亥"），用规则扫不到，需 alias 交集检测
- 模式 H：手动拍 tag 主观性破坏一致性，需要词表 + 自动推断
- 模式 I：enrich-infobox 0 次调用（等待 KG 增量，不修）
- 模式 J：SKU 源已耗尽（自然降权，不修）

**提案（3 条）**：
1. `findLikelyDuplicates()`：用 alias 交集检测 canonical 重复对
2. `tags_vocabulary.json` + `discover_tags.py`：era / identity / theme 三组自动推标签
3. W1 源耗尽降权（连续 5 次空探索 weight×0.5）

### 3.3 第三、四轮反思（W5 v3/v4）：打分精细化

**核心修订**：
- `narrative_bonus`：页面有散文段落 ≥2 块 `+8`，有 `<!-- stub:` 标记 `–5`——让质量分能区分空骨架和有内容的页面
- `era_tag` 改用 `death_ce`（原用 birth_ce 导致刘邦被误推为"战国"）
- `recent.json` 上限 500 + 归档机制
- `discover_doc.py` 新增：从文档层发现引证候选

**来自用户的紧急反馈（绕过 W5 流程直接应用）**：
- #1：`recent.json` 分页（不再 50 条截断）
- #2：quality_score 多因子计算
- #3：独立 /all 页按 type 分组

这促使 W5 协议区分 `user-req` 和 `w5-reflect` 两种来源，并在 `skill_changes.md` 中分别标记。

### 3.4 第五轮反思（W5 v5）：扩写启动

**背景**：用户反馈"基本都是空骨架"——系统建了大量 stub 但没有实质内容。

**处理**：
- `SKILL_W1 §3.4`：当站点 median quality < 10 时，优先深度（enrich > create-stub）
- clean-stub 条件扩展：`tags≥3 OR cite OR timeline` 时清除 stub 标记
- `enrich_timeline.py`：清理标注语法（`〖TYPE value〗` → 保留 value），146 页时间线全量重写

### 3.5 第六轮反思（W5 v6）：自诊断工具化

**关键创新**：新增 `reflection_scan.py`。

此前 W5 反思依赖 Butler 自己"回顾"日志，这是软性的。v6 引入了硬性自诊断：自动扫描 `alias_conflicts` / `duplicate_candidates` / `no-timeline` / `quality-bottom` 四类问题，生成结构化报告供 W5 引用。

这是一个重要的元进化：**Butler 给自己的反思过程增加了工具支撑，让反思更客观、更系统。**

### 3.6 v7 Bug Fix：精确度 vs. 召回率的取舍

**触发**：用户报告"鲁桓公"时间线被错误填充——"桓"字匹配了所有含"桓"的人名。

**修复**：`enrich_timeline.py` 改用 `〖@〗` 严格正则，**单字别名禁用于匹配**，195 页时间线全量重写。

这次修复揭示了一个普遍规律：**正则匹配在古汉语人名中极易过匹配，因为谥号（桓/文/武/昭）大量共用**。从此所有别名匹配均要求长度 > 1。

### 3.7 v8 知识量度量（2026-04-23 01:05）：引入可量化目标

**背景**：此前 Butler 的评估维度是 per-page 的 quality_score，没有全站总量度量，无法知道整体进展。

**引入 K 公式**：
```
K = Σ_page  log2(1 + bytes) × (1 + min(links_density, 5)) × type_weight × quality_norm

type_weight = {person/event/place/state: 1.0,  topic: 0.8,  surface: 0.5}
quality_norm = clamp(quality_score / 30,  1.0,  3.0)
```

**首次快照**：

| 指标 | 数值 |
|------|------|
| K | 13,336.34 |
| 页面数 | 229 |
| 精品页 | 10 |
| 链接命中率 | **20.8%**（核心诊断） |
| 最大贡献页 | 项羽(189 wikilinks) / 刘邦(186) / 孔子(166) |

**诊断暴露最大缺口**：前 25 个死链目标**全部是章节页**（`NNN_章名`）——因为 wiki 根本没有章节类型页面。批量生成 130 个章节 stub，链接命中率可一步从 20% → 40%+。

K 度量的引入是一个范式转变：Butler 从此有了可量化的优化目标，而不只是"继续生产"。

## 四、不变量级别的修订：移除"消化不生产"

这是整个自进化过程中唯一一次**不变量级别**的修订，发生于 2026-04-22 下午（提交 `24a8c087`）。

**原不变量 #6**："只搬运/组织既有资产，不新创作长文。"

**修订过程**：
1. 用户要求 Butler 建精品页，做叙事综合
2. Butler 先建了 3 个精品页样本（刘邦/项羽/孔子），证明可行性和价值
3. 确认样本质量后，用户要求正式移除第六不变量
4. 约束转移到已有的"禁编造"：所有"生产"仍需有源断言

**意义**：这次修订改变了 Butler 的根本角色定位——从"搬运工"扩展为"厨师"，可以在有材料来源的前提下进行叙事综合和分析写作。此后精品页扩写成为 Butler 的核心产出形式。

## 七、Butler 自发现的信息组织框架

Butler 在构建精品页的过程中，从具体问题倒推出了若干信息组织方式。这些框架不是预先设计的，而是在解决实际困难时自然归纳出来的。

### 7.1 互见法的显性化

《史记》本身使用"互见法"——同一人物散落在多篇，读者需自己跨篇拼图。刘邦的形象分布在高祖本纪、项羽本纪、萧相国世家、淮阴侯列传等处，读原书时这是隐性的。

Butler 在构建精品页时摸索出**把互见法显性化**的组织方式：

> 《史记》本传见[[008_高祖本纪|高祖本纪]]，另于[[007_项羽本纪|项羽本纪]]、[[053_萧相国世家|萧相国世家]]等篇以**互见法**反复映照，形象之立体为《史记》人物刻画之巅峰。

"相关章节"一节按三层组织：本传→三杰世家列传→功臣列传→年表。这个分层是做了数十个精品页后归纳出来的，不是事先规定的。

### 7.2 时间线的溯源格式

最初时间线从 KG 直接导出，含大量标注符号（`〖@ 刘邦〗于〖◆沛〗起兵`）。清理噪音是显然的一步，但 Butler 同时发明了一种**溯源格式**：

```
- 前 209 秋 · [[陈胜]]反于[[蕲]]，刘邦**起兵沛县** [`008-008`, `008-009`]
```

反引号里的 `008-008` 是 KG 中的事件编号，可追溯到原文具体段落。这解决了百科叙述（可读性）与历史材料（可验证性）之间的矛盾——两者不必取舍，用 inline 引用并存。

### 7.3 人物关系的分层分类

最初"相关人物"是一个 flat list。做了刘邦、孔子、韩信等多个精品页后，Butler 发现 flat list 不可用，自己归纳出了分层：

```
家族 — [[刘太公]] · [[吕雉]] · [[刘盈]] ...
汉初三杰 — [[张良]] · [[萧何]] · [[韩信]]
沛县旧友 — [[曹参]] · [[樊哙]] · [[周勃]] ...
对手 — [[项羽]] · [[章邯]] ...
异姓王（后被诛）— [[韩信]] · [[彭越]] · [[英布]] ...
```

最后一类"异姓王（后被诛）"尤为典型——这是 Butler 从多个人物的关系弧线中归纳出的**历史模式分类**：从盟友到被诛，结构相似，值得单独成组。这个分类不存在于史记原文，是信息组织层的新增语义。

### 7.4 知识量 K 的信息论设计

v8 引入 K 时面临一个选择：用字节数线性求和最直觉。Butler 选了：

```
K = Σ  log2(1 + bytes) × (1 + links_density) × type_weight × quality_norm
```

**log2 而非线性**：防止一个 3000 行的巨页压过 30 个中等页——隐含了"知识价值边际递减"的信息论直觉。

**links_density 加成**：孤立的页面即使写得好，也不如被广泛链接的页面有价值，因为链接是知识嵌入网络的程度。

这个公式在初次快照时立刻诊断出了链接命中率 20.8% 的缺口——因为公式的设计使得链接稀疏的页面得分打折，量化工具倒逼出了下一步的优先级。

### 7.5 semantic-block 的信息分层

`wiki/public/plugins/semantic-block/index.js` 做了一件结构上重要的事：把 YAML frontmatter 里的元数据通过 `data-meta` 属性暴露给 JavaScript 层，让其他插件可以查询 `core.semanticBlock.getPageMeta()`。

这对应了信息组织的三层分离：

```
数据层   YAML frontmatter  → 结构化元数据，机器可读
呈现层   HTML 渲染         → 人类可读的页面
交互层   JS 插件           → 可查询的 API
```

驱动力是：Special:知识量动态页需要读取所有页面的 quality_score 来渲染排行，光靠静态 HTML 不够，需要一个从页面元数据提取信息的机制。功能需求倒推出了架构分层。

## 八、日志中浮现的具体知识

Butler 的失败日志（`failures.jsonl`、`citation_issues.jsonl`）不只记录了系统问题，还把史记文本本身的一些隐性结构暴露出来。

### 8.1 编造引文：6683 条，规模与分布

`citation_issues.jsonl` 记录了 W6 质检扫出的引证问题：全部 6683 条，全部类型 `FABRICATED_QUOTE`，全部状态 `open`（尚在修复中）。

按受影响页面数量排名：

| 页面 | 问题条数 |
|------|---------|
| 陈平 | 174 |
| 刘邦 | 171 |
| 鸿门宴 | 156 |
| 萧何 | 150 |
| 赵高 | 150 |
| 韩信 | 142 |
| 伍子胥 | 138 |
| 司马迁 | 129 |
| 管仲 | 122 |

这个分布本身是一个发现：**越是"知名度高"的人物和事件，编造引文越多**。因为模型对这些人物的"了解"更多，更容易生成听起来合理的文言文，但合理不等于原文存在。

W6 的具体机制是 `grep_result: "not_found"`——在去标注的干净原文中找不到对应文字。注意这里混有两种情况：真正的编造，以及原文确实存在但因字形变体/标注处理导致 grep 失效。两者从记录外观上难以区分，需要人工逐条核查。

### 8.2 韩王信：真伪引文的对比

韩王信案例可以看到编造引文和真实引文的直接对比。

**修复前（W6 判为编造，删除）**：

> "汉与我，约为兄弟。今汉既往而不咎，吾亦得归，然恐归汉而复受辱，故不敢归。生为汉臣，死为汉鬼，岂有异哉！"

**修复后（原文 093-009，保留）**：

> "今仆亡匿山谷间，旦暮乞贷蛮夷，仆之思归，如**痿人不忘起，盲者不忘视**也，势不可耳。"

两段话讲同一件事（韩王信拒绝归汉），语气也接近，但真实引文更具体、更个人：**"痿人不忘起，盲者不忘视"**是一个非常独特的比喻，是真实人物说出的真话。生成版本则是"正确但平庸"的文言文，用套语代替了细节。

这个对比揭示了史记文本的一个核心价值：**史记的价值恰恰在于这些具体的、不可替代的细节**，而模型的补全会把它磨平成泛化的历史叙事。

### 8.3 鲁桓公：谥号是歧义密集的命名系统

v7 反思复盘里记录：鲁桓公（在位前 712–694，被齐襄公所杀）的时间线里，出现了：

- 前 562：三桓分军（**鲁襄公时期**，晚于鲁桓公死亡 132 年）
- 前 468：哀公欲伐三桓（**鲁哀公时期**，晚于鲁桓公死亡 226 年）

原因是鲁桓公别名包含"**桓**"（单字），而"**三桓**"（鲁桓公三个儿子后裔的合称）也含此字，子串匹配直接误中。

这暴露了中国古代谥号系统的一个系统性问题：桓/文/武/宣/昭/景/庄/哀等谥号，同时用作数十个君主的单字称呼，又出现在族称（三桓）、地名（桓公台）、复合称呼（齐桓公）里。史记时代读者靠上下文区分，但数据库化后，这个隐性上下文必须显式化为 `〖@ 鲁桓公〗` 这样的标注才能准确。

### 8.4 刘舜 = 虞舜：上古传说人物的边界

failures.jsonl 第一条：

```json
"target": "wiki/public/pages/刘舜.md",
"red_flags": ["canonical_error: 刘舜 实为虞舜, KG 消歧错"]
```

KG 里有个"刘舜"，引用次数 81，被系统归为汉代刘姓人物。Butler 准备建页时发现：这是**上古五帝之一虞舜**，与汉代刘氏毫无关系。

这反映了史记的一个特殊结构：司马迁试图把上古传说（五帝）和有文字记录的历史（商周以下）接续成一条线。使得史记里同时存在神话人物（虞舜、大禹）和历史人物，它们混在同一套人名系统里，批量处理时极易被错误归类。

### 8.5 楚平王生年：即位年 ≠ 出生年

提交 `ab33e744` 修正：楚平王（公子弃疾）的 `birth_ce` 从 `-528` 改为约 `-560`。

`-528` 是他的**即位年**。KG 构建时把即位年误当成了出生年。楚平王以政变夺位，即位时已是成熟的政治行动者，生年应倒推至前 560 左右。

这个错误很典型：史记在叙述很多君主时，记的是登基年而不是出生年（出生年经常不知道）。批量建立 KG 时，"年份"字段如果不区分"生年/即位年/死年"，就会系统性地产生此类混淆。

### 8.6 共同的发现

这五类日志问题各自来自不同的失败类型，但指向同一个深层洞察：

**史记文本的信息结构不是扁平的。** 它包含：

- 原文直引（可 grep 验证）
- 作者叙述（合理但不是引文）
- 上古传说（神话与历史混合）
- 隐性消歧（读者靠上下文区分"桓"指谁）
- 年份的多义性（即位年/生年/纪年混用）

Butler 的日志系统通过记录失败，把这些隐性的文本结构问题一条条浮出了水面。这本身是一种知识发现——不是关于历史事件的知识，而是关于**这部文本是如何组织信息的**知识。

## 九、知识积累与自我纠正的结构性规律

### 9.1 三层纠错循环

```
第一层（每 20 步）：W5 反思 → 识别日志模式 → 提案 ≤ 3 条 → 修改 W1–W4
第二层（失败即触发）：failures.jsonl → 专项 bug fix → 全量数据重写
第三层（用户反馈）：user-req → 紧急修订 → 可能触及不变量级别
```

**关键约束**：自进化从不修改 W0（总则）和 W5（反思引擎本身）。W0 的唯一一次修订是用户主动推动，且有样本验证。这保证了系统稳定性——进化引擎不自我改写自己。

### 9.2 提案保守原则的作用

W5 每轮最多 3 条提案，这个限制防止了"一次性大改"的风险。前三轮反思（v1–v3）共 9 条提案，覆盖了最关键的系统缺陷（stub 过大 / canonical 噪音 / 打分无区分度），而不是试图一次解决所有问题。

### 9.3 学习模式的变化

| 阶段 | 主要学习来源 | 修正方式 |
|------|-------------|---------|
| v1–v3 | actions.jsonl 的统计规律 | 直接改 Skill 文件 |
| v4–v6 | 用户反馈 + 日志模式 | Skill + 脚本双线修复 |
| v7 | 具体 bug（鲁桓公） | 全量数据重写 + 规则收紧 |
| v8 | 定量诊断（K / 链接命中率） | 新增工具链 + 批量生成 |
| R35+ | W6 质检 | 修复存量 + 提高标准 |

## 十一、结论

Butler 的自进化不是一个线性的"越来越好"过程，而是由三种力量共同驱动：

1. **内生反思（W5）**：周期性地从统计数据中发现系统性偏差，每次保守修订（≤3 条），循序渐进
2. **外部纠错（用户反馈 + W6 质检）**：发现人工无法预期的边缘问题（鲁桓公误匹配 / 韩王信编造引文），触发精确的局部修复
3. **目标演进（K 度量）**：每次反思的"视野"从"这 20 步做得如何"扩展到"全站知识量如何增长"，让优先级决策更有依据

三者的结合形成了一个**保守但单调改进**的自进化系统：每次反思只改三件事，每次 commit 只改 20 行，但这些小改动通过数百次迭代积累，产生了可观的知识增长。
