## 二、现状诊断

### 2.1 renderAll（#?all）— 已出现性能问题

**代码位置**：`wiki/public/js/renderer.js` L662–L728

**机制**：
1. `Object.keys(pages)` 取出全部 671 个页面
2. 每页生成约 200B HTML（`<li><a>…</a><span>…</span></li>`）
3. 全部拼接后一次性 `innerHTML` 写入

**当前代价**：
- 生成 HTML 约 131 KB，含 2684 个 DOM 节点
- person 组单独 351 项，占全部渲染的 52%
- 浏览器需一次 layout 全部节点，低配设备/手机上明显卡顿
- 页面无任何过滤/跳转手段，351 个人名靠眼睛逐行扫描

**预测**：
- N=1000 时：person 组约 520 项，全页 DOM 节点约 4000，体验显著恶化
- N=1500 时：全页 DOM 节点约 6000，部分手机浏览器会冻结数秒

### 2.2 renderCategory（#?type=person）— 缺少过滤手段

**代码位置**：`wiki/public/js/renderer.js` L432–L478

**机制**：全量渲染匹配该 type 的所有页面，`<ol>` 排列，无分页无筛选。

**当前代价**：351 个人名有序排列，但列表已超出一屏，用户无法按姓氏、朝代等维度快速定位。

### 2.3 pages.json 全量加载 — 短期无问题但增速快

**代码位置**：`wiki/public/index.html` L48–L56（搜索 datalist 填充）+ `js/core.js`（registry 加载）

**机制**：页面加载时 `fetch('pages.json')` 取全量数据（329 KB），填充 `<datalist>` + 构建客户端 registry。

**现状**：329 KB 尚在合理范围，但按当前速度 1.6 个月后到 500 KB。500 KB JSON 的 parse 耗时约 15–30 ms（移动端 50–80 ms），开始影响首屏。

### 2.4 78 个无 type 页面

`pages.json` 中有 78 个 `type` 字段为空的页面，在 renderAll 中以 `unknown` 组呈现，可读性差。需要补分类或排除。

## 三、分层改进方案

### 层级定义

| 层级 | 触发条件 | 原则 |
|------|----------|------|
| **T1 当下（N≈671）** | 现在可做 | 纯前端改动，不动数据结构，≤1 天工作量 |
| **T2 近期（N≈800–1000）** | 0.6–1.6 月内 | 轻量数据改造，1–3 天 |
| **T3 中期（N≈1500）** | 3–4 月内 | 预建索引，2–5 天 |
| **T4 远期（N≈3000+）** | 1 年以上 | 后端查询层，需架构决策 |

---

### T1：首字过滤栏（当下可做，工作量约 2–3 小时）

**改 `renderAll` 和 `renderCategory`**：在列表上方加一行首字筛选按钮。

```
[全部] [刘] [张] [赵] [陈] [王] [李] [吕] [田] [晋] [秦] [楚] …
```

- 点击某字：隐藏不以该字开头的 `<li>`（用 CSS `display:none`，不重建 DOM）
- "全部"：恢复所有
- 工作量：renderer.js 新增约 40 行，CSS 新增约 20 行
- **零数据改动，不破坏任何现有功能**

**同时改 renderAll 的 person 组**：默认折叠非 person 类型（person 展开），每组标题可点击展开/折叠。

---

### T2：按需渲染（N=800–1000 时执行，工作量约 1 天）

**改 `renderAll`**：
- 每个类型组只默认渲染前 30 项
- 组底部加"展开全部 N 项"按钮，点击后动态插入剩余 `<li>`（懒渲染）
- person 组加字母/笔画分组子标题

**改 `renderCategory`**：
- 每页显示 50 项，加翻页控件（← 上一页 / 下一页 →）
- URL 参数 `#?type=person&page=2`（router.js 已有 `page` 参数处理 recent，可复用）

**改搜索 datalist**：
- 新建 `pages_index.json`：只含 `{id, label, type}`，约 40–60 KB（替代 329 KB 的 pages.json 用于 datalist）
- `pages.json` 全量数据只在页面渲染时按需加载

---

### T3：预建搜索索引（N=1500 时执行，工作量约 2–3 天）

- 构建时生成 `search_index.json`（lunr.js 格式），包含人名/别名/标签全文
- 搜索框改为查询索引，而不是遍历 datalist
- datalist 退化为"最近查看" + "featured 精品页"快速选择（约 30 项）

---

### T4：后端查询层（N=3000+，暂不规划）

- 将 pages/history/recent 迁移到 SQLite 或 DuckDB
- 轻量 FastAPI 或 Deno HTTP server 提供查询接口
- **此层需要用户明确决策**，现阶段不推进
