# 架构提案 2026-04-30

## 触发指标

| 指标 | 当前值 | 临界线 | 警戒线 | 状态 |
|------|--------|--------|--------|------|
| pages.json 大小 | 5347 KB | 1024 KB | 500 KB | 🔴 ×5.2 |
| wiki 页面总数 | 20408 | 1500 | 800 | 🔴 ×13.6 |
| person 页数 | 4327 | 1000 | 600 | 🔴 ×4.3 |
| history/ 文件数 | 20227 | 2000 | 1000 | 🔴 ×10.1 |

## 当前问题描述

四项基础设施指标全部远超临界线，且增长速度很快（butler 每 23 轮约新增 20-30 页，知识量从 749K→812K）。wiki 的纯静态文件架构已无法支撑当前规模：

1. **pages.json（5.3MB）** — 每次需要查找页面信息时全量加载，Special:Recent 等依赖 pages.json 的页面加载缓慢
2. **history/ 文件数（20K+）** — 每个页面一个独立 JSONL 文件，`ls` 操作耗时长，批量操作效率低
3. **页面总数（20408）** — 列表页（全部人物/全部事件）渲染全量数据，客户端负担重

## 推荐方案

### 方案 A（最小改动）：pages.json 分型索引 + history/ 惰性加载

- pages.json 拆为 `pages_index.json`（仅 id/label/type/quality，约 400KB）和 `pages_detail.json`（完整元数据，按需加载）
- history/ 文件保持独立，但 Special:Recent 改为仅加载最近 500 条
- 工作量：≈ 2h
- 限界：不解决 history/ 文件数膨胀，20K 文件不会减少

### 方案 B（推荐 ✅）：history/ 合并为 SQLite + pages.json 拆分

- 将 `wiki/public/history/*.jsonl` 合并为 `wiki/public/history.sqlite3`（单文件替代 20K+ 文件）
- pages.json 拆分为 `pages_index.json`（轻量索引）+ 按类型子文件
- `build_recent.py` 改为从 SQLite 读取
- 工作量：≈ 6h（含迁移脚本 + 前端兼容）
- 收益：消除 20K 文件 IO 瓶颈，pages.json 加载量降 90%

### 方案 C（较大改动）：全量数据库迁移

- 将 pages.json + history/ + recent.json 全部迁移到 SQLite
- 引入轻量后端（FastAPI 或 Node.js 静态服务器）提供查询 API
- 前端按需加载，不依赖全量静态 JSON
- 工作量：≈ 3 天
- 收益：完全解决规模问题，可支撑到 100K 页
- 代价：需要维护后端服务，不再是无服务静态站点

## 建议时间窗口

- 方案 A 目前即可执行（2h，作为临时缓解）
- 方案 B 建议在 **下个发布周期** 执行（未来 2-3 天内），可与其他架构工作合并
- 方案 C 建议在页面超过 **30000** 时考虑（当前增长速率约 1500 页/周，约 8-10 周后达到）
